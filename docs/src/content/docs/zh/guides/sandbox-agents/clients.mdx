## 决策指南

| 目标 | 首选方案 | 原因 |
| --- | --- | --- |
| 在 macOS 或 Linux 上实现最快的本地迭代 | `UnixLocalSandboxClient` | 无需额外的服务依赖，并提供简单的本地文件系统工作流。 |
| 基本的容器隔离 | `DockerSandboxClient` | 使用指定镜像在 Docker 内运行任务。 |
| 托管执行或生产环境级隔离 | 托管沙盒客户端 | 将工作区边界移至由提供商管理的环境。 |

## 本地客户端

对于大多数用户，建议从以下两个沙盒客户端之一开始：

| 客户端 | 安装要求 | 适用场景 |
| --- | --- | --- |
| `UnixLocalSandboxClient` | 无 | 在 macOS 或 Linux 上实现最快的本地迭代。适合作为本地开发的默认选择。 |
| `DockerSandboxClient` | 本地可用 Docker CLI | 需要容器隔离，或需要使用指定镜像确保本地环境一致性。 |

Unix 本地模式是基于本地文件系统开始开发的最简单方式。当您需要更强的环境隔离或与生产环境保持一致时，可以迁移到 Docker 或托管提供商。

### 恢复与快照

沙盒状态与对话状态彼此独立：

- SDK 对话状态存储在 `result.history`、SDK `Session`、`conversationId` 或 `previousResponseId` 中。
- 沙盒状态存储在活动沙盒会话、序列化的 `sessionState`、`RunState` 沙盒载荷或快照中。

如果希望通过沙盒客户端重新连接到同一个后端会话，请使用 `sessionState`。如果希望创建一个以已保存工作区内容为初始状态的新会话，请使用快照。

### 文件系统存在性检查

当会话实现 `pathExists()` 时，`false` 表示后端已确认该路径不存在。权限错误、I/O 故障和提供商探测失败会以提供商错误拒绝操作，而不会被视为路径不存在。这可以防止编辑器、清单和记忆流程覆盖或替换可能存在但无法访问的路径。

## 挂载与远程存储

挂载条目描述要公开的存储，挂载策略描述沙盒后端如何附加该存储。从 `@openai/agents/sandbox` 导入内置挂载条目和通用策略。

常用挂载选项：

- `mountPath`：存储在沙盒中的显示位置。相对路径基于清单根目录解析；绝对路径按原样使用。
- `readOnly`：当沙盒不应将数据写回挂载的存储时，请设置此选项。
- `mountStrategy`：使用同时匹配挂载条目和沙盒后端的策略。

挂载会被视为临时工作区条目。快照和持久化流程会分离或跳过已挂载路径，而不会将挂载的远程存储复制到已保存的工作区中。

通用本地和容器策略：

| 策略或模式 | 适用场景 | 说明 |
| --- | --- | --- |
| `inContainerMountStrategy(...)` | 沙盒镜像可以运行 `rclone`、`mount-s3` 或 `blobfuse2` 等挂载命令。 | 可作为通用策略使用；支持情况取决于后端。 |
| `dockerVolumeMountStrategy(...)` | Docker 应在容器启动前附加由卷驱动支持的挂载。 | 仅限 Docker。 |
| `localBindMountStrategy()` | 本地后端应将绝对本地路径绑定挂载到工作区。 | 在允许的情况下，由本地工作区物化流程提供支持。 |

后端支持情况有意采用显式定义：

| 后端 | 挂载说明 |
| --- | --- |
| `UnixLocalSandboxClient` | 通过本地工作区模型支持本地绑定式挂载。 |
| `DockerSandboxClient` | 支持本地绑定挂载，以及 Docker 能够附加存储时的 Docker 卷式策略。 |
| 托管提供商 | 提供商专用策略随各提供商的实现一起提供。请查看相应提供商的文档，了解支持的挂载方式和所需配置。 |

不要假设挂载条目适用于所有后端。如果客户端无法实施清单元数据、身份或挂载行为，应尽早失败，而不是静默忽略清单的相应部分。

## 支持的托管平台

当您需要托管环境时，通常可以沿用相同的 `SandboxAgent` 定义，只需更改 `sandbox` 运行选项中的沙盒客户端。

托管提供商实现可从 `@openai/agents-extensions` 的提供商子路径中获取。请查看提供商文档，了解确切的环境变量、可运行代码示例、端口行为、PTY 支持、快照行为和清理行为。

安装 `@openai/agents-extensions` 并满足其包级对等依赖。每个提供商可能还需要提供商 SDK 包或后端配置：

| 客户端 | 导入路径 | 提供商要求 |
| --- | --- | --- |
| `BlaxelSandboxClient` | `@openai/agents-extensions/sandbox/blaxel` | npm 对等依赖：`@blaxel/core` |
| `CloudflareSandboxClient` | `@openai/agents-extensions/sandbox/cloudflare` | Cloudflare Sandbox 桥接 Worker URL 和 Worker 身份验证 |
| `DaytonaSandboxClient` | `@openai/agents-extensions/sandbox/daytona` | npm 对等依赖：`@daytonaio/sdk` |
| `E2BSandboxClient` | `@openai/agents-extensions/sandbox/e2b` | npm 对等依赖：`e2b` 或 `@e2b/code-interpreter` |
| `ModalSandboxClient` | `@openai/agents-extensions/sandbox/modal` | npm 对等依赖：`modal` |
| `RunloopSandboxClient` | `@openai/agents-extensions/sandbox/runloop` | npm 对等依赖：`@runloop/api-client` |
| `VercelSandboxClient` | `@openai/agents-extensions/sandbox/vercel` | npm 对等依赖：`@vercel/sandbox` |

`CloudflareSandboxClient` 不会导入 Cloudflare npm SDK，而是通过 HTTP 与已部署的 Cloudflare Sandbox 桥接 Worker 通信。

`VercelSandboxClient` 按以下优先级解析每个 PAT 凭据字段：每次创建时的选项、构造函数选项，然后是 `VERCEL_PROJECT_ID`、`VERCEL_TEAM_ID` 和 `VERCEL_TOKEN`。仅当最终得到的 `projectId`、`teamId` 和 `token` 均非空时，才会转发凭据。否则，它会省略全部三个字段，让 `@vercel/sandbox` 解析身份验证，包括平台 OIDC 或本地提供商凭据。会话序列化并恢复时，会保留完整解析后的凭据。序列化凭据会被视为一个完整的三字段组合，不会与当前选项或环境变量混用；在将当前配置作为后备方案进行解析前，不完整的序列化凭据会被丢弃。由于令牌仍会保留在序列化会话状态中，请安全存储该状态。

托管沙盒客户端会公开提供商专用的挂载策略。请选择最符合您的存储提供商需求的后端和挂载策略：

| 后端 | 挂载说明 |
| --- | --- |
| Docker | 支持将 `s3Mount()`、`gcsMount()`、`r2Mount()`、`azureBlobMount()`、`boxMount()` 和 `s3FilesMount()` 与 `inContainerMountStrategy()`、`dockerVolumeMountStrategy()` 等本地策略结合使用。 |
| `ModalSandboxClient` | 支持在 S3、R2 和使用 HMAC 身份验证的 GCS 挂载条目上，通过 `ModalCloudBucketMountStrategy` 挂载云存储桶。 |
| `CloudflareSandboxClient` | 支持在 S3、R2 和使用 HMAC 身份验证的 GCS 挂载条目上，通过 `CloudflareBucketMountStrategy` 挂载 Cloudflare 存储桶。 |
| `BlaxelSandboxClient` | 支持在 S3、R2 和 GCS 挂载条目上，通过 `BlaxelCloudBucketMountStrategy` 挂载云存储桶。还支持通过 `BlaxelDriveMount` 和 `BlaxelDriveMountStrategy` 挂载持久化 Blaxel Drive。 |
| `DaytonaSandboxClient` | 支持在 S3、GCS、R2、Azure Blob 和 Box 挂载条目上，通过 `DaytonaCloudBucketMountStrategy` 进行由 rclone 支持的挂载。 |
| `E2BSandboxClient` | 支持在 S3、GCS、R2、Azure Blob 和 Box 挂载条目上，通过 `E2BCloudBucketMountStrategy` 进行由 rclone 支持的挂载。 |
| `RunloopSandboxClient` | 支持在 S3、GCS、R2、Azure Blob 和 Box 挂载条目上，通过 `RunloopCloudBucketMountStrategy` 进行由 rclone 支持的挂载。 |
| `VercelSandboxClient` | 支持通过 `VercelCloudBucketMountStrategy` 在创建时挂载 S3。内联凭据需要设置 `allowS3CredentialExposure: true`；请复用已挂载的活动会话，因为序列化的挂载状态无法恢复。 |

对于 E2B 和 Runloop 上由 rclone 支持的挂载，如果已有 `rclone` 二进制文件可用，SDK 会直接使用它。否则，只有在验证 SHA-256 校验和后，SDK 才会安装由其固定版本的 Linux 归档文件；不支持的架构和校验和验证失败都会中止挂载。

下表汇总了每个后端可直接挂载的远程存储条目：

| 后端 | AWS S3 | Cloudflare R2 | GCS | Azure Blob Storage | Box | S3 Files |
| --- | --- | --- | --- | --- | --- | --- |
| Docker | 是 | 是 | 是 | 是 | 是 | 是 |
| `ModalSandboxClient` | 是 | 是 | 是 | 否 | 否 | 否 |
| `CloudflareSandboxClient` | 是 | 是 | 是 | 否 | 否 | 否 |
| `BlaxelSandboxClient` | 是 | 是 | 是 | 否 | 否 | 否 |
| `DaytonaSandboxClient` | 是 | 是 | 是 | 是 | 是 | 否 |
| `E2BSandboxClient` | 是 | 是 | 是 | 是 | 是 | 否 |
| `RunloopSandboxClient` | 是 | 是 | 是 | 是 | 是 | 否 |
| `VercelSandboxClient` | 是 | 否 | 否 | 否 | 否 | 否 |

### 功能支持矩阵

| 功能 | Unix 本地模式 | Docker |
| --- | --- | --- |
| `exec_command` | 支持 | 支持 |
| PTY `write_stdin` | 支持 | 支持 |
| `apply_patch` | 支持 | 通过工作区文件 API 支持 |
| `view_image` | 支持 | 通过工作区文件 API 支持 |
| 命令的 `runAs` | 当主机可以解析并切换到相应用户时支持 | 受容器和用户配置限制 |
| 本地快照 | 支持 | 支持 |
| 本地和 Docker 挂载 | 支持本地绑定式挂载 | 支持绑定挂载和 Docker 卷式挂载 |

本地 PTY 支持会在 SDK 进程中使用一个小型 Python 3 桥接程序。该桥接程序仅用于 `tty: true` 会话，因为 Node.js 不提供内置 PTY API，而 SDK 需要标准 POSIX PTY 行为来处理交互式标准输入、信号和退出状态报告。请在运行 SDK 代码的环境中安装 `python3`，或将 `OPENAI_AGENTS_PYTHON` 设置为 Python 3 可执行文件。这与 Docker 沙盒镜像内安装的 Python 版本（如果有）无关。

托管提供商的支持情况因提供商而异。请查看提供商专用文档，了解确切的选项、环境变量、端口行为、PTY 支持、快照行为和清理行为。
