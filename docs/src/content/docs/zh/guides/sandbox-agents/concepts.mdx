现代智能体在能够对文件系统中的真实文件进行操作时效果最佳。**Sandbox Agents** 可以使用专门的工具和 shell 命令，对大型文档集进行搜索和操作、编辑文件、生成产物并运行命令。沙盒为模型提供了一个持久化工作区，智能体可以用它代您完成工作。Agents SDK 中的 Sandbox Agents 可帮助您运行与沙盒环境配对的智能体，让您能够轻松将正确的文件放入文件系统，并大规模编排沙盒的启动、停止和任务恢复。

您可以围绕智能体所需的数据定义工作区。它可以从 GitHub 仓库、本地文件和目录、合成任务文件、S3 或 Azure Blob Storage 等远程文件系统，以及您提供的其他沙盒输入开始。

`SandboxAgent` 扩展自 `Agent`，因此它本质上仍然是一个 `Agent`。它保留了常规智能体接口，如 `instructions`、`tools`、`handoffs`、`mcpServers`、`modelSettings`、输出类型、护栏和 hooks，并且仍通过标准的 `run()` 和 `Runner` API 运行。变化之处在于执行边界：

- `SandboxAgent` 定义智能体本身：包括常规智能体配置，以及沙盒专属默认项，如 `defaultManifest`、`baseInstructions`、`runAs`，以及文件系统工具、shell 访问、skills、memory 或 compaction 等能力。
- `Manifest` 声明一个全新沙盒工作区所需的初始内容和布局，包括文件、仓库、挂载和环境。
- 沙盒会话是命令执行和文件变更发生的实时执行环境。
- `sandbox` 运行选项决定本次运行如何获得该沙盒会话，例如直接注入、从序列化的沙盒会话状态重新连接，或通过沙盒客户端创建一个新的沙盒会话。
- 保存的沙盒状态和快照可让后续运行重新连接到先前的工作，或用保存的内容为一个新的沙盒会话提供初始数据。

`Manifest` 定义新沙盒工作区的初始内容。它并不描述每个实时沙盒中的当前文件，因为复用的会话、序列化的会话状态和快照都可能在运行时提供或更改工作区。

在本页中，"沙盒会话"指由沙盒客户端管理的实时执行环境。其具体边界取决于客户端：Unix 本地会话在主机上的本地工作区中运行，而 Docker 和托管客户端则提供更强的环境隔离。这不同于 [会话](/openai-agents-js/zh/guides/sessions) 中描述的 SDK 对话式 `Session` 接口。

外层运行时仍负责审批、追踪、交接和恢复记录。沙盒会话则负责命令、文件变更和环境隔离。这种职责拆分是该模型的核心部分。

### 组件协作方式

一次沙盒运行会将智能体定义与每次运行的沙盒配置结合起来。运行器会准备智能体，将其绑定到实时沙盒会话，并可为后续运行保存状态。

沙盒专属默认项保留在 `SandboxAgent` 上。每次运行的沙盒会话选择保留在 `sandbox` 运行选项中。

可以将其生命周期分为三个阶段：

1. 使用 `SandboxAgent`、`Manifest` 和能力定义智能体及其初始工作区内容。
2. 通过向 `run()` 或 `Runner` 提供一个 `sandbox` 运行选项来执行运行，该选项可注入、恢复或创建沙盒会话。
3. 稍后从运行器管理的 `RunState`、显式的沙盒 `sessionState` 或已保存的工作区快照继续。

如果 shell 访问只是一个偶尔使用的工具，请从[工具](/openai-agents-js/zh/guides/tools)指南中的托管 shell 开始。当工作区隔离、沙盒客户端选择或沙盒会话恢复行为本身就是设计的一部分时，再使用沙盒智能体。

## 适用场景

沙盒智能体非常适合以工作区为中心的工作流，例如：

- **编码与调试**：为 GitHub 仓库中的问题报告编排自动修复，并运行有针对性的测试。
- **文档处理与编辑**：从用户的财务文档中提取信息，并创建已完成的报税表草稿。
- **基于文件的审查或分析**：在回答之前检查入职资料包、生成的报告或产物包。
- **隔离的多智能体模式**：为每个审查智能体或编码子智能体提供独立工作区。
- **多步骤工作区任务**：在一次运行中修复 bug，稍后再添加回归测试，或从快照或沙盒会话状态恢复。

如果您不需要访问文件或持续存在的文件系统，请继续使用 `Agent`。如果 shell 访问只是偶尔需要的一项能力，可添加托管 shell；如果工作区边界本身就是功能的一部分，请使用沙盒智能体。

## 选择沙盒客户端

本地开发时请先使用 `UnixLocalSandboxClient`。当您需要容器隔离或镜像一致性时，切换到 `DockerSandboxClient`。当您需要由提供方管理执行环境时，切换到托管提供方。

在大多数情况下，`SandboxAgent` 定义保持不变，只需在 `sandbox` 运行选项中修改沙盒客户端及其选项。有关本地、Docker、托管和远程挂载选项，请参阅[沙盒客户端](/openai-agents-js/zh/guides/sandbox-agents/clients)。

## 核心组成

| 层级 | 主要 SDK 组件 | 回答的问题 |
| --- | --- | --- |
| 智能体定义 | `SandboxAgent`、`Manifest`、capabilities | 将运行什么智能体，以及它应从什么样的全新会话工作区契约开始？ |
| 沙盒执行 | `sandbox` 运行选项、沙盒客户端和实时沙盒会话 | 此次运行如何获得一个实时沙盒会话，工作将在何处执行？ |
| 保存的沙盒状态 | `RunState` 的沙盒负载、`sessionState` 和 snapshots | 此工作流如何重新连接到先前的沙盒工作，或用保存的内容为新的沙盒会话提供初始数据？ |

主要 SDK 组件与这些层级的映射如下：

| 组件 | 负责内容 | 请问自己这个问题 |
| --- | --- | --- |
| `SandboxAgent` | 智能体定义 | 这个智能体应做什么，哪些默认项应随它一起携带？ |
| `Manifest` | 全新会话工作区中的文件和文件夹 | 运行开始时，文件系统中应存在哪些文件和文件夹？ |
| `Capability` | 沙盒原生行为 | 哪些工具、instructions 片段或运行时行为应附加到该智能体上？ |
| `sandbox` 运行选项 | 每次运行的沙盒客户端和沙盒会话来源 | 此次运行应注入、恢复还是创建一个沙盒会话？ |
| `RunState` | 运行器管理的已保存沙盒状态 | 我是否正在恢复一个先前由运行器管理的工作流，并自动延续它的沙盒状态？ |
| `sandbox.sessionState` | 显式序列化的沙盒会话状态 | 我是否想从已在 `RunState` 之外序列化的沙盒状态恢复？ |
| `sandbox.snapshot` | 用于新沙盒会话的已保存工作区内容 | 新沙盒会话是否应从已保存的文件和产物开始？ |

一个实用的设计顺序是：

1. 用 `Manifest` 或 manifest init 对象定义全新会话工作区契约。
2. 使用 `SandboxAgent` 定义智能体。
3. 添加内置或自定义能力。
4. 决定每次运行应如何在 `run(agent, input, { sandbox: ... })` 或 `new Runner({ sandbox: ... })` 中获取其沙盒会话。

## 沙盒运行的准备方式

在运行时，运行器会将该定义转换为一个具体的、由沙盒支持的运行：

1. 它从 `sandbox` 运行选项解析沙盒会话。
2. 它确定本次运行的有效工作区输入。
3. 它让各项能力处理生成的 manifest。
4. 它按固定顺序构建最终 instructions：SDK 的默认沙盒提示词，或者如果您显式覆盖则使用 `baseInstructions`，然后是 `instructions`，再然后是能力说明片段，接着是任何远程挂载策略文本，最后是渲染后的文件系统树。
5. 它将能力工具绑定到实时沙盒会话，并通过标准的 `run()` 和 `Runner` API 运行准备好的智能体。

沙盒机制不会改变一次 turn 的含义。turn 仍然是一个模型步骤，而不是单个 shell 命令或沙盒动作。沙盒侧操作与 turn 之间没有固定的一对一映射。实际规则是：只有当智能体运行时在沙盒工作发生后还需要模型再给出一次响应时，才会消耗另一个 turn。

## `SandboxAgent` 选项

这些是在常规 `Agent` 字段之上的沙盒专属选项：

| 选项 | 最佳用途 |
| --- | --- |
| `defaultManifest` | 运行器创建新的沙盒会话时使用的默认工作区。 |
| `instructions` | 追加在 SDK 沙盒提示词之后的额外角色、工作流和成功标准。 |
| `baseInstructions` | 用于替换 SDK 沙盒提示词的高级逃生口。 |
| `capabilities` | 应随该智能体一起携带的沙盒原生工具和行为。 |
| `runAs` | 面向模型的沙盒工具（如 shell 命令、文件读取和补丁）所使用的用户身份。 |

沙盒客户端选择、沙盒会话复用、manifest 覆盖和快照选择应放在 `sandbox` 运行选项中，而不是智能体上。

### `defaultManifest`

`defaultManifest` 是当运行器为该智能体创建新的沙盒会话时使用的默认工作区。您可以传入一个 `Manifest` 实例，或传入与 `new Manifest(...)` 相同的 init 对象。用它来指定智能体通常应从哪些文件、仓库、辅助材料、输出目录和挂载开始。

这只是默认值。某次运行可以通过 `sandbox.manifest` 覆盖它，而被复用或恢复的沙盒会话会保留其现有工作区状态。

### `instructions` 和 `baseInstructions`

对那些应在不同提示词之间持续保留的简短规则，请使用 `instructions`。在 `SandboxAgent` 中，这些 instructions 会追加在 SDK 的沙盒基础提示词之后，因此您既保留了内置的沙盒指导，又添加了自己的角色、工作流和成功标准。

只有在您想替换 SDK 沙盒基础提示词时，才使用 `baseInstructions`。大多数智能体都不应设置它。

| 放在......中 | 用途 | 示例 |
| --- | --- | --- |
| `instructions` | 智能体稳定的角色、工作流规则和成功标准。 | "检查入职文档，然后交接。"、"将最终文件写入 `output/`。" |
| `baseInstructions` | 对 SDK 沙盒基础提示词的完整替换。 | 自定义的低层沙盒包装提示词。 |
| 用户提示词 | 本次运行的一次性请求。 | "总结这个工作区。" |
| manifest 中的工作区文件 | 更长的任务说明、仓库本地 instructions 或受限参考材料。 | `repo/task.md`、文档包、示例材料包。 |

避免将用户的一次性任务复制到 `instructions` 中，避免把应放入 manifest 的长篇参考材料嵌进去，避免重复描述内置能力已注入的工具文档，也避免混入模型在运行时不需要的本地安装说明。

### `capabilities`

能力会将沙盒原生行为附加到 `SandboxAgent`。它们可以在运行开始前塑造工作区、追加沙盒专属 instructions、暴露绑定到实时沙盒会话的工具，并调整该智能体的模型行为或输入处理方式。

内置能力包括：

| 能力 | 添加时机 | 说明 |
| --- | --- | --- |
| `shell()` | 智能体需要 shell 访问时。 | 添加 `exec_command`；当沙盒客户端支持 PTY 交互时，也会添加 `write_stdin`。 |
| `filesystem()` | 智能体需要编辑文件或查看本地图像时。 | 添加 `apply_patch` 和 `view_image`；补丁路径相对于工作区根目录。 |
| `skills()` | 您希望在沙盒中进行 skill 发现和具体化时。 | 对于沙盒本地 `SKILL.md` skills，优先使用它，而不是手动挂载 `.agents` 或 `.agents/skills`。 |
| `memory()` | 后续运行应读取或生成 memory 产物时。 | 需要 `shell()`；若要实时更新，还需要 `filesystem()`。 |
| `compaction()` | 长时间运行的流程需要在 compaction 项后进行上下文裁剪时。 | 会调整模型采样和输入处理。 |

默认情况下，`SandboxAgent.capabilities` 使用 `Capabilities.default()`，其中包含 `filesystem()`、`shell()` 和 `compaction()`。如果您传入 `capabilities: [...]`，该列表会替换默认值，因此请包含您仍希望使用的默认能力。

## 概念

### Manifest

`Manifest` 描述一个全新沙盒会话的工作区。它可以设置工作区 `root`、声明文件和目录、复制本地文件、克隆 Git 仓库、附加远程存储挂载、设置环境变量、定义用户或组，并授予对工作区外特定绝对路径的访问权限。

Manifest 中的环境值默认会被持久化。对于 API key、访问令牌或其他不应随沙盒状态一起保存的短期凭据，请使用如 `{ value: "...", ephemeral: true }` 这样的临时条目。

Manifest 条目的路径相对于工作区。它们不能是绝对路径，也不能通过 `..` 逃逸出工作区，这确保了工作区契约在本地、Docker 和托管客户端之间的可移植性。

对于智能体在开始工作前所需的材料，请使用 manifest 条目：

| Manifest 条目 | 用途 |
| --- | --- |
| `file()`、`dir()` | 小型合成输入、辅助文件或输出目录。 |
| `localFile()`、`localDir()` | 应在沙盒中具体化的主机文件或目录。 |
| `gitRepo()` | 应拉取到工作区中的仓库。 |
| 如 `s3Mount()`、`gcsMount()`、`r2Mount()`、`azureBlobMount()`、`s3FilesMount()` 等 mounts | 应出现在沙盒中的外部存储。 |

挂载条目描述要暴露哪些存储；挂载策略描述沙盒后端如何附加这些存储。有关挂载选项和提供方支持情况，请参阅[沙盒客户端](/openai-agents-js/zh/guides/sandbox-agents/clients#mounts-and-remote-storage)。

### 权限

`Permissions` 控制 manifest 条目的文件系统权限。它针对的是沙盒具体化出来的文件，而不是模型权限、审批策略或 API 凭据。

用户是可以在沙盒中执行工作的身份。当您希望某个身份存在于沙盒中时，请将该用户添加到 manifest，然后在希望面向模型的沙盒工具（如 shell 命令、文件读取和补丁）以该用户身份运行时设置 `SandboxAgent.runAs`。

如果您还需要文件级共享规则，请将用户与 manifest 组以及条目的 `group` 元数据结合使用。`runAs` 用户控制谁来执行沙盒原生操作；`Permissions` 控制该用户在沙盒具体化工作区后可以读取、写入或执行哪些文件。

### SnapshotSpec

`SnapshotSpec` 告诉新的沙盒会话应从哪里恢复保存的工作区内容，以及应将内容持久化回哪里。它是沙盒工作区的快照策略，而 `sessionState` 是用于恢复特定沙盒后端的序列化连接状态。

对于本地持久快照，请使用本地 snapshots；当您的应用提供远程快照客户端时，请使用远程 snapshots。已挂载路径和临时路径不会作为持久工作区内容复制进快照中。

## `sandbox` 运行选项

`sandbox` 运行选项包含每次运行的配置，用于决定沙盒会话来自何处，以及新的会话应如何初始化。

### 沙盒来源

这些选项决定运行器应复用、恢复还是创建沙盒会话：

| 选项 | 使用场景 | 说明 |
| --- | --- | --- |
| `client` | 您希望运行器为您创建、恢复并清理沙盒会话。 | 除非您提供了实时沙盒 `session`，否则必填。 |
| `session` | 您已自行创建了一个实时沙盒会话。 | 生命周期由调用方拥有；运行器会复用该实时沙盒会话。 |
| `sessionState` | 您拥有已序列化的沙盒会话状态，但没有实时沙盒会话对象。 | 需要 `client`；运行器会以拥有方会话的方式从该显式状态恢复。 |

### 全新会话输入

这些选项仅在运行器创建新的沙盒会话时才重要：

| 选项 | 使用场景 | 说明 |
| --- | --- | --- |
| `manifest` | 您希望一次性覆盖全新会话工作区。 | 接受 `Manifest` 或 manifest init 对象。省略时回退到 `agent.defaultManifest`。 |
| `snapshot` | 新的沙盒会话应从快照中初始化。 | 适用于类似恢复的流程或远程快照客户端。 |
| `options` | 沙盒客户端需要创建时选项。 | 常见于 Docker 镜像、提供方超时等客户端特定设置。 |

`concurrencyLimits` 控制可以并行运行多少沙盒具体化工作。当大型 manifests 或本地目录复制需要更严格的资源控制时，请使用 `manifestEntries` 和 `localDirFiles`。

### 具体化控制

具体化控制有意设计为每次运行级别。将它们放在 `sandbox` 运行选项附近，这样同一个 `SandboxAgent` 就可以在大型本地目录复制时使用保守限制，在小型 manifests 时使用更宽松的限制。

当 manifest 包含大量彼此独立的条目（如文件、目录、仓库和挂载）时，请使用 `concurrencyLimits.manifestEntries`。当 `localDir()` 条目包含大量文件、需要限制本地复制压力时，请使用 `concurrencyLimits.localDirFiles`。

## 常见模式

请从上面的完整示例开始。在很多情况下，同一个 `SandboxAgent` 可以保持不变，只需更改沙盒客户端、沙盒会话来源或工作区来源。

### 切换沙盒客户端

保持智能体定义不变，只修改运行配置。当您希望获得容器隔离或镜像一致性时，请使用 Docker；当您希望由提供方管理执行环境时，请使用托管提供方。示例和提供方选项请参阅[沙盒客户端](/openai-agents-js/zh/guides/sandbox-agents/clients)。

### 覆盖工作区

保持智能体定义不变，只通过 `sandbox: { client, manifest }` 替换全新会话 manifest。当相同的智能体角色需要针对不同的仓库、材料包或任务包运行，而无需重建智能体时，可使用此方式。

### 注入沙盒会话

当您需要显式生命周期控制、运行后检查或复制输出时，请注入一个实时沙盒会话。对此次运行使用 `sandbox: { session }`，并在您的应用代码中关闭该会话。

### 从会话状态恢复

如果您已经在 `RunState` 之外序列化了沙盒状态，可通过 `sandbox: { client, sessionState }` 让运行器从该状态重新连接。当沙盒状态保存在您自己的存储或作业系统中，并且您希望 `Runner` 直接从中恢复时，请使用此方式。

### 从快照开始

使用 `sandbox: { client, snapshot }` 从保存的文件和产物为新沙盒提供初始数据。当一次新的运行应从已保存的工作区内容开始，而不仅仅是 `agent.defaultManifest` 时，请使用此方式。

### 从 Git 加载 skills

使用 `skills({ from: gitRepo(...) })` 将本地 skill 来源替换为基于仓库的来源。当 skills 包有自己的发布节奏，或应在多个沙盒之间共享时，请使用此方式。

### 作为工具暴露

工具智能体既可以拥有自己的沙盒边界，也可以复用父运行中的实时沙盒。对于一个快速的只读探索智能体，复用非常有用：它可以检查父级正在使用的完全相同的工作区，而无需为创建、填充或快照另一个沙盒付出成本。

而当工具智能体确实需要真正隔离时，请通过 `sandboxAgent.asTool(...)` 为它提供自己的 `runConfig`。当工具智能体应能自由修改、运行不受信任的命令，或使用不同的后端或镜像时，请使用独立沙盒。

### 与本地工具和 MCP 结合

在保留沙盒工作区的同时，您仍可在同一个智能体上使用常规工具。沙盒能力可以与 `tools`、`mcpServers`、交接、模型设置和输出配置共存。

## 智能体记忆

当未来的沙盒智能体运行应从先前运行中学习时，请使用 `memory()` 能力。memory 不同于 SDK 的对话式 `Session` 记忆：它会将经验提炼为沙盒工作区中的文件，后续运行则可以读取这些文件。

有关设置、读取/生成行为、多轮对话和布局隔离，请参阅[智能体记忆](/openai-agents-js/zh/guides/sandbox-agents/memory)。

## 组合模式

一旦单智能体模式清晰了，下一个设计问题就是：在更大的系统中，沙盒边界应位于何处。

沙盒智能体仍可与 SDK 的其余部分组合：

- [交接](/openai-agents-js/zh/guides/handoffs)：将文档密集型工作从非沙盒的接收智能体交给沙盒审查智能体。
- [Agents as tools](/openai-agents-js/zh/guides/tools#agents-as-tools)：将多个沙盒智能体作为工具暴露，通常是在每次 `asTool(...)` 调用中传入沙盒运行配置，以便每个工具都有自己的沙盒边界。
- [MCP 集成](/openai-agents-js/zh/guides/mcp) 和普通函数工具：沙盒能力可以与 `mcpServers` 和常规工具共存。
- [运行智能体](/openai-agents-js/zh/guides/running-agents)：沙盒运行仍使用标准的 `run()` 和 `Runner` API。

使用交接时，仍然只有一个顶层运行和一个顶层 turn 循环。活跃智能体会变化，但运行不会变成嵌套。

而使用 `asTool(...)` 时，关系则不同。外层编排器会用一个外层 turn 来决定调用该工具，而该工具调用会为沙盒智能体启动一个嵌套运行。嵌套运行有自己的 turn 循环、`maxTurns`、审批，并且通常也有自己的沙盒运行配置。从外层编排器的角度看，所有这些工作仍然都封装在一次工具调用之后，因此嵌套 turn 不会增加外层运行的 turn 计数。
