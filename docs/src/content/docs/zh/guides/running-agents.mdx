## Runner 生命周期与配置

### 智能体循环

当你在 Runner 中使用 run 方法时，需要传入一个起始智能体和输入。输入可以是字符串（会被视为用户消息），也可以是输入项列表（即 OpenAI Responses API 中的 items）。

随后 runner 会执行一个循环：

1. 使用当前输入调用当前智能体的模型。
2. 检查 LLM 响应。
   - **Final output** → 返回。
   - **Handoff** → 切换到新智能体，保留累计对话历史，回到步骤 1。
   - **Tool calls** → 执行工具，将结果追加到对话中，回到步骤 1。
3. 当达到 `maxTurns` 时，抛出 [`MaxTurnsExceededError`](/openai-agents-js/openai/agents-core/classes/maxturnsexceedederror)。

<Aside type="note">
  判断 LLM 输出是否被视为“final output”的规则是：
  产出了符合目标类型的文本输出，且没有 tool calls。
</Aside>

#### Runner 生命周期

在应用启动时创建一个 `Runner`，并在后续请求中复用它。该实例会保存全局配置，例如模型提供方和追踪选项。只有在你需要完全不同的配置时，才创建另一个 `Runner`。对于简单脚本，你也可以直接调用 `run()`，它内部会使用默认 runner。

### 运行参数

`run()` 方法的输入包括：用于启动运行的初始智能体、运行输入，以及一组选项。

输入可以是字符串（会被视为用户消息）、[input items](/openai-agents-js/openai/agents-core/type-aliases/agentinputitem) 列表，或者在你构建 [human-in-the-loop](/openai-agents-js/guides/human-in-the-loop) 智能体时传入 [`RunState`](/openai-agents-js/openai/agents-core/classes/runstate) 对象。

额外选项如下：

| Option                  | Default | Description                                                                                                                       |
| ----------------------- | ------- | --------------------------------------------------------------------------------------------------------------------------------- |
| `stream`                | `false` | 若为 `true`，调用将返回 `StreamedRunResult`，并在模型返回事件时持续发出事件。                                                     |
| `context`               | –       | 传递给每个 tool / guardrail / handoff 的上下文对象。详见 [context guide](/openai-agents-js/guides/context)。                      |
| `maxTurns`              | `10`    | 安全上限——达到后抛出 [`MaxTurnsExceededError`](/openai-agents-js/openai/agents-core/classes/maxturnsexceedederror)。              |
| `signal`                | –       | 用于取消的 `AbortSignal`。                                                                                                        |
| `session`               | –       | 会话持久化实现。参见 [sessions guide](/openai-agents-js/guides/sessions)。                                                        |
| `sessionInputCallback`  | –       | 自定义会话历史与新输入的合并逻辑；在模型调用前执行。参见 [sessions](/openai-agents-js/guides/sessions)。                          |
| `callModelInputFilter`  | –       | 在调用模型前，编辑模型输入（items + 可选 instructions）的钩子。参见 [Call model input filter](#call-model-input-filter)。         |
| `toolErrorFormatter`    | –       | 自定义返回给模型的工具审批拒绝消息的钩子。参见 [Tool error formatter](#tool-error-formatter)。                                    |
| `reasoningItemIdPolicy` | –       | 控制在将先前运行项转换回模型输入时，是否保留 reasoning-item 的 `id`。参见 [Reasoning item ID policy](#reasoning-item-id-policy)。 |
| `tracing`               | –       | 按次运行覆盖追踪配置（例如导出 API key）。                                                                                        |
| `errorHandlers`         | –       | 受支持运行时错误的处理器（当前为 `maxTurns`）。参见 [Error handlers](#error-handlers)。                                           |
| `conversationId`        | –       | 复用服务端会话（仅 OpenAI Responses API + Conversations API）。                                                                   |
| `previousResponseId`    | –       | 在不创建会话的情况下，从上一次 Responses API 调用继续（仅 OpenAI Responses API）。                                                |

### 流式传输

流式传输可让你在 LLM 运行时额外接收流式事件。流开始后，`StreamedRunResult` 将包含本次运行的完整信息，包括所有新产生的输出。你可以通过 `for await` 循环迭代流式事件。更多内容见 [streaming guide](/openai-agents-js/guides/streaming)。

| `modelSettings`             | `ModelSettings`          | 覆盖各智能体设置的全局调优参数。                                              |
| `handoffInputFilter`        | `HandoffInputFilter`     | 执行交接时变更输入项（若交接本身未定义该过滤器）。                            |

| `outputGuardrails`          | `OutputGuardrail[]`      | 应用于*最终*输出的护栏。                                                      |
| `tracingDisabled`           | `boolean`                | 完全禁用 OpenAI 追踪。                                                        |

| `tracing`                   | `TracingConfig`          | 按次运行覆盖追踪配置（例如导出 API key）。                                    |

| `callModelInputFilter`      | `CallModelInputFilter`   | 在每次模型调用前编辑模型输入的全局钩子。                                      |
| `toolErrorFormatter`        | `ToolErrorFormatter`     | 自定义返回给模型的工具审批拒绝消息的全局钩子。                                |
| `reasoningItemIdPolicy`     | `ReasoningItemIdPolicy`  | 在将已生成项回放到后续模型调用时，保留或省略 reasoning-item `id` 的默认策略。 |

## 状态与对话管理

### 内存策略选择

将状态带入下一轮常见有四种方式：

| Strategy             | Where state lives         | Best for                                                                 | What you pass on the next turn                        |
| -------------------- | ------------------------- | ------------------------------------------------------------------------ | ----------------------------------------------------- |
| `result.history`     | Your app memory           | Small chat loops, full manual control, any provider                      | `result.history`                                      |
| `session`            | Your storage + the SDK    | Persistent chat state, resumable runs, custom stores                     | The same `session` instance (or a store-backed one)   |
| `conversationId`     | OpenAI Conversations API  | Shared server-side state across workers/services                         | The same `conversationId` plus only the new user turn |
| `previousResponseId` | OpenAI Responses API only | The simplest server-managed continuation without creating a conversation | `result.lastResponseId` plus only the new user turn   |

`result.history` 和 `session` 由客户端管理。`conversationId` 和 `previousResponseId` 由 OpenAI 管理，且仅在使用 OpenAI Responses API 时适用。在大多数应用中，每个会话选择一种持久化策略即可。若将客户端管理历史与服务端管理状态混用，除非你有意协调两层数据，否则可能造成上下文重复。

#### 服务端管理对话

你可以让 OpenAI Responses API 为你持久化对话历史，而不必在每轮都发送完整本地历史。当你在协调长对话或多个服务时，这很有用。使用下面任一服务端管理方式时，每次请求只需传入新一轮输入，API 会自动复用先前状态。详情见 [Conversation state guide](https://platform.openai.com/docs/guides/conversation-state?api-mode=responses)。

## 钩子与自定义

### 调用模型输入过滤器

使用 `callModelInputFilter` 可在调用模型**之前**编辑模型输入。该钩子会接收当前智能体、上下文以及合并后的输入项（包含会话历史，如有）。你可以返回更新后的 `input` 数组和可选 `instructions`，用于脱敏、删除旧消息或注入额外系统引导。

可按次运行设置（`runner.run(..., { callModelInputFilter })`），也可在 `Runner` 配置中设为默认（`RunConfig` 中的 `callModelInputFilter`）。

返回值必须是 `ModelInputData` 对象：`{ input: AgentInputItem[], instructions? }`。
`input` 字段是必需的，且必须是数组。返回其他结构会抛出 `UserError`。

SDK 在调用过滤器前会克隆准备好的轮次输入。如果你也在使用 `session`，被过滤后的克隆内容会被持久化，因此这里执行的脱敏或截断也会反映到存储的会话历史中。

使用 `conversationId` 或 `previousResponseId` 时，该钩子会作用于下一次 Responses API 调用的已准备 payload。此前由服务端管理的上下文由 API 恢复，因此该次调用被过滤的数组可能只表示新轮次增量，而非完整历史回放。若你需要在最终过滤步骤前改变“已存储历史 + 当前轮次”的合并方式，请使用 `sessionInputCallback`。

### 工具错误格式化器

使用 `toolErrorFormatter` 可自定义当工具调用被拒绝时返回给模型的审批拒绝消息。这样你可以返回领域特定措辞（例如合规指引），而不是 SDK 默认消息。

该格式化器可按次运行设置（`runner.run(..., { toolErrorFormatter })`），也可在 `RunConfig` 中全局设置（`new Runner(...)` 的 `toolErrorFormatter`）。

当前格式化器会在 `approval_rejected` 事件中运行，并接收：

- `kind`（当前始终为 `'approval_rejected'`）
- `toolType`（`'function'`、`'computer'`、`'shell'` 或 `'apply_patch'`）
- `toolName`
- `callId`
- `defaultMessage`（SDK 回退消息）
- `runContext`

返回字符串可覆盖消息；返回 `undefined` 则保留 SDK 默认值。若格式化器抛错（或返回非字符串值），SDK 会记录 warning，并回退到默认审批拒绝消息。

### Reasoning 项 ID 策略

使用 `reasoningItemIdPolicy` 可控制：当 SDK 将先前生成的运行项转换回 `AgentInputItem[]` 作为后续模型输入时，reasoning 项是否保留其 `id` 字段。

这会影响 SDK 将已生成模型项回放为输入的场景，例如：

- 同一次运行中的后续模型调用（如工具执行后）；
- 复用已生成项作为输入/历史的后续轮次；
- 从已保存 `RunState` 恢复运行；
- 派生结果视图，如 `result.history` / `result.output`（它们是模型输入形状数组）。
- `'preserve'`（默认）保留 reasoning-item ID。
- `'omit'` 在将 reasoning 项作为输入发回前移除 `id` 字段。
- 非 reasoning 项不受影响。

该策略**不会**改变：

- 原始模型响应（`result.rawResponses`）；
- 运行项（`result.newItems`）；
- provider 返回的模型当前轮次输出。

换句话说，该策略仅在 SDK 基于先前生成项构建**下一次输入**时生效。

你可以按次运行设置该策略（`runner.run(..., { reasoningItemIdPolicy: 'omit' })`），也可设为 runner 默认值（`new Runner({ reasoningItemIdPolicy: 'omit', ... })`）。从已保存 `RunState` 恢复时，会复用先前已解析的策略，除非你显式覆盖。

#### 与 `callModelInputFilter` 的交互

`reasoningItemIdPolicy` 会先于 `callModelInputFilter` 应用。若你需要自定义行为，`callModelInputFilter` 仍可检查准备好的输入，并在模型调用前手动重新引入或移除 reasoning ID。

#### 何时使用 `'omit'`

当你希望回放的 reasoning 项在无 ID 的情况下标准化时，请使用 `'omit'`（例如，让转发/回放的模型输入更简洁，或满足你应用流水线的集成要求）。

如果你的后端/provider 拒绝包含回放 reasoning 项的请求并报校验错误（例如后续输入中 reasoning item ID 相关的 HTTP `400` 错误），这也是一个有用的排障选项。在这些情况下，使用 `'omit'` 去除回放 reasoning ID，可避免发送被后端视为新请求无效的 ID。

如果你希望 SDK 在回放输入中保留 reasoning-item ID，且你的集成可接受它们，请保持 `'preserve'`。

## 错误与恢复

### 错误处理器

使用 `errorHandlers` 可将受支持的运行时错误转换为最终输出，而不是抛出异常。当前仅支持 `maxTurns`。

- `errorHandlers.maxTurns` 仅处理最大轮次错误。
- `errorHandlers.default` 用作受支持类型的兜底处理。
- 处理器接收 `{ error, context, runData }`，并可返回 `{ finalOutput, includeInHistory? }`。

### 异常

SDK 会抛出一小组可捕获错误：

- [`MaxTurnsExceededError`](/openai-agents-js/openai/agents-core/classes/maxturnsexceedederror) – 达到 `maxTurns`。
- [`ModelBehaviorError`](/openai-agents-js/openai/agents-core/classes/modelbehaviorerror) – 模型产出无效输出（如格式错误 JSON、未知工具）。
- [`InputGuardrailTripwireTriggered`](/openai-agents-js/openai/agents-core/classes/inputguardrailtripwiretriggered) / [`OutputGuardrailTripwireTriggered`](/openai-agents-js/openai/agents-core/classes/outputguardrailtripwiretriggered) – 护栏违规。
- [`ToolInputGuardrailTripwireTriggered`](/openai-agents-js/openai/agents-core/classes/toolinputguardrailtripwiretriggered) / [`ToolOutputGuardrailTripwireTriggered`](/openai-agents-js/openai/agents-core/classes/tooloutputguardrailtripwiretriggered) – 工具护栏违规。
- [`GuardrailExecutionError`](/openai-agents-js/openai/agents-core/classes/guardrailexecutionerror) – 护栏执行未完成。
- [`ToolTimeoutError`](/openai-agents-js/openai/agents-core/classes/tooltimeouterror) – 函数工具超过 `timeoutMs` 且使用了 `timeoutBehavior: 'raise_exception'`。
- [`ToolCallError`](/openai-agents-js/openai/agents-core/classes/toolcallerror) – 函数工具执行因非超时错误失败。
- [`UserError`](/openai-agents-js/openai/agents-core/classes/usererror) – 基于配置或用户输入抛出的任意错误。

这些错误都继承自 `AgentsError` 基类，该类可能提供 `state` 属性以访问当前运行状态。
