## Runner 生命周期与配置

### 智能体循环

当您使用 Runner 中的 run 方法时，需要传入一个起始智能体和输入。输入可以是字符串（会被视为用户消息），也可以是输入项列表，即 OpenAI Responses API 中的项目。

随后 runner 会运行一个循环：

1. 使用当前输入调用当前智能体的模型。
2. 检查 LLM 响应。
   - **最终输出** → 返回。
   - **交接** → 切换到新的智能体，保留累积的对话历史，转到 1。
   - **工具调用** → 执行工具，将其结果追加到对话中，转到 1。
3. 当达到 `maxTurns` 时抛出 [`MaxTurnsExceededError`](/openai-agents-js/openai/agents-core/classes/maxturnsexceedederror)，除非 `maxTurns` 为 `null`。

<Aside type="note">
  判断 LLM 输出是否被视为"最终输出"的规则是：
  它会产生所需类型的文本输出，并且没有工具调用。
</Aside>

#### Runner 生命周期

在应用启动时创建一个 `Runner`，并在各个请求之间复用它。该实例会存储全局配置，例如模型提供方和追踪选项。只有在需要完全不同的设置时，才创建另一个 `Runner`。对于简单脚本，也可以调用 `run()`，它会在内部使用默认 runner。

### 运行参数

传给 `run()` 方法的输入包括：用于开始运行的初始智能体、运行输入，以及一组选项。

输入可以是字符串（会被视为用户消息），也可以是[输入项](/openai-agents-js/openai/agents-core/type-aliases/agentinputitem)列表；如果您正在构建[人机协作](/openai-agents-js/zh/guides/human-in-the-loop)智能体，也可以传入 [`RunState`](/openai-agents-js/openai/agents-core/classes/runstate) 对象。

其他选项如下：

| 选项 | 默认值 | 描述 |
| --- | --- | --- |
| `stream` | `false` | 如果为 `true`，调用会返回 `StreamedRunResult`，并在事件从模型到达时发出这些事件。 |
| `context` | – | 转发给每个工具 / 护栏 / 交接的上下文对象。更多内容请参阅[上下文管理](/openai-agents-js/zh/guides/context)。 |
| `maxTurns` | `10` | 安全限制——达到时会抛出 [`MaxTurnsExceededError`](/openai-agents-js/openai/agents-core/classes/maxturnsexceedederror)。传入 `null` 可禁用该限制。 |
| `signal` | – | 用于取消的 `AbortSignal`。 |
| `session` | – | 会话持久化实现。请参阅[会话](/openai-agents-js/zh/guides/sessions)。 |
| `sessionInputCallback` | – | 用于会话历史和新输入的自定义合并逻辑；在模型调用之前运行。请参阅[会话](/openai-agents-js/zh/guides/sessions)。 |
| `callModelInputFilter` | – | 在即将调用模型前编辑模型输入（items + 可选 instructions）的钩子。请参阅[调用模型输入过滤器](#call-model-input-filter)。 |
| `toolErrorFormatter` | – | 用于自定义返回给模型的工具错误消息的钩子。请参阅[工具错误格式化器](#tool-error-formatter)。 |
| `reasoningItemIdPolicy` | – | 控制在将之前运行的项目重新转换为模型输入时，是否保留推理项的 `id`。请参阅[推理项 ID 策略](#reasoning-item-id-policy)。 |
| `tracing` | – | 每次运行的追踪配置覆盖项（例如导出 API 密钥）。 |
| `sandbox` | – | 沙盒客户端、实时会话、会话状态、快照、manifest 覆盖项，或 `SandboxAgent` 运行的并发限制。请参阅[概念](/openai-agents-js/zh/guides/sandbox-agents/concepts)。 |
| `toolExecution` | – | 本地工具调用的 SDK 侧执行设置。使用 `toolExecution.maxFunctionToolConcurrency` 限制同时运行的函数工具数量，并使用 `toolExecution.preApprovalInputGuardrails` 在待审批请求之前运行函数工具输入护栏。 |
| `toolNotFoundBehavior` | `'raise_error'` | 控制模型发出的未解析函数工具调用。使用 `'return_error_to_model'` 可返回模型可见的工具错误并继续运行。 |
| `errorHandlers` | – | 受支持运行时错误的处理器（当前为 `maxTurns`）。请参阅[错误处理器](#error-handlers)。 |
| `conversationId` | – | 复用服务器端对话（仅限 OpenAI Responses API + Conversations API）。 |
| `previousResponseId` | – | 不创建对话而从上一个 Responses API 调用继续（仅限 OpenAI Responses API）。 |

### 流式传输

流式传输允许您在 LLM 运行时额外接收流式事件。流开始后，`StreamedRunResult` 将包含本次运行的完整信息，包括生成的所有新输出。您可以使用 `for await` 循环迭代流式事件。更多内容请参阅[流式传输](/openai-agents-js/zh/guides/streaming)。

### 运行配置

如果您要创建自己的 `Runner` 实例，可以传入一个 `RunConfig` 对象来配置 runner。

| 字段 | 类型 | 用途 |
| --- | --- | --- |
| `model` | `string \| Model` | 为运行中的**所有**智能体强制指定特定模型。 |
| `modelProvider` | `ModelProvider` | 解析模型名称——默认使用 OpenAI 提供方。 |
| `modelSettings` | `ModelSettings` | 覆盖每个智能体设置的全局调优参数。详情请参阅[模型](/openai-agents-js/zh/guides/models#modelsettings)，包括可选择启用的重试配置。 |
| `handoffInputFilter` | `HandoffInputFilter` | 执行交接时修改输入项（如果交接本身尚未定义）。 |
| `inputGuardrails` | `InputGuardrail[]` | 应用于*初始*用户输入的护栏。 |
| `outputGuardrails` | `OutputGuardrail[]` | 应用于*最终*输出的护栏。 |
| `tracingDisabled` | `boolean` | 完全禁用 OpenAI 追踪。 |
| `traceIncludeSensitiveData` | `boolean` | 在仍发出 span 的同时，从追踪中排除 LLM/工具输入和输出。 |
| `workflowName` | `string` | 出现在 Traces 仪表板中——帮助对相关运行分组。 |
| `traceId` / `groupId` | `string` | 手动指定 trace 或 group ID，而不是让 SDK 生成。 |
| `traceMetadata` | `Record<string, string>` | 附加到每个 span 的任意元数据。 |
| `tracing` | `TracingConfig` | 每次运行的追踪覆盖项（例如导出 API 密钥）。 |
| `sessionInputCallback` | `SessionInputCallback` | 此 runner 上所有运行的默认历史合并策略。 |
| `callModelInputFilter` | `CallModelInputFilter` | 每次模型调用前编辑模型输入的全局钩子。 |
| `toolErrorFormatter` | `ToolErrorFormatter` | 自定义返回给模型的工具错误消息的全局钩子。 |
| `reasoningItemIdPolicy` | `ReasoningItemIdPolicy` | 在将生成项重放到后续模型调用时，保留或省略推理项 `id` 的默认策略。 |
| `sandbox` | `SandboxRunConfig` | `SandboxAgent` 运行的默认沙盒运行时配置。 |
| `toolExecution` | `ToolExecutionConfig` | 本地工具调用的默认 SDK 侧执行设置。`maxFunctionToolConcurrency` 会限制每个回合的本地函数工具并发数；未设置或为 `null` 时，会启动该回合中发出的所有函数工具调用。`preApprovalInputGuardrails` 选择启用在待审批请求之前运行函数工具输入护栏。 |
| `toolNotFoundBehavior` | `ToolNotFoundBehavior` | 未解析函数工具调用的默认行为。`'raise_error'` 会抛出 `ModelBehaviorError`；`'return_error_to_model'` 会返回模型可见的工具错误，并让运行继续。 |

`toolExecution.maxFunctionToolConcurrency` 必须是大于等于 `1` 的整数。此设置仅限制本地函数工具的 SDK 侧执行。它不会更改提供方侧的 `modelSettings.parallelToolCalls`。

`toolExecution.preApprovalInputGuardrails` 默认禁用。当设为 `true` 时，需要审批的本地函数工具会在 SDK 记录待审批中断之前运行其输入护栏。如果护栏返回 `rejectContent`，SDK 会将该拒绝消息作为工具输出发回，而不是请求审批。如果护栏允许该调用，仍会发起审批请求；审批解决后，在工具执行前，同样的输入护栏会立即再次运行。

## 状态与对话管理

### 记忆策略选择

有四种常见方式可以将状态带入下一回合：

| 策略 | 状态所在位置 | 最适合 | 下一回合传入的内容 |
| --- | --- | --- | --- |
| `result.history` | 您的应用内存 | 小型聊天循环、完全手动控制、任何提供方 | `result.history` |
| `session` | 您的存储 + SDK | 持久聊天状态、可恢复的运行、自定义存储 | 相同的 `session` 实例（或由存储支持的实例） |
| `conversationId` | OpenAI Conversations API | 跨 worker/服务的共享服务器端状态 | 相同的 `conversationId`，并且只传入新的用户回合 |
| `previousResponseId` | 仅 OpenAI Responses API | 无需创建对话的最简单服务器管理延续方式 | `result.lastResponseId`，并且只传入新的用户回合 |

`result.history` 和 `session` 由客户端管理。`conversationId` 和 `previousResponseId` 由 OpenAI 管理，并且只在使用 OpenAI Responses API 时适用。在大多数应用中，每个对话选择一种持久化策略。将客户端管理的历史与服务器管理的状态混用，可能会重复上下文，除非您有意在协调这两层。

沙盒智能体还会增加另一层状态：实时沙盒工作区。使用常规 SDK 的 `session`、`conversationId` 或 `previousResponseId` 管理对话历史，并使用 `sandbox.session`、`sandbox.sessionState`、`RunState` 或快照管理沙盒文件系统状态。有关工作区生命周期，请参阅[概念](/openai-agents-js/zh/guides/sandbox-agents/concepts)。

#### 服务器管理的对话

您可以让 OpenAI Responses API 为您持久化对话历史，而不是在每个回合都发送完整的本地对话历史。当您在协调长对话或多个服务时，这很有用。对于下面任一服务器管理方式，每个请求只传入新回合的输入。API 会为您复用之前的状态。详情请参阅[对话状态指南](https://developers.openai.com/api/docs/guides/conversation-state)。

##### 1. 用于整个对话的 `conversationId`

您可以先使用 [Conversations API](https://platform.openai.com/docs/api-reference/conversations/create) 创建一次对话，然后在每个回合复用其 ID。SDK 会自动只包含新生成的项目。

## 钩子与自定义

### 调用模型输入过滤器

使用 `callModelInputFilter` 就在模型调用*之前*编辑模型输入。此钩子接收当前智能体、上下文和合并后的输入项（如果存在 session，也包括会话历史）。返回更新后的 `input` 数组和可选 `instructions`，以便脱敏敏感数据、丢弃旧消息，或注入额外的系统指导。

可以在每次运行中通过 `runner.run(..., { callModelInputFilter })` 设置，也可以在 `Runner` 配置中作为默认值设置（`RunConfig` 中的 `callModelInputFilter`）。

返回值必须是 `ModelInputData` 对象：`{ input: AgentInputItem[], instructions? }`。`input` 字段是必需的，且必须是数组。返回任何其他结构都会抛出 `UserError`。

SDK 会在调用过滤器前克隆准备好的本回合输入。如果您也使用 `session`，被过滤后的克隆会被持久化，因此此处应用的脱敏或截断也会反映在已存储的会话历史中。

使用 `conversationId` 或 `previousResponseId` 时，该钩子会在下一次 Responses API 调用的准备好载荷上运行。之前由服务器管理的上下文会由 API 恢复，因此该调用的过滤后数组可能已经只表示新回合的增量，而不是对早期历史的完整重放。如果您需要在此最终过滤步骤之前更改已存储历史与当前回合的合并方式，请使用 `sessionInputCallback`。

### 工具错误格式化器

使用 `toolErrorFormatter` 自定义发送回模型的工具错误消息。这样您可以返回特定领域的措辞（例如合规指引），而不是 SDK 默认消息。

格式化器可以按每次运行设置（`runner.run(..., { toolErrorFormatter })`），也可以在 `RunConfig` 中全局设置（`new Runner(...)` 中的 `toolErrorFormatter`）。

此格式化器是审批拒绝的全局兜底。如果您使用 `result.state.reject(interruption, { message: '...' })` 拒绝某个特定中断，则该次调用的 `message` 优先于 `toolErrorFormatter`。如果两者都未提供，SDK 会回退到默认拒绝文本：`Tool execution was not approved.`

当 `toolNotFoundBehavior: 'return_error_to_model'` 将未解析的函数工具调用转换为模型可见的工具输出时，也会运行此格式化器。在这种情况下，默认消息为 `Tool '<name>' not found.`

格式化器接收：

- `kind`（`'approval_rejected'` 或 `'tool_not_found'`）
- `toolType`（`'function'`、`'computer'`、`'shell'` 或 `'apply_patch'`）
- `toolName`
- `callId`
- `defaultMessage`（当前错误类别的 SDK 兜底消息）
- `runContext`

返回字符串可覆盖消息；返回 `undefined` 则保留 SDK 默认值。如果格式化器抛出异常（或返回非字符串值），SDK 会记录警告，并回退到当前错误类别的默认消息。

### 推理项 ID 策略

使用 `reasoningItemIdPolicy` 控制当 SDK 将此前生成的运行项目转换回 `AgentInputItem[]` 作为后续模型输入时，推理项是否保留其 `id` 字段。

这会影响 SDK 将生成的模型项目重放为输入的场景，例如：

- 同一次运行中的后续模型调用（例如工具执行之后），
- 后续回合复用生成项目作为输入/历史，
- 从已保存的 `RunState` 恢复的运行，
- 派生的执行结果视图，如 `result.history` / `result.output`（它们是形如模型输入的数组）。
- `'preserve'`（默认）保留推理项 ID。
- `'omit'` 会在推理项被发送回输入之前移除其 `id` 字段。
- 非推理项不受影响。

这**不会**改变：

- 原始模型响应（`result.rawResponses`），
- 运行项目（`result.newItems`），
- 提供方返回的模型当前回合输出。

换言之，该策略适用于 SDK 从之前生成的项目构建**下一个输入**的时候。

您可以按每次运行设置该策略（`runner.run(..., { reasoningItemIdPolicy: 'omit' })`），也可以将其设为 runner 默认值（`new Runner({ reasoningItemIdPolicy: 'omit', ... })`）。从已保存的 `RunState` 恢复时，除非您覆盖，否则会复用之前已解析的策略。

#### 与 `callModelInputFilter` 的交互

`reasoningItemIdPolicy` 会在 `callModelInputFilter` 之前应用。如果需要自定义行为，`callModelInputFilter` 仍可检查准备好的输入，并在模型调用前手动重新引入或移除推理 ID。

#### `'omit'` 的使用场景

当您希望重放的推理项在不带 ID 的情况下被规范化时，请使用 `'omit'`（例如，让转发/重放的模型输入更简单，或匹配应用流水线中的集成要求）。

如果您的后端/提供方因请求校验错误而拒绝重放的推理项（例如与后续输入中的推理项 ID 相关的 HTTP `400` 错误），这也是一个有用的故障排除选项。在这些情况下，使用 `'omit'` 去除重放的推理 ID，可以避免发送后端认为对新请求无效的 ID。

如果您希望 SDK 在重放输入中携带推理项 ID，且您的集成接受这些 ID，请保留 `'preserve'`。

## 错误与恢复

### 错误处理器

使用 `errorHandlers` 将受支持的运行时错误转换为最终输出，而不是抛出异常。受支持的键为 `maxTurns` 和 `modelRefusal`。

- `errorHandlers.maxTurns` 只处理最大回合数错误。
- `errorHandlers.modelRefusal` 处理以 [`ModelRefusalError`](/openai-agents-js/openai/agents-core/classes/modelrefusalerror) 形式暴露的模型拒绝。
- `errorHandlers.default` 用作受支持类别的兜底。
- 处理器接收 `{ error, context, runData }`，并可返回 `{ finalOutput, includeInHistory? }`。

### 异常

SDK 会抛出少量您可以捕获的错误：

- [`MaxTurnsExceededError`](/openai-agents-js/openai/agents-core/classes/maxturnsexceedederror) – 达到 `maxTurns`。
- [`ModelBehaviorError`](/openai-agents-js/openai/agents-core/classes/modelbehaviorerror) – 模型生成了无效输出（例如格式错误的 JSON、未知工具）。
- [`ModelRefusalError`](/openai-agents-js/openai/agents-core/classes/modelrefusalerror) – 模型拒绝生成请求的输出。
- [`InputGuardrailTripwireTriggered`](/openai-agents-js/openai/agents-core/classes/inputguardrailtripwiretriggered) / [`OutputGuardrailTripwireTriggered`](/openai-agents-js/openai/agents-core/classes/outputguardrailtripwiretriggered) – 护栏违规。
- [`ToolInputGuardrailTripwireTriggered`](/openai-agents-js/openai/agents-core/classes/toolinputguardrailtripwiretriggered) / [`ToolOutputGuardrailTripwireTriggered`](/openai-agents-js/openai/agents-core/classes/tooloutputguardrailtripwiretriggered) – 工具护栏违规。
- [`GuardrailExecutionError`](/openai-agents-js/openai/agents-core/classes/guardrailexecutionerror) – 护栏未能完成执行。
- [`ToolTimeoutError`](/openai-agents-js/openai/agents-core/classes/tooltimeouterror) – 函数工具超过 `timeoutMs`，并使用了 `timeoutBehavior: 'raise_exception'`。
- [`ToolCallError`](/openai-agents-js/openai/agents-core/classes/toolcallerror) – 函数工具执行因非超时错误失败。
- [`UserError`](/openai-agents-js/openai/agents-core/classes/usererror) – 基于配置或用户输入抛出的任何错误。
