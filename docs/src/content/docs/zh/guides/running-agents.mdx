## Runner 生命周期与配置

### 智能体循环

当您使用 Runner 中的 run 方法时，需要传入一个起始智能体和输入。输入可以是字符串（会被视为一条用户消息），也可以是输入项列表，这些输入项对应 OpenAI Responses API 中的项目。

随后，runner 会执行一个循环：

1. 使用当前输入调用当前智能体的模型。
2. 检查 LLM 响应。
   - **最终输出** → 返回。
   - **交接** → 切换到新的智能体，保留累计的对话历史，然后回到 1。
   - **工具调用** → 执行工具，将其结果追加到对话中，然后回到 1。
3. 当达到 `maxTurns` 时，抛出 [`MaxTurnsExceededError`](/openai-agents-js/openai/agents-core/classes/maxturnsexceedederror)。

<Aside type="note">
  判断 LLM 输出是否被视为“最终输出”的规则是：
  它会生成符合所需类型的文本输出，且没有任何工具调用。
</Aside>

#### Runner 生命周期

在应用启动时创建一个 `Runner`，并在多个请求之间复用它。该实例会保存全局配置，例如模型提供方和追踪选项。只有在您需要完全不同的设置时，才应创建另一个 `Runner`。对于简单脚本，您也可以直接调用 `run()`，它会在内部使用一个默认 runner。

### 运行参数

`run()` 方法的输入包括：一个用于启动运行的初始智能体、本次运行的输入，以及一组选项。

输入可以是字符串（会被视为一条用户消息）、[输入项](/openai-agents-js/openai/agents-core/type-aliases/agentinputitem)列表，或者在构建[人机协作](/openai-agents-js/zh/guides/human-in-the-loop)智能体时传入一个 [`RunState`](/openai-agents-js/openai/agents-core/classes/runstate) 对象。

额外选项包括：

| Option | Default | Description |
| --- | --- | --- |
| `stream` | `false` | 如果为 `true`，调用会返回一个 `StreamedRunResult`，并在模型返回事件时持续发出事件。 |
| `context` | – | 会传递给每个工具 / 护栏 / 交接的上下文对象。更多信息见[上下文管理指南](/openai-agents-js/zh/guides/context)。 |
| `maxTurns` | `10` | 安全限制——达到时会抛出 [`MaxTurnsExceededError`](/openai-agents-js/openai/agents-core/classes/maxturnsexceedederror)。 |
| `signal` | – | 用于取消的 `AbortSignal`。 |
| `session` | – | 会话持久化实现。参见[会话指南](/openai-agents-js/zh/guides/sessions)。 |
| `sessionInputCallback` | – | 用于合并会话历史与新输入的自定义逻辑；在模型调用前运行。参见[会话](/openai-agents-js/zh/guides/sessions)。 |
| `callModelInputFilter` | – | 在调用模型前、用于编辑模型输入（项目 + 可选 instructions）的钩子。参见[模型输入过滤](#call-model-input-filter)。 |
| `toolErrorFormatter` | – | 用于自定义返回给模型的工具审批拒绝消息的钩子。参见[工具错误格式化](#tool-error-formatter)。 |
| `reasoningItemIdPolicy` | – | 控制在先前运行项目被重新转换为模型输入时，是否保留或省略 reasoning-item 的 `id`。参见[推理项 ID 策略](#reasoning-item-id-policy)。 |
| `tracing` | – | 每次运行的追踪配置覆盖项（例如导出 API key）。 |
| `sandbox` | – | 用于 `SandboxAgent` 运行的 Sandbox client、实时会话、会话状态、快照、manifest 覆盖项或并发限制。参见[Sandbox agents](/openai-agents-js/zh/guides/sandbox-agents/concepts)。 |
| `errorHandlers` | – | 支持的运行时错误的处理器（当前为 `maxTurns`）。参见[错误处理器](#error-handlers)。 |
| `conversationId` | – | 复用服务器端对话（仅适用于 OpenAI Responses API + Conversations API）。 |
| `previousResponseId` | – | 在不创建对话的情况下，从上一次 Responses API 调用继续（仅适用于 OpenAI Responses API）。 |

### 流式传输

流式传输使您能够在 LLM 运行时额外接收流式事件。流开始后，`StreamedRunResult` 将包含本次运行的完整信息，包括所有新生成的输出。您可以使用 `for await` 循环遍历这些流式事件。更多内容请阅读[流式传输指南](/openai-agents-js/zh/guides/streaming)。

### 运行配置

如果您要创建自己的 `Runner` 实例，可以传入一个 `RunConfig` 对象来配置 runner。

| Field | Type | Purpose |
| --- | --- | --- |
| `model` | `string \| Model` | 为本次运行中的**所有**智能体强制指定一个模型。 |
| `modelProvider` | `ModelProvider` | 解析模型名称——默认使用 OpenAI provider。 |
| `modelSettings` | `ModelSettings` | 全局调优参数，会覆盖每个智能体自身的设置。详情参见[模型指南](/openai-agents-js/zh/guides/models#modelsettings)，其中也包括可选启用的重试配置。 |
| `handoffInputFilter` | `HandoffInputFilter` | 在执行交接时修改输入项（如果交接本身未定义过滤器）。 |
| `inputGuardrails` | `InputGuardrail[]` | 应用于*初始*用户输入的护栏。 |
| `outputGuardrails` | `OutputGuardrail[]` | 应用于*最终*输出的护栏。 |
| `tracingDisabled` | `boolean` | 完全禁用 OpenAI Tracing。 |
| `traceIncludeSensitiveData` | `boolean` | 在仍然发出 span 的同时，排除追踪中的 LLM / 工具输入与输出。 |
| `workflowName` | `string` | 会显示在 Traces 仪表盘中——有助于对相关运行进行分组。 |
| `traceId` / `groupId` | `string` | 手动指定 trace 或 group ID，而不是让 SDK 自动生成。 |
| `traceMetadata` | `Record<string, string>` | 附加到每个 span 的任意元数据。 |
| `tracing` | `TracingConfig` | 每次运行的追踪覆盖配置（例如导出 API key）。 |
| `sessionInputCallback` | `SessionInputCallback` | 此 runner 上所有运行的默认历史合并策略。 |
| `callModelInputFilter` | `CallModelInputFilter` | 在每次模型调用前编辑模型输入的全局钩子。 |
| `toolErrorFormatter` | `ToolErrorFormatter` | 用于自定义返回给模型的工具审批拒绝消息的全局钩子。 |
| `reasoningItemIdPolicy` | `ReasoningItemIdPolicy` | 在将已生成项目回放到后续模型调用中时，保留或省略 reasoning-item `id` 的默认策略。 |
| `sandbox` | `SandboxRunConfig` | `SandboxAgent` 运行的默认 sandbox 运行时配置。 |

## 状态与对话管理

### 内存策略选择

将状态带入下一轮的常见方式有四种：

| Strategy | Where state lives | Best for | What you pass on the next turn |
| --- | --- | --- | --- |
| `result.history` | 您应用的内存 | 小型聊天循环、完全手动控制、任意 provider | `result.history` |
| `session` | 您的存储 + SDK | 持久化聊天状态、可恢复运行、自定义存储 | 同一个 `session` 实例（或由存储支持的实例） |
| `conversationId` | OpenAI Conversations API | 跨 worker / 服务共享服务器端状态 | 相同的 `conversationId`，再加上仅新的用户轮次输入 |
| `previousResponseId` | 仅 OpenAI Responses API | 不创建对话时，最简单的服务器托管续接方式 | `result.lastResponseId`，再加上仅新的用户轮次输入 |

`result.history` 和 `session` 由客户端管理。`conversationId` 和 `previousResponseId` 由 OpenAI 管理，并且仅适用于使用 OpenAI Responses API 的场景。在大多数应用中，每段对话只选择一种持久化策略即可。除非您有意协调这两层状态，否则将客户端管理的历史与服务器托管状态混用，可能会导致上下文重复。

Sandbox agents 还增加了一层状态：实时 sandbox 工作区。对话历史请使用常规的 SDK `session`、`conversationId` 或 `previousResponseId`；sandbox 文件系统状态请使用 `sandbox.session`、`sandbox.sessionState`、`RunState` 或快照。有关工作区生命周期，请参见[Sandbox agents](/openai-agents-js/zh/guides/sandbox-agents/concepts)。

#### 服务器托管的对话

您可以让 OpenAI Responses API 为您持久化对话历史，而不必在每一轮都发送完整的本地对话历史。这在您需要协调长对话或多个服务时非常有用。对于下面两种服务器托管方式，您在每次请求中都只需传入新一轮的输入，API 会为您复用之前的状态。详情参见[对话状态指南](https://developers.openai.com/api/docs/guides/conversation-state)。

##### 1. 使用 `conversationId` 表示整个对话

您可以先通过 [Conversations API](https://platform.openai.com/docs/api-reference/conversations/create) 创建一次对话，然后在每一轮中复用其 ID。SDK 会自动只包含新生成的项目。

## 钩子与自定义

### 模型输入过滤

使用 `callModelInputFilter` 可以在调用模型**之前**编辑模型输入。此钩子会接收当前智能体、上下文以及合并后的输入项（若存在会话历史，也会包含在内）。您可以返回更新后的 `input` 数组和可选的 `instructions`，用于脱敏、移除旧消息或注入额外的系统指导。

您可以在单次运行中通过 `runner.run(..., { callModelInputFilter })` 设置它，也可以在 `Runner` 配置中作为默认值设置（`RunConfig` 中的 `callModelInputFilter`）。

返回值必须是一个 `ModelInputData` 对象：`{ input: AgentInputItem[], instructions? }`。其中 `input` 字段是必需的，且必须是数组。返回任何其他形状都会抛出 `UserError`。

SDK 会在调用过滤器前克隆已准备好的当前轮次输入。如果您还在使用 `session`，那么被过滤后的克隆内容也会被持久化，因此这里应用的脱敏或截断也会反映到已存储的会话历史中。

对于 `conversationId` 或 `previousResponseId`，该钩子会作用于下一次 Responses API 调用的已准备 payload。之前由服务器托管的上下文会由 API 自动恢复，因此该次调用中被过滤的数组，可能本身就只表示新一轮的增量，而不是对更早历史的完整重放。如果您需要在最后这一步过滤之前，先改变已存储历史与当前轮次的合并方式，请使用 `sessionInputCallback`。

### 工具错误格式化

使用 `toolErrorFormatter` 可以自定义当工具调用被拒绝时，发送回模型的审批拒绝消息。这样您就可以返回特定领域的措辞（例如合规提示），而不是使用 SDK 的默认消息。

该格式化器可以按运行设置（`runner.run(..., { toolErrorFormatter })`），也可以在 `RunConfig` 中全局设置（在 `new Runner(...)` 中设置 `toolErrorFormatter`）。

此格式化器是审批拒绝场景下的全局兜底逻辑。如果您通过 `result.state.reject(interruption, { message: '...' })` 拒绝某个特定中断，那么该次调用中的 `message` 会优先于 `toolErrorFormatter`。如果两者都未提供，SDK 会回退到默认拒绝文本：`Tool execution was not approved.`

该格式化器当前会在 `approval_rejected` 事件中运行，并接收以下内容：

- `kind`（当前始终为 `'approval_rejected'`）
- `toolType`（`'function'`、`'computer'`、`'shell'` 或 `'apply_patch'`）
- `toolName`
- `callId`
- `defaultMessage`（SDK 的兜底消息，当前为 `Tool execution was not approved.`）
- `runContext`

返回一个字符串即可覆盖该消息；如果返回 `undefined`，则保留 SDK 默认值。如果格式化器抛出异常（或返回非字符串值），SDK 会记录一条警告，并回退到默认的审批拒绝消息。

### 推理项 ID 策略

使用 `reasoningItemIdPolicy` 可以控制：当 SDK 将先前生成的运行项目重新转换为 `AgentInputItem[]` 以供后续模型输入时，推理项是否保留其 `id` 字段。

这会影响 SDK 将生成的模型项目回放为输入的场景，例如：

- 同一次运行中的后续模型调用（例如工具执行之后）；
- 将生成项目作为输入 / 历史复用的后续轮次；
- 从已保存的 `RunState` 恢复的运行；
- 派生出的结果视图，如 `result.history` / `result.output`（它们都是模型输入形状的数组）。
- `'preserve'`（默认）会保留推理项 ID。
- `'omit'` 会在将推理项重新作为输入发送前移除其 `id` 字段。
- 非推理项不受影响。

此策略**不会**改变的内容包括：

- 原始模型响应（`result.rawResponses`）；
- 运行项目（`result.newItems`）；
- provider 返回的当前轮次模型输出。

换句话说，该策略只在 SDK 根据先前生成项目构建**下一次输入**时生效。

您可以按运行设置该策略（`runner.run(..., { reasoningItemIdPolicy: 'omit' })`），也可以设置为 runner 默认值（`new Runner({ reasoningItemIdPolicy: 'omit', ... })`）。从已保存的 `RunState` 恢复时，除非您显式覆盖，否则会复用先前已解析出的策略。

#### 与 `callModelInputFilter` 的交互

`reasoningItemIdPolicy` 会先于 `callModelInputFilter` 应用。如果您需要自定义行为，`callModelInputFilter` 仍然可以检查已准备好的输入，并在模型调用前手动重新添加或移除推理 ID。

#### 何时使用 `'omit'`

当您希望被回放的推理项在不带 ID 的情况下进行标准化时，请使用 `'omit'`（例如，为了让转发 / 回放的模型输入更简单，或满足您应用流水线中的集成要求）。

如果您的后端 / provider 会因被回放的推理项而拒绝请求验证（例如，与后续输入中推理项 ID 相关的 HTTP `400` 错误），这也是一个很有用的故障排除选项。在这些情况下，使用 `'omit'` 去除被回放的推理 ID，可以避免发送那些被后端视为对新请求无效的 ID。

如果您希望 SDK 在回放输入中保留推理项 ID，并且您的集成能够接受它们，请保持使用 `'preserve'`。

## 错误与恢复

### 错误处理器

使用 `errorHandlers` 可以将受支持的运行时错误转换为最终输出，而不是直接抛出。当前支持的键有 `maxTurns` 和 `modelRefusal`。

- `errorHandlers.maxTurns` 仅处理最大轮次错误。
- `errorHandlers.modelRefusal` 处理以 [`ModelRefusalError`](/openai-agents-js/openai/agents-core/classes/modelrefusalerror) 形式暴露的模型拒绝。
- `errorHandlers.default` 用作受支持类型的兜底处理器。
- 处理器会接收 `{ error, context, runData }`，并可以返回 `{ finalOutput, includeInHistory? }`。

### 异常

SDK 会抛出一小组可捕获的错误：

- [`MaxTurnsExceededError`](/openai-agents-js/openai/agents-core/classes/maxturnsexceedederror)——达到 `maxTurns`。
- [`ModelBehaviorError`](/openai-agents-js/openai/agents-core/classes/modelbehaviorerror)——模型生成了无效输出（例如格式错误的 JSON、未知工具）。
- [`ModelRefusalError`](/openai-agents-js/openai/agents-core/classes/modelrefusalerror)——模型拒绝生成所请求的输出。
- [`InputGuardrailTripwireTriggered`](/openai-agents-js/openai/agents-core/classes/inputguardrailtripwiretriggered) / [`OutputGuardrailTripwireTriggered`](/openai-agents-js/openai/agents-core/classes/outputguardrailtripwiretriggered)——护栏违规。
- [`ToolInputGuardrailTripwireTriggered`](/openai-agents-js/openai/agents-core/classes/toolinputguardrailtripwiretriggered) / [`ToolOutputGuardrailTripwireTriggered`](/openai-agents-js/openai/agents-core/classes/tooloutputguardrailtripwiretriggered)——工具护栏违规。
- [`GuardrailExecutionError`](/openai-agents-js/openai/agents-core/classes/guardrailexecutionerror)——护栏未能完成执行。
- [`ToolTimeoutError`](/openai-agents-js/openai/agents-core/classes/tooltimeouterror)——函数工具超过 `timeoutMs`，且使用了 `timeoutBehavior: 'raise_exception'`。
- [`ToolCallError`](/openai-agents-js/openai/agents-core/classes/toolcallerror)——函数工具执行时出现非超时错误。
- [`UserError`](/openai-agents-js/openai/agents-core/classes/usererror)——由配置或用户输入导致抛出的任意错误。

它们都继承自基础 `AgentsError` 类，该类可能会提供 `state` 属性以访问当前运行状态。
