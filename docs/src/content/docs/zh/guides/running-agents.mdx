## Runner 生命周期与配置

### 智能体循环

当你使用 Runner 的 run 方法时，需要传入一个起始智能体和输入。输入可以是字符串（会被视为一条用户消息），也可以是输入项列表（即 OpenAI Responses API 中的 items）。

随后 runner 会运行一个循环：

1. 使用当前输入调用当前智能体的模型。
2. 检查 LLM 响应。
   - **最终输出** → 返回。
   - **交接** → 切换到新智能体，保留累计对话历史，转到 1。
   - **工具调用** → 执行工具，将结果追加到对话，转到 1。
3. 当达到 `maxTurns` 时，抛出 [`MaxTurnsExceededError`](/openai-agents-js/openai/agents-core/classes/maxturnsexceedederror)。

<Aside type="note">
  判断 LLM 输出是否为“最终输出”的规则是：
  它产出了期望类型的文本输出，且没有工具调用。
</Aside>

#### Runner 生命周期

在应用启动时创建一个 `Runner`，并在多个请求间复用它。该实例会保存全局配置（如模型提供方和追踪选项）。只有在你需要完全不同的配置时，才创建另一个 `Runner`。对于简单脚本，你也可以直接调用 `run()`，它内部会使用默认 runner。

### 运行参数

`run()` 方法的输入包括：用于启动运行的初始智能体、运行输入，以及一组选项。

输入可以是字符串（会被视为一条用户消息）、[输入项](/openai-agents-js/openai/agents-core/type-aliases/agentinputitem)列表，或在构建[人机协作](/openai-agents-js/zh/guides/human-in-the-loop)智能体时传入 [`RunState`](/openai-agents-js/openai/agents-core/classes/runstate) 对象。

其他可选项如下：

| 选项 | 默认值 | 描述 |
| --- | --- | --- |
| `stream` | `false` | 若为 `true`，调用将返回 `StreamedRunResult`，并在模型产生事件时持续发出。 |
| `context` | – | 传递给每个工具 / 护栏 / 交接的上下文对象。详见[上下文管理指南](/openai-agents-js/zh/guides/context)。 |
| `maxTurns` | `10` | 安全上限——达到时抛出 [`MaxTurnsExceededError`](/openai-agents-js/openai/agents-core/classes/maxturnsexceedederror)。 |
| `signal` | – | 用于取消的 `AbortSignal`。 |
| `session` | – | 会话持久化实现。详见[会话指南](/openai-agents-js/zh/guides/sessions)。 |
| `sessionInputCallback` | – | 自定义会话历史与新输入的合并逻辑；在模型调用前运行。详见[会话](/openai-agents-js/zh/guides/sessions)。 |
| `callModelInputFilter` | – | 在调用模型前编辑模型输入（items + 可选 instructions）的 hook。详见[模型调用输入过滤器](#call-model-input-filter)。 |
| `toolErrorFormatter` | – | 自定义返回给模型的工具审批拒绝消息的 hook。详见[工具错误格式化器](#tool-error-formatter)。 |
| `reasoningItemIdPolicy` | – | 控制在将先前运行项转回模型输入时，是否保留 reasoning-item 的 `id`。详见[Reasoning item ID 策略](#reasoning-item-id-policy)。 |
| `tracing` | – | 按次运行覆盖的追踪配置（例如导出 API key）。 |
| `errorHandlers` | – | 受支持运行时错误的处理器（当前为 `maxTurns`）。详见[错误处理器](#error-handlers)。 |
| `conversationId` | – | 复用服务端对话（仅 OpenAI Responses API + Conversations API）。 |
| `previousResponseId` | – | 在不创建对话的情况下，基于上一次 Responses API 调用继续（仅 OpenAI Responses API）。 |

### 流式传输

流式传输允许你在 LLM 运行时额外接收流式事件。流启动后，`StreamedRunResult` 将包含运行的完整信息，包括所有新产生的输出。你可以使用 `for await` 循环遍历流式事件。详见[流式传输指南](/openai-agents-js/zh/guides/streaming)。

### 运行配置

如果你要创建自己的 `Runner` 实例，可以传入 `RunConfig` 对象来配置 runner。

| 字段 | 类型 | 用途 |
| --- | --- | --- |
| `model` | `string \| Model` | 为运行中的**所有**智能体强制指定模型。 |
| `modelProvider` | `ModelProvider` | 解析模型名称——默认是 OpenAI provider。 |
| `modelSettings` | `ModelSettings` | 覆盖各智能体设置的全局调优参数。 |
| `handoffInputFilter` | `HandoffInputFilter` | 在执行交接时变更输入项（前提是交接本身未定义）。 |
| `inputGuardrails` | `InputGuardrail[]` | 作用于*初始*用户输入的护栏。 |
| `outputGuardrails` | `OutputGuardrail[]` | 作用于*最终*输出的护栏。 |
| `tracingDisabled` | `boolean` | 完全禁用 OpenAI 追踪。 |
| `traceIncludeSensitiveData` | `boolean` | 从追踪中排除 LLM/工具输入输出，但仍发送 span。 |
| `workflowName` | `string` | 显示在 Traces 面板中——有助于对相关运行分组。 |
| `traceId` / `groupId` | `string` | 手动指定 trace 或 group ID，而不是由 SDK 自动生成。 |
| `traceMetadata` | `Record<string, string>` | 附加到每个 span 的任意元数据。 |
| `tracing` | `TracingConfig` | 按次运行覆盖的追踪配置（例如导出 API key）。 |
| `sessionInputCallback` | `SessionInputCallback` | 此 runner 上所有运行的默认历史合并策略。 |
| `callModelInputFilter` | `CallModelInputFilter` | 在每次模型调用前编辑模型输入的全局 hook。 |
| `toolErrorFormatter` | `ToolErrorFormatter` | 自定义返回给模型的工具审批拒绝消息的全局 hook。 |
| `reasoningItemIdPolicy` | `ReasoningItemIdPolicy` | 在将已生成项回放到后续模型调用时，默认保留或省略 reasoning-item `id` 的策略。 |

## 状态与对话管理

### 选择一种记忆策略

将状态带入下一轮常见有四种方式：

| 策略 | 状态存储位置 | 最适用场景 | 下一轮传入内容 |
| --- | --- | --- | --- |
| `result.history` | 你的应用内存 | 小型聊天循环、完全手动控制、任意 provider | `result.history` |
| `session` | 你的存储 + SDK | 持久聊天状态、可恢复运行、自定义存储 | 相同的 `session` 实例（或基于存储的实例） |
| `conversationId` | OpenAI Conversations API | 在多个 worker/服务间共享服务端状态 | 相同的 `conversationId` + 仅新的用户轮次 |
| `previousResponseId` | 仅 OpenAI Responses API | 不创建对话时最简单的服务端托管续接 | `result.lastResponseId` + 仅新的用户轮次 |

`result.history` 和 `session` 由客户端管理。`conversationId` 和 `previousResponseId` 由 OpenAI 管理，仅在使用 OpenAI Responses API 时适用。在大多数应用中，每段对话选择一种持久化策略即可。除非你有意协调两层状态，否则混用客户端历史与服务端状态可能导致上下文重复。

#### 服务端托管对话

你可以让 OpenAI Responses API 为你持久化对话历史，而不是在每轮都发送完整的本地历史。当你在协调长对话或多个服务时，这很有用。使用下面任一服务端托管方式时，每次请求只需传入新一轮输入，API 会为你复用已有状态。详情参见[对话状态指南](https://developers.openai.com/api/docs/guides/conversation-state)。

OpenAI 提供两种复用服务端状态的方式：

##### 1. 使用 `conversationId` 管理整段对话

你可以先通过[Conversations API](https://platform.openai.com/docs/api-reference/conversations/create)创建一次对话，然后在每一轮复用其 ID。SDK 会自动仅包含新生成项。

## Hook 与自定义

### 模型调用输入过滤器

使用 `callModelInputFilter` 在调用模型**之前**编辑模型输入。该 hook 会接收当前智能体、上下文及合并后的输入项（若有 session，也包含会话历史）。你可以返回更新后的 `input` 数组和可选 `instructions`，用于脱敏、丢弃旧消息或注入额外系统引导。

可在单次运行中设置：`runner.run(..., { callModelInputFilter })`，也可在 `Runner` 配置中设为默认值（`RunConfig` 中的 `callModelInputFilter`）。

返回值必须是 `ModelInputData` 对象：`{ input: AgentInputItem[], instructions? }`。`input` 字段必填且必须为数组。返回其他结构会抛出 `UserError`。

SDK 在调用过滤器前会克隆已准备好的当前轮输入。如果你同时使用 `session`，被过滤后的克隆内容会被持久化，因此这里做的脱敏或截断也会反映到存储的会话历史中。

使用 `conversationId` 或 `previousResponseId` 时，该 hook 运行于下一次 Responses API 调用的已准备 payload 上。先前的服务端托管上下文由 API 恢复，因此该次调用的过滤数组可能仅表示新轮次增量，而非完整历史回放。若你需要在最终过滤步骤前调整“已存历史 + 当前轮次”的合并方式，请使用 `sessionInputCallback`。

### 工具错误格式化器

使用 `toolErrorFormatter` 自定义当工具调用被拒绝时回传给模型的审批拒绝消息。这样你可以返回领域特定措辞（例如合规提示），而不是 SDK 默认消息。

格式化器可按次运行设置（`runner.run(..., { toolErrorFormatter })`），也可在 `RunConfig` 中全局设置（`new Runner(...)` 的 `toolErrorFormatter`）。

该格式化器当前在 `approval_rejected` 事件触发，接收以下字段：

- `kind`（当前始终为 `'approval_rejected'`）
- `toolType`（`'function'`、`'computer'`、`'shell'` 或 `'apply_patch'`）
- `toolName`
- `callId`
- `defaultMessage`（SDK 回退消息）
- `runContext`

返回字符串可覆盖消息；返回 `undefined` 则保留 SDK 默认值。如果格式化器抛错（或返回非字符串值），SDK 会记录警告并回退到默认审批拒绝消息。

### Reasoning item ID 策略

使用 `reasoningItemIdPolicy` 控制：当 SDK 将此前生成的运行项转换回 `AgentInputItem[]` 用于后续模型输入时，reasoning items 是否保留其 `id` 字段。

这会影响 SDK 将已生成模型项回放为输入的场景，例如：

- 同一次运行中的后续模型调用（例如工具执行后）；
- 复用生成项作为输入/历史的后续轮次；
- 从已保存 `RunState` 恢复运行；
- 衍生结果视图如 `result.history` / `result.output`（它们是模型输入形态的数组）。
- `'preserve'`（默认）会保留 reasoning-item ID。
- `'omit'` 会在将 reasoning items 回传为输入前移除其 `id` 字段。
- 非 reasoning items 不受影响。

该策略**不会**改变：

- 原始模型响应（`result.rawResponses`）；
- 运行项（`result.newItems`）；
- provider 返回的模型当前轮输出。

换言之，该策略仅在 SDK 基于先前生成项构建**下一次输入**时生效。

你可按次运行设置（`runner.run(..., { reasoningItemIdPolicy: 'omit' })`），也可设为 runner 默认（`new Runner({ reasoningItemIdPolicy: 'omit', ... })`）。从已保存 `RunState` 恢复时，若不覆盖，会复用此前已解析策略。

#### 与 `callModelInputFilter` 的交互

`reasoningItemIdPolicy` 会先于 `callModelInputFilter` 应用。若你需要自定义行为，`callModelInputFilter` 仍可在模型调用前检查已准备输入，并手动重新引入或移除 reasoning ID。

#### 何时使用 `'omit'`

当你希望回放的 reasoning items 以无 ID 形式规范化时，请使用 `'omit'`（例如让转发/回放的模型输入更简洁，或匹配你的应用集成要求）。

如果你的后端/provider 因请求校验错误而拒绝回放的 reasoning items（例如后续输入中与 reasoning item ID 相关的 HTTP `400` 错误），它也是一个有用的排障选项。此时，使用 `'omit'` 去除回放的 reasoning ID 可避免发送被后端视为无效的新请求 ID。

若你希望 SDK 在回放输入中保留 reasoning-item ID，且你的集成可接受它们，请保持 `'preserve'`。

## 错误与恢复

### 错误处理器

使用 `errorHandlers` 可将受支持的运行时错误转换为最终输出，而不是直接抛出。目前仅支持 `maxTurns`。

- `errorHandlers.maxTurns` 仅处理最大轮次错误。
- `errorHandlers.default` 作为受支持类型的兜底处理器。
- 处理器接收 `{ error, context, runData }`，并可返回 `{ finalOutput, includeInHistory? }`。

### 异常

SDK 会抛出一小组可捕获错误：

- [`MaxTurnsExceededError`](/openai-agents-js/openai/agents-core/classes/maxturnsexceedederror) – 达到 `maxTurns`。
- [`ModelBehaviorError`](/openai-agents-js/openai/agents-core/classes/modelbehaviorerror) – 模型产出了无效输出（如 JSON 格式错误、未知工具）。
- [`InputGuardrailTripwireTriggered`](/openai-agents-js/openai/agents-core/classes/inputguardrailtripwiretriggered) / [`OutputGuardrailTripwireTriggered`](/openai-agents-js/openai/agents-core/classes/outputguardrailtripwiretriggered) – 护栏违规。
- [`ToolInputGuardrailTripwireTriggered`](/openai-agents-js/openai/agents-core/classes/toolinputguardrailtripwiretriggered) / [`ToolOutputGuardrailTripwireTriggered`](/openai-agents-js/openai/agents-core/classes/tooloutputguardrailtripwiretriggered) – 工具护栏违规。
- [`GuardrailExecutionError`](/openai-agents-js/openai/agents-core/classes/guardrailexecutionerror) – 护栏执行失败。
- [`ToolTimeoutError`](/openai-agents-js/openai/agents-core/classes/tooltimeouterror) – 函数工具超出 `timeoutMs` 且使用了 `timeoutBehavior: 'raise_exception'`。
- [`ToolCallError`](/openai-agents-js/openai/agents-core/classes/toolcallerror) – 函数工具执行发生非超时错误。
- [`UserError`](/openai-agents-js/openai/agents-core/classes/usererror) – 基于配置或用户输入抛出的任意错误。

以上错误都继承自基础 `AgentsError` 类，该类可通过 `state` 属性访问当前运行状态。

输入重试 vs. 输出重试：

- 输入护栏仅在一次运行中的第一条用户输入上执行，因此重试时必须用相同输入/上下文启动全新运行——传入保存的 `state` 不会再次触发输入护栏。

```
Output guardrail tripped after retry with saved state
```
