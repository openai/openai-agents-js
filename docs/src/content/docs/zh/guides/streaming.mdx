Agents SDK 可以按增量方式交付来自模型及其他执行步骤的输出。流式传输可让 UI 保持响应，并避免在更新用户前必须等待完整的最终结果。

## 启用流式传输

启用流式传输后，返回的 `stream` 会实现 `AsyncIterable` 接口。每个产出的事件都是一个对象，用于描述本次运行中发生的情况。该流会产出三类事件，分别对应智能体执行的不同部分。不过大多数应用只关心模型文本，因此该流也提供了辅助方法。

### 文本输出

调用 `stream.toTextStream()` 可获得已输出文本的流。当 `compatibleWithNodeStreams` 为 `true` 时，返回值是标准 Node.js `Readable`。我们可以将其直接管道到 `process.stdout` 或其他目标。

当运行及所有待处理回调都完成后，`stream.completed` 这个 promise 会 resolve。如果你想确保后续不再有输出，请始终等待它。这也包括在最后一个文本 token 到达后才完成的后处理工作，例如会话持久化或历史压缩钩子。

`toTextStream()` 仅输出 assistant 文本。工具调用、交接、审批及其他运行时事件可从完整事件流中获取。

### Responses WebSocket 传输（可选）

本页的流式传输 API 同样适用于 OpenAI Responses WebSocket 传输。

可通过 `setOpenAIResponsesTransport('websocket')` 全局启用，或使用带有 `useResponsesWebSocket: true` 的自定义 `OpenAIProvider`。

你无需仅为了通过 WebSocket 流式传输就使用 `withResponsesWebSocketSession(...)` 或自定义 `OpenAIProvider`。如果可接受在多次运行之间重连，那么启用该传输后，`run()` / `Runner.run(..., { stream: true })` 仍可正常工作。

当你希望复用连接并更显式地控制 provider 生命周期时，请使用 `withResponsesWebSocketSession(...)` 或自定义 `OpenAIProvider` / `Runner`。

使用 `previousResponseId` 进行续接时，其语义与 HTTP 传输相同。区别仅在于传输方式和连接生命周期。

如果你自行构建 provider，请记得在关闭时调用 `await provider.close()`。默认情况下，基于 Websocket 的模型包装器会被缓存以复用，关闭 provider 会释放这些连接。`withResponsesWebSocketSession(...)` 也能提供相同复用能力，但会自动将清理范围限定在单个回调内。

完整示例可见 [`examples/basic/stream-ws.ts`](https://github.com/openai/openai-agents-js/tree/main/examples/basic/stream-ws.ts)，其中包含流式传输、工具调用、审批和 `previousResponseId`。

`name` 用于标识产出了哪一类条目：

| `name` | 含义 |
| --- | --- |
| `message_output_created` | 已创建消息输出条目。 |
| `handoff_requested` | 模型请求了一次交接。 |
| `handoff_occurred` | 运行时已完成向另一个智能体的交接。 |
| `tool_search_called` | 已产出一个 `tool_search_call` 条目。 |
| `tool_search_output_created` | 已产出一个包含已加载工具定义的 `tool_search_output` 条目。 |
| `tool_called` | 已产出一个工具调用条目。 |
| `tool_output` | 已产出一个工具结果条目。 |
| `reasoning_item_created` | 已产出一个推理条目。 |
| `tool_approval_requested` | 一个工具调用因人工审批而暂停。 |

`tool_search_*` 事件仅会出现在使用 `toolSearchTool()` 并在运行期间加载延迟工具的 Responses 运行中。

交接负载示例：

```json
{
  "type": "run_item_stream_event",
  "name": "handoff_occurred",
  "item": {
    "type": "handoff_call",
    "id": "h1",
    "status": "completed",
    "name": "transfer_to_refund_agent"
  }
}
```

## 流式传输中的人工干预

流式传输与会暂停执行的交接兼容（例如工具需要审批时）。流对象上的 `interruptions` 字段会暴露待处理审批，你可以对每项调用 `state.approve()` 或 `state.reject()` 以继续执行。流暂停后，`stream.completed` 会 resolve，且 `stream.interruptions` 包含需要处理的审批项。随后再次以 `{ stream: true }` 执行即可恢复流式输出。

## 提示

- 退出前请记得等待 `stream.completed`，以确保所有输出都已刷新。
- 初始的 `{ stream: true }` 选项只对其所在的调用生效。如果你使用 `RunState` 重新运行，必须再次指定该选项。
- 如果你的应用只关心文本结果，优先使用 `toTextStream()`，以避免处理单个事件对象。

借助流式传输和事件系统，你可以将智能体集成到聊天界面、终端应用，或任何用户能从增量更新中受益的场景中。
