智能体是 OpenAI Agents SDK 的核心构建模块。**Agent** 是一个经过如下配置的大语言模型（LLM）：

- **Instructions**——系统提示，用于告诉模型*它是谁*以及*应如何响应*。
- **Model**——要调用的 OpenAI 模型，以及可选的模型调优参数。
- **Tools**——LLM 可调用的一组函数或 API，用于完成任务。

## Agent 基础

### 基础配置

`Agent` 构造函数接收一个配置对象。最常用的属性如下所示。

| 属性 | 必填 | 说明 |
| --- | --- | --- |
| `name` | 是 | 简短的人类可读标识符。 |
| `instructions` | 是 | 系统提示（字符串**或**函数——见[动态 instructions](#dynamic-instructions)）。 |
| `prompt` | 否 | OpenAI Responses API 的 prompt 配置。可接收静态 prompt 对象或函数。见[Prompt](/openai-agents-js/zh/guides/models#prompt)。 |
| `handoffDescription` | 否 | 当该智能体作为交接工具提供时使用的简短描述。 |
| `handoffs` | 否 | 将对话委派给专业智能体。见[组合模式](#composition-patterns)和[交接指南](/openai-agents-js/zh/guides/handoffs)。 |
| `model` | 否 | 模型名称**或**自定义 [`Model`](/openai-agents-js/openai/agents/interfaces/model/) 实现。 |
| `modelSettings` | 否 | 调优参数（temperature、top_p 等）。见[模型](/openai-agents-js/zh/guides/models#modelsettings)。如果你需要的属性不在顶层，可放在 `providerData` 下。 |
| `tools` | 否 | 模型可调用的 [`Tool`](/openai-agents-js/openai/agents/type-aliases/tool/) 实例数组。见[工具](/openai-agents-js/zh/guides/tools)。 |
| `mcpServers` | 否 | 智能体的 MCP 支持工具。见[MCP 集成指南](/openai-agents-js/zh/guides/mcp)。 |
| `inputGuardrails` | 否 | 应用于此智能体链首个用户输入的护栏。见[护栏](/openai-agents-js/zh/guides/guardrails)。 |
| `outputGuardrails` | 否 | 应用于该智能体最终输出的护栏。见[护栏](/openai-agents-js/zh/guides/guardrails)。 |
| `outputType` | 否 | 返回结构化输出而非纯文本。见[输出类型](#output-types)和[执行结果](/openai-agents-js/zh/guides/results#final-output)。 |
| `toolUseBehavior` | 否 | 控制函数工具结果是回传给模型还是结束运行。见[强制工具调用](#forcing-tool-use)。 |
| `resetToolChoice` | 否 | 工具调用后将 `toolChoice` 重置为默认值（默认：`true`），以防止工具调用循环。见[强制工具调用](#forcing-tool-use)。 |
| `handoffOutputTypeWarningEnabled` | 否 | 当交接输出类型不一致时发出警告（默认：`true`）。见[执行结果](/openai-agents-js/zh/guides/results#final-output)。 |

### 上下文

智能体在其上下文类型上是**泛型**的——即 `Agent<TContext, TOutput>`。*上下文*是你创建并传入 `Runner.run()` 的依赖注入对象。它会传递给每个工具、护栏、交接等，适合用于存储状态或提供共享服务（数据库连接、用户元数据、功能开关等）。

### 输出类型

默认情况下，Agent 返回**纯文本**（`string`）。如果你希望模型返回结构化对象，可以指定 `outputType` 属性。SDK 支持：

1. [Zod](https://github.com/colinhacks/zod) schema（`z.object({...})`）。
2. 任意兼容 JSON schema 的对象。

### OpenAI 平台映射

一些智能体概念会直接映射到 OpenAI 平台概念，另一些则是在运行智能体时配置，而不是定义时配置。

| SDK 概念 | OpenAI 指南 | 适用场景 |
| --- | --- | --- |
| `outputType` | [Structured Outputs](https://developers.openai.com/api/docs/guides/structured-outputs) | 智能体应返回带类型的 JSON 或经 Zod 校验的对象，而不是文本。 |
| `tools` / 托管工具 | [Tools guide](https://developers.openai.com/api/docs/guides/tools) | 模型应执行搜索、检索、代码执行，或调用你的函数/工具。 |
| `conversationId` / `previousResponseId` | [Conversation state](https://developers.openai.com/api/docs/guides/conversation-state) | 你希望 OpenAI 在轮次之间持久化或串联对话状态。 |

`conversationId` 和 `previousResponseId` 是运行时控制项，不是 `Agent` 构造字段。需要这些 SDK 入口时，请使用[运行智能体](/openai-agents-js/zh/guides/running-agents)。

## 组合模式

当智能体参与更大工作流时，最常见的两个 SDK 入口是：

1. **Manager（Agents as tools）**——中心智能体掌控对话，并调用作为工具暴露的专业智能体。
2. **交接**——初始智能体在识别用户请求后，将整个对话委派给专业智能体。

这两种方法是互补的。Manager 让你在一个位置统一执行护栏或速率限制，而交接让每个智能体专注单一任务且不保留对话控制权。关于设计权衡和何时选择各模式，见[智能体编排](/openai-agents-js/zh/guides/multi-agent)。

#### Manager（Agents as tools）

在这种模式中，manager 不会移交控制权——LLM 使用工具，manager 汇总最终答案。详见[工具指南](/openai-agents-js/zh/guides/tools#agents-as-tools)。

<Code lang="typescript" code={agentsAsTools} title="Agents as tools" />

#### 交接

在交接模式中，分诊智能体负责路由请求，但一旦发生交接，专业智能体会接管对话，直到产出最终输出。这可让提示更短，并便于你独立分析每个智能体。详见[交接指南](/openai-agents-js/zh/guides/handoffs)。

<Code lang="typescript" code={agentWithHandoffs} title="带交接的 Agent" />

如果你的交接目标可能返回不同输出类型，优先使用 `Agent.create(...)` 而不是 `new Agent(...)`。这样 TypeScript 可以推断交接图中可能的 `finalOutput` 形状联合类型，并避免由 `handoffOutputTypeWarningEnabled` 控制的运行时警告。完整示例见[执行结果指南](/openai-agents-js/zh/guides/results#final-output)。

### 生命周期钩子

在高级用例中，你可以通过监听事件来观察 Agent 生命周期。

`Agent` 实例会为该特定智能体实例发出生命周期事件，而 `Runner` 会以覆盖整次运行的单一事件流发出相同事件名。这在多智能体工作流中很有用，你可以在一个位置观察交接和工具调用。

共享事件名如下：

| 事件 | Agent hook 参数 | Runner hook 参数 |
| --- | --- | --- |
| `agent_start` | `(context, agent, turnInput?)` | `(context, agent, turnInput?)` |
| `agent_end` | `(context, output)` | `(context, agent, output)` |
| `agent_handoff` | `(context, nextAgent)` | `(context, fromAgent, toAgent)` |
| `agent_tool_start` | `(context, tool, { toolCall })` | `(context, agent, tool, { toolCall })` |
| `agent_tool_end` | `(context, tool, result, { toolCall })` | `(context, agent, tool, result, { toolCall })` |

### 强制工具调用

提供工具并不保证 LLM 会调用它。你可以通过 `modelSettings.toolChoice` **强制**工具调用：

1. `'auto'`（默认）——由 LLM 决定是否使用工具。
2. `'required'`——LLM *必须*调用工具（可自行选择调用哪个）。
3. `'none'`——LLM **不得**调用工具。
4. 指定工具名，例如 `'calculator'`——LLM 必须调用该特定工具。

<Code lang="typescript" code={agentForcingToolUse} title="强制工具调用" />

#### 防止无限循环

工具调用后，SDK 会自动将 `toolChoice` 重置为 `'auto'`。这可防止模型反复尝试调用工具而进入无限循环。你可以通过 `resetToolChoice` 标志或配置 `toolUseBehavior` 来覆盖此行为：

- `'run_llm_again'`（默认）——携带工具结果再次运行 LLM。
- `'stop_on_first_tool'`——将第一次工具结果视为最终答案。
- `{ stopAtToolNames: ['my_tool'] }`——当调用列表中的任一工具时停止。
- `(context, toolResults) => ...`——返回是否应结束运行的自定义函数。

```typescript
const agent = new Agent({
  ...,
  toolUseBehavior: 'stop_on_first_tool',
});
```

注意：`toolUseBehavior` 仅适用于**函数工具**。托管工具始终会返回给模型进行处理。
