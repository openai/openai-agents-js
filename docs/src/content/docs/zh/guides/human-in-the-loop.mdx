本指南介绍 SDK 基于审批的人机协作流程。当某个工具调用需要审批时，SDK 会暂停运行，返回 `interruptions`，并允许您稍后从同一个 `RunState` 恢复。

该审批界面作用于整个运行范围，而不局限于当前顶层智能体。无论工具属于当前智能体、属于通过交接到达的智能体，还是属于嵌套的 `agent.asTool()` 执行，都适用同样的模式。在嵌套 `agent.asTool()` 的情况下，中断仍会出现在外层运行中，因此您需要在外层的 `result.state` 上批准或拒绝它，并恢复原始的根运行。

对于 `agent.asTool()`，审批可能发生在两个不同层级：智能体工具本身可通过 `asTool({ needsApproval })` 要求审批，而嵌套智能体内部的工具也可能在嵌套运行开始后进一步发起自己的审批。这两种情况都通过同一个外层运行中断流程处理。

本页重点介绍通过 `interruptions` 进行手动审批的流程。如果您的应用可以在代码中自行决定，某些工具类型也支持编程式审批回调，从而让运行无需暂停即可继续。如果您正在设置 `agent.asTool()` 本身，请参阅[工具指南](/openai-agents-js/zh/guides/tools#4-agents-as-tools)；本页讲解的是该运行层级中任意工具一旦需要审批后会发生什么。

### 流程

1. 当某个工具调用即将执行时，SDK 会评估其审批规则（`needsApproval` 或托管 MCP 的等效配置）。
2. 如果需要审批且尚未存储决策，该工具调用不会执行。相反，运行会记录一个 [`RunToolApprovalItem`](/openai-agents-js/openai/agents/classes/runtoolapprovalitem)。
3. 在该轮结束时，运行会暂停，并在[执行结果](/openai-agents-js/zh/guides/results)的 `interruptions` 数组中返回所有待处理审批。这也包括在嵌套 `agent.asTool()` 运行中发起的审批。
4. 使用 `result.state.approve(interruption)` 或 `result.state.reject(interruption)` 处理每个待处理项。如果同一个工具在本次运行的剩余过程中应始终保持已批准或已拒绝状态，可传入 `{ alwaysApprove: true }` 或 `{ alwaysReject: true }`。拒绝时，您还可以传入 `{ message: '...' }` 来控制发回给模型、针对该特定工具调用的拒绝文本。
5. 通过将更新后的 `result.state` 重新传给 `runner.run(agent, state)` 来恢复，其中 `agent` 是该次运行的原始顶层智能体。SDK 会从中断点继续，包括嵌套的智能体工具执行。

使用 `{ alwaysApprove: true }` 或 `{ alwaysReject: true }` 创建的粘性决策会存储在运行状态中，因此当您稍后恢复同一个已暂停运行时，它们会在 `toString()` / `fromString()` 之间保留下来。

在 GA 模型上，计算机工具中断可以表示单个 `computer_call` 中的一批操作。SDK 会在执行前按操作评估 `needsApproval`，因此一个待审批项可以覆盖如移动 + 点击这样的操作序列。如果您通过 `interruption.rawItem` 检查并渲染 UI，请同时处理 GA 的 `actions` 数组和旧版的单个 `action` 字段。

序列化后的 `RunState` 也会同时保留当前 `computer` 工具名称和旧版 `computer_use_preview` 名称下的计算机审批，因此在 preview 到 GA 的迁移过程中，暂停的运行也能顺利恢复。

如果您未提供 `message`，SDK 会依次回退到已配置的 `toolErrorFormatter`（如果有），然后再使用默认拒绝文本。

您不需要在同一次处理中解决所有待审批项。如果您只批准或拒绝了其中一部分后就重新运行，那么这些已处理的调用可以继续，而未处理的项会继续保留在 `interruptions` 中，并再次暂停运行。

### 自动审批决策

手动 `interruptions` 是最通用的模式，但不是唯一方式：

- 本地 `shellTool()` 和 `applyPatchTool()` 可以使用 `onApproval` 在代码中立即批准或拒绝。
- 托管 MCP 工具可以结合 `requireApproval` 和 `onApproval` 来进行同类编程式决策。
- 普通函数工具使用本页介绍的手动中断流程。

当这些回调返回决策时，运行会继续，而不会为了等待人工响应而暂停。对于 Realtime / 语音会话 API，请参阅[构建语音智能体](/openai-agents-js/zh/guides/voice-agents/build#approval-requests)中的审批流程。

### 流式传输和会话

同样的中断流程也适用于流式传输运行。流式运行暂停后，等待 `stream.completed`，读取 `stream.interruptions`，处理它们，然后在需要让恢复后的输出继续流式传输时，使用 `{ stream: true }` 再次调用 `run()`。关于此模式的流式版本，请参阅[流式传输中的人机协作](/openai-agents-js/zh/guides/streaming#human-in-the-loop-while-streaming)。

如果您还在使用 `session`，那么从 `RunState` 恢复时请继续传入同一个 `session`。这样恢复后的轮次会追加到会话记忆中，而无需重新准备输入。有关会话生命周期的详细信息，请参阅[会话指南](/openai-agents-js/zh/guides/sessions)。

## 处理较长审批时间

人机协作流程被设计为可中断较长一段时间，而无需让您的服务器持续运行。如果您需要关闭请求并在之后继续，可以序列化状态并稍后恢复。

您可以使用 `result.state.toString()`（或 `JSON.stringify(result.state)`）来序列化状态，之后通过将序列化后的状态传入 `RunState.fromString(agent, serializedState)` 来恢复，其中 `agent` 是触发整个运行的智能体实例。

当 `RunState` 被序列化时，SDK 会为交接和 `Agent.asTool()` 图记录稳定的智能体身份。这使得暂停的运行即使在不同智能体共享相同 `name` 的情况下也能恢复，只要恢复运行的进程重建了同样的智能体图即可。

如果恢复运行的进程需要注入一个新的上下文对象，请改用 `RunState.fromStringWithContext(agent, serializedState, context, { contextStrategy })`。

- `contextStrategy: 'merge'`（默认）会保留提供的 `RunContext`，合并序列化的审批状态，并在新上下文尚未定义 `toolInput` 时恢复序列化的 `toolInput`。
- `contextStrategy: 'replace'` 会按原样使用提供的 `RunContext` 重新构建运行。

序列化的运行状态包含您的应用上下文，以及由 SDK 管理的运行时元数据，例如审批、用量、嵌套的 `toolInput` 和待恢复的嵌套智能体工具执行。如果您计划存储或传输序列化状态，请将 `runContext.context` 视为持久化数据处理，除非您有意让这些机密随状态一起传递，否则不要将机密放入其中。

默认情况下，追踪 API 密钥不会包含在序列化状态中，以避免您意外持久化机密。仅当您确实需要让追踪凭证随状态一起移动时，才传入 `result.state.toString({ includeTracingApiKey: true })`。

这样，您就可以将序列化状态存入数据库，或与您的请求一起存储。

### 待处理任务的版本管理

<Aside>
  这主要适用于您尝试在较长时间内存储序列化状态，同时还在修改智能体的情况。
</Aside>

如果您的审批请求耗时较长，并且您打算以有意义的方式为智能体定义做版本管理，或升级 Agents SDK 版本，我们目前建议您通过包别名并行安装两个版本的 Agents SDK，以实现您自己的分支逻辑。

在实际操作中，这意味着为您自己的代码分配一个版本号，与序列化状态一起存储，并在反序列化时引导到正确版本的代码。
