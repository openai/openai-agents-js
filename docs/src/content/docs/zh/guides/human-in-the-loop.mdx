本指南介绍 SDK 中基于审批的人工干预流程。当工具调用需要审批时，SDK 会暂停运行、返回 `interruptions`，并允许您稍后从同一 `RunState` 恢复运行。

该审批机制作用于整个运行，而不仅限于当前的顶层智能体。无论工具属于当前智能体、通过交接到达的智能体，还是嵌套的 `agent.asTool()` 执行，都适用相同的模式。对于嵌套的 `agent.asTool()`，中断仍会在外层运行中呈现，因此您需要在外层的 `result.state` 上批准或拒绝该中断，然后恢复原始的根运行。

使用 `agent.asTool()` 时，审批可能发生在两个不同层级：智能体工具本身可以通过 `asTool({ needsApproval })` 要求审批；嵌套智能体中的工具也可以在嵌套运行开始后提出自己的审批请求。两者都通过相同的外层运行中断流程进行处理。

本页重点介绍通过 `interruptions` 进行的手动审批流程。如果您的应用可以通过代码做出决定，某些工具类型也支持程序化审批回调，使运行无需暂停即可继续。如果您正在配置 `agent.asTool()` 本身，请参阅[工具](/openai-agents-js/zh/guides/tools#4-agents-as-tools)；本页介绍该运行层级结构中的任何工具需要审批后会发生什么。

### 流程

1. 工具调用即将执行时，SDK 会评估其审批规则（`needsApproval` 或托管 MCP 中的对应设置）。
2. 如果需要审批且尚未存储任何决定，工具调用将不会执行。运行会改为记录一个 [`RunToolApprovalItem`](/openai-agents-js/openai/agents/classes/runtoolapprovalitem)。
3. 在该轮次结束时，运行会暂停，并在[执行结果](/openai-agents-js/zh/guides/results)的 `interruptions` 数组中返回所有待审批项。其中包括嵌套 `agent.asTool()` 运行中提出的审批请求。
4. 使用 `result.state.approve(interruption)` 或 `result.state.reject(interruption)` 处理每个待审批项。如果要在本次运行的剩余时间内始终批准或拒绝同一工具，请传入 `{ alwaysApprove: true }` 或 `{ alwaysReject: true }`。拒绝时，您还可以传入 `{ message: '...' }`，以控制针对该特定工具调用发送回模型的拒绝文本。
5. 将更新后的 `result.state` 传回 `runner.run(agent, state)` 以恢复运行，其中 `agent` 是本次运行最初的顶层智能体。SDK 会从中断点继续运行，包括嵌套的智能体工具执行。

默认情况下，函数工具输入护栏仅在审批后、工具执行前立即运行。如果希望这些输入护栏在显示待审批项之前验证本地函数工具调用，请向 `run()` 或 `Runner` 传入 `toolExecution: { preApprovalInputGuardrails: true }`。如果审批前护栏拒绝调用，SDK 会将护栏消息作为工具输出返回给模型，而不会创建审批中断。如果护栏允许调用，运行仍会暂停以等待审批；审批后，输入护栏还会再次运行，以防工具调用在等待期间变得不安全。

当 `needsApproval` 是函数时，SDK 仅在工具参数解析为可检查对象后才会调用它。对于格式错误的 JSON 和非对象值，SDK 会采用封闭策略：请求审批，但不会调用回调或执行工具。即使批准该调用，工具仍不会执行，而是继续进入常规的参数解析错误处理流程。Realtime 函数工具遵循相同规则，并会针对无效调用发出 `tool_approval_requested`。

使用 `{ alwaysApprove: true }` 或 `{ alwaysReject: true }` 创建的持久决定会存储在运行状态中，因此稍后恢复同一暂停运行时，这些决定可在 `toString()` / `fromString()` 过程中保留。

在 GA 模型中，计算机工具中断可以表示一个 `computer_call` 中的一批操作。SDK 会在执行前按操作评估 `needsApproval`，因此一个待审批项可以涵盖移动 + 点击等操作序列。如果您通过检查 `interruption.rawItem` 来呈现 UI，请同时处理 GA 版本的 `actions` 数组和旧版的单个 `action` 字段。

序列化的 `RunState` 还会针对当前的 `computer` 工具名称和旧版的 `computer_use_preview` 名称保留计算机操作审批，因此暂停的运行可在从预览版迁移到 GA 期间顺利恢复。

如果未提供 `message`，SDK 会依次回退到配置的 `toolErrorFormatter`（如果有）和默认拒绝文本。

您无需在同一轮处理中解决所有待审批项。如果仅批准或拒绝其中一部分后重新运行，已处理的调用可以继续，而未处理的调用仍会保留在 `interruptions` 中，并再次暂停运行。

### 自动审批决策

手动处理 `interruptions` 是最通用的模式，但并非唯一方式：

- 本地 `shellTool()` 和 `applyPatchTool()` 可以使用 `onApproval`，直接在代码中批准或拒绝。
- 托管 MCP 工具可以结合使用 `requireApproval` 和 `onApproval`，进行同类程序化决策。
- 普通函数工具使用本页介绍的手动中断流程。

当这些回调返回决定时，运行会继续，而无需暂停等待人工响应。对于 Realtime / 语音会话 API，请参阅[构建语音智能体](/openai-agents-js/zh/guides/voice-agents/build#approval-before-tool-execution)中的审批流程。

### 流式传输与会话

相同的中断流程也适用于流式运行。流式运行暂停后，请等待 `stream.completed`，读取 `stream.interruptions` 并进行处理；如果希望恢复后的输出继续以流式方式传输，请使用 `{ stream: true }` 再次调用 `run()`。有关此模式的流式版本，请参阅[流式传输中的人工干预](/openai-agents-js/zh/guides/streaming#human-in-the-loop-while-streaming)。

如果您还在使用 `session`，从 `RunState` 恢复运行时，请继续传入同一个 `session`。随后，恢复的轮次会追加到会话记忆中，而无需重新准备输入。有关会话生命周期的详细信息，请参阅[会话](/openai-agents-js/zh/guides/sessions)。

## 长时间审批的处理

人工干预流程可以长时间保持可中断状态，而无需让服务器持续运行。如果您需要结束请求并在稍后继续，可以序列化状态并在之后恢复。

您可以使用 `result.state.toString()`（或 `JSON.stringify(result.state)`）序列化状态，稍后再将序列化状态传入 `RunState.fromString(agent, serializedState)` 以恢复运行，其中 `agent` 是触发整个运行的智能体实例。

序列化 `RunState` 时，SDK 会记录交接和 `Agent.asTool()` 图中的稳定智能体标识。只要恢复运行的进程重新构建相同的智能体图，即使不同的智能体具有相同的 `name`，暂停的运行也能恢复。

传给 `RunState.fromString(agent, serializedState)` 的 `agent` 是重建图的根节点。在反序列化过程中，SDK 会遍历该智能体的交接和 `Agent.asTool()` 引用，然后依据重建图解析状态中的每个序列化智能体引用。其中包括当前智能体，以及生成项、已处理的模型响应和排队的后续步骤中保存的嵌套引用。

如果需要使用替换后的图恢复运行，例如智能体的模型或工具由其他运行时封装，请先使用原始图反序列化状态，再次将其序列化，然后使用替换后的根智能体反序列化该字符串。调用 `state.setCurrentAgent(agent)` 只会更改当前活动的智能体，不会重写已在反序列化期间解析的嵌套引用。

如果恢复运行的进程需要注入新的上下文对象，请改用 `RunState.fromStringWithContext(agent, serializedState, context, { contextStrategy })`。

- `contextStrategy: 'merge'`（默认）会保留传入的 `RunContext`、合并序列化的审批状态，并在新上下文尚未定义 `toolInput` 时恢复序列化的 `toolInput`。
- `contextStrategy: 'replace'` 会按原样使用传入的 `RunContext` 重建运行。

序列化的运行状态包括应用上下文，以及 SDK 管理的运行时元数据，例如审批、用量、嵌套的 `toolInput` 和待恢复的嵌套智能体工具运行。如果计划存储或传输序列化状态，请将 `runContext.context` 视为持久化数据；除非您明确希望密钥随状态一起传输，否则请勿在其中存放密钥。

默认情况下，追踪 API 密钥会从序列化状态中省略，以免意外持久化密钥。仅当确实需要随状态传输追踪凭据时，才传入 `result.state.toString({ includeTracingApiKey: true })`。

这样，您就可以将序列化状态存储在数据库中，或与请求一起存储。

### 待处理任务的版本管理

<Aside>
  这主要适用于您在修改智能体期间，尝试将序列化状态存储 更长时间的情况。
</Aside>

如果审批请求耗时较长，并且您打算对智能体定义进行实质性的版本管理，或升级 Agents SDK 版本，我们目前建议您使用软件包别名并行安装两个版本的 Agents SDK，自行实现分支逻辑。

实际上，这意味着为自己的代码分配版本号，将其与序列化状态一起存储，并在反序列化时路由到正确版本的代码。
