### 流程

1. 如果智能体决定调用一个（或多个）工具，它会通过计算 `needsApproval` 来检查该工具是否需要批准。
2. 如果需要批准，智能体会检查该批准是否已通过或已拒绝。
   - 如果尚未通过或拒绝，工具会向智能体返回一条静态消息，说明该工具调用无法执行。
   - 如果缺少通过/拒绝结果，则会触发一个工具批准请求。
3. 智能体会收集所有工具批准请求并中断执行。
4. 如果存在任何中断，[result](/openai-agents-js/guides/results) 将包含一个 `interruptions` 数组，用于描述待处理步骤。当工具调用需要确认时，会出现一个 `type: "tool_approval_item"` 的 `ToolApprovalItem`。
5. 你可以调用 `result.state.approve(interruption)` 或 `result.state.reject(interruption)` 来通过或拒绝该工具调用。如果同一个工具在本次运行剩余期间应始终保持通过或拒绝，可传入 `{ alwaysApprove: true }` 或 `{ alwaysReject: true }`。
6. 处理完所有中断后，你可以将 `result.state` 传回 `runner.run(agent, state)` 以恢复执行，其中 `agent` 是触发整次运行的原始智能体。
7. 流程会从步骤 1 再次开始。

## 处理较长的批准等待时间

人工干预流程被设计为可在较长时间内中断，而无需让服务器持续运行。如果你需要先关闭请求、稍后再继续，可以序列化状态并在之后恢复。

你可以用 `result.state.toString()`（或 `JSON.stringify(result.state)`）来序列化状态；之后可将序列化后的状态传入 `RunState.fromString(agent, serializedState)` 进行恢复，其中 `agent` 是触发整次运行的智能体实例。

如果恢复后的流程需要注入新的上下文对象，请改用 `RunState.fromStringWithContext(agent, serializedState, context, { contextStrategy })`。

- `contextStrategy: 'merge'`（默认）会保留序列化的批准状态，并将其合并到提供的 `RunContext` 中。
- `contextStrategy: 'replace'` 会按原样使用提供的 `RunContext` 重建运行。

默认情况下，追踪 API 密钥会从序列化状态中省略，以避免意外持久化敏感信息。只有在你明确需要随状态一起迁移追踪凭据时，才传入 `result.state.toString({ includeTracingApiKey: true })`。

这样，你就可以将序列化状态存入数据库，或与请求一起保存。

### 待处理任务版本管理

<Aside>
  这主要适用于你在修改智能体的同时，想要将序列化状态存储较长时间的场景。
</Aside>

如果你的批准请求耗时较长，并且你打算以有意义的方式管理智能体定义版本或升级 Agents SDK 版本，我们目前建议你使用 package aliases 并行安装两个 Agents SDK 版本，以实现自己的分支逻辑。

在实践中，这意味着为你的代码分配版本号，将其与序列化状态一并存储，并在反序列化时引导到对应版本的代码。
