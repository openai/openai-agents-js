<Aside type="caution">

请尽早确定架构：

- `OpenAIRealtimeWebRTC` 是最简单的浏览器端方案，并会自动处理音频输入和输出。
- `OpenAIRealtimeWebSocket` 提供更强的控制能力，但您必须自行管理音频采集和播放。
- 函数工具会在 `RealtimeSession` 所在的环境中运行。如果会话在浏览器中运行，工具也会在浏览器中运行。
- 实时交接会沿用同一个实时会话模型。只有在会话尚未产生音频输出之前，才能更改语音。如果需要使用不同的后端模型，请通过工具委派任务，而不是使用交接。

</Aside>

## 会话设置

### 音频处理

某些传输层（例如默认的 `OpenAIRealtimeWebRTC`）会自动处理音频输入和输出。对于 `OpenAIRealtimeWebSocket` 等其他传输机制，您必须自行处理会话音频：

当底层传输机制支持时，`session.muted` 会报告当前的静音状态，`session.mute(true | false)` 可切换麦克风采集状态。`OpenAIRealtimeWebSocket` 不支持静音：`session.muted` 返回 `null`，而 `session.mute()` 会抛出异常。因此，在 WebSocket 配置中，您应自行暂停采集，并停止调用 `sendAudio()`，直到需要重新启用麦克风。

### 会话配置

创建 [`RealtimeSession`](/openai-agents-js/openai/agents-realtime/classes/realtimesession/) 时配置会话本身，通常通过 `model` 选项和 `config` 对象进行配置。`connect(...)` 用于处理连接时的事项，例如凭证、端点 URL 和 SIP 通话关联，而不是任意会话字段。

<Code lang="typescript" code={configureSessionExample} />

在底层，SDK 会将此配置规范化为 Realtime [`session.update`](https://developers.openai.com/api/reference/resources/realtime/client-events/#session.update) 格式。如果需要使用 [RealtimeSessionConfig](/openai-agents-js/openai/agents-realtime/type-aliases/realtimesessionconfig/) 中没有对应属性的原始会话字段，请使用 `providerData`，或通过 `session.transport.sendEvent(...)` 发送原始 `session.update`。

建议使用较新的 SDK 配置格式，包括 `outputModalities`、`audio.input` 和 `audio.output`。为保持向后兼容，SDK 仍会规范化 `modalities`、`inputAudioFormat`、`outputAudioFormat`、`inputAudioTranscription` 和 `turnDetection` 等旧别名，但新代码应使用此处所示的嵌套 `audio` 结构。

对于支持推理的 Realtime 模型（例如 `gpt-realtime-2.1`），请在会话配置中设置 `reasoning.effort`。更高的推理强度可能增加延迟和 token 用量。如果要控制模型是否可以并行调用多个工具，还可以设置 `parallelToolCalls`。

对于语音到语音会话，通常选择 `outputModalities: ['audio']`，它会同时提供音频输出和转录文本。只有需要纯文本响应时，才切换为 `['text']`。

对于较新且在 [RealtimeSessionConfig](/openai-agents-js/openai/agents-realtime/type-aliases/realtimesessionconfig/) 中没有对应参数的参数，可以使用 `providerData`。传入 `providerData` 的所有内容都会作为原始 `session` 对象的一部分转发。

还可以在构造时设置以下 `RealtimeSession` 选项：

### 对话生命周期

`RealtimeSession` 构建在长期存在的 Realtime 连接之上。它会保留对话历史记录的本地副本、监听传输事件、运行工具和输出护栏，并使当前智能体配置与传输层保持同步。

底层 API 的行为仍然很重要：

- 成功连接后会先触发 `session.created` 事件，之后的配置更改会触发 `session.updated`。
- 大多数会话属性都可以随时间更改，但 `model` 无法在对话期间更改；`voice` 只能在会话尚未产生音频输出之前更改；追踪应预先确定，因为 Realtime API 不允许在启用追踪后修改其配置。
- Realtime API 目前将单个会话限制为 60 分钟。
- 输入音频转录以异步方式进行，因此最新话语的转录文本可能在响应生成已经开始后才到达。

在 SDK 层，`await session.connect()` 表示"传输层已准备就绪，可以开始对话"，但具体时点因传输机制而异：

- 在默认的浏览器 WebRTC 传输中，SDK 会在数据通道打开后立即发送初始 `session.update`，并尝试等待相应的 `session.updated` 事件，然后再完成 `connect()`。这样可以避免在应用您的 instructions、工具和模态之前，音频就已到达服务器。如果始终未收到该确认，`connect()` 会在短暂超时后回退为直接完成。
- 在默认的服务器端 WebSocket 传输中，套接字打开并发送初始配置后，`connect()` 就会完成。因此，对应的 `session.updated` 事件可能在 `connect()` 已经完成之后才到达。

如果需要了解原始事件模型，请结合本页阅读官方的 [Realtime 对话指南](https://developers.openai.com/api/docs/guides/realtime-conversations/)。

## 交互流程

### 轮次检测与语音活动检测

默认情况下，Realtime 会话使用内置语音活动检测（VAD），以便 API 判断用户何时开始或停止说话，以及何时创建响应。SDK 通过 `audio.input.turnDetection` 公开此功能。

<Code lang="typescript" code={turnDetectionExample} />

两种常用模式如下：

- `semantic_vad` 旨在提供更自然的轮次边界；当用户听起来还没有说完时，它可以多等待一会儿。
- `server_vad` 更依赖阈值，并提供 `threshold`、`prefixPaddingMs`、`silenceDurationMs` 和 `idleTimeoutMs` 等设置。

如果要自行管理轮次边界，请将 `audio.input.turnDetection` 设置为 `null`。官方的[语音活动检测指南](https://developers.openai.com/api/docs/guides/realtime-vad/)和 [Realtime 对话指南](https://developers.openai.com/api/docs/guides/realtime-conversations/#voice-activity-detection)更详细地介绍了底层行为。

### 中断

启用 VAD 后，在智能体说话时插话可以中断当前响应。在 WebSocket 传输中，SDK 会监听 `input_audio_buffer.speech_started`，将助手音频截断到用户实际听到的位置，并发出 `audio_interrupted` 事件。在 WebSocket 配置中自行管理播放时，该事件尤其有用。

### 手动响应控制

在更高层的 SDK 中，`sendMessage()` 和 `addImage()` 默认会自动触发响应。使用原始传输事件、按键通话流程或自定义审核/验证步骤时，手动响应控制非常重要。

<Code lang="typescript" code={transportEventsExample} />

常见情况有两种：

1. 如果通过 `audio.input.turnDetection = null` 完全禁用 VAD，则需要自行提交音频轮次，然后发送 `response.create`。
2. 如果保留 VAD，但设置 `turnDetection.interruptResponse = false` 和 `turnDetection.createResponse = false`，API 仍会检测轮次，但由您决定是否创建响应。

## 智能体能力

### 交接

与常规智能体类似，您可以使用交接将一个智能体拆分为多个智能体，并在它们之间进行编排，从而提高性能并更准确地界定问题范围。

<Code lang="typescript" code={multiAgentsExample} />

与常规智能体不同，实时智能体的交接行为略有差异。执行交接时，当前会话会就地更新为新的智能体配置。因此，新智能体会自动访问当前对话历史记录，并且目前不会应用输入过滤器。

由于会话始终保持实时状态，该会话使用的模型不会在交接期间更改。语音更改遵循底层 Realtime API 规则：只有在会话尚未产生音频输出之前才能更改。实时交接主要用于在同一会话中切换不同的 `RealtimeAgent` 配置；如果需要使用其他模型（例如 `gpt-5.4` 这样的推理模型），或者将任务委派给非实时后端智能体，请使用[通过工具委派](#delegation-through-tools)。

### 工具

与常规智能体一样，实时智能体可以调用工具来执行操作。Realtime 支持**函数工具**（本地执行）和**托管 MCP 工具**（由 Realtime API 远程执行）。您可以使用与常规智能体相同的 `tool()` 辅助函数来定义函数工具。

Responses 专用的工具选项不适用于实时智能体。包含 `'programmatic'` 的函数工具 `outputSchema` 和 `allowedCallers` 值会被拒绝，托管 MCP 工具上的程序化调用方也同样会被拒绝。在 Realtime 会话中，请使用可直接调用的函数工具或托管 MCP 工具；需要程序化工具调用的工作，则应委派给 Responses 智能体。

<Code lang="typescript" code={defineToolExample} />

#### 函数工具

函数工具与 `RealtimeSession` 在同一环境中运行。这意味着，如果会话在浏览器中运行，工具也会在浏览器中执行。如果需要执行敏感操作，请从工具内部调用后端，并让服务器执行需要特权的工作。

这样，浏览器端工具就可以充当连接服务器端逻辑的轻量级反向通道。例如，[`examples/realtime-next`](https://github.com/openai/openai-agents-js/tree/main/examples/realtime-next) 在浏览器中定义了 `refundBackchannel` 工具，它会将请求和当前对话历史记录转发给服务器上的 `handleRefundRequest(...)`。在服务器上，单独的 `Runner` 可以使用其他智能体或模型评估退款请求，然后将结果返回给语音会话。

#### 托管 MCP 工具

可以使用 `hostedMcpTool` 配置托管 MCP 工具，这些工具会远程执行。当 MCP 工具的可用性发生变化时，会话会发出 `mcp_tools_changed`。若要防止会话在 MCP 工具调用完成后自动触发模型响应，请设置 `automaticallyTriggerResponseForMcpToolCalls: false`。

当前经过筛选的 MCP 工具列表也可通过 `session.availableMcpTools` 获取。该属性和 `mcp_tools_changed` 事件都仅反映当前智能体上启用的托管 MCP 服务器，并且已应用智能体配置中的所有 `allowed_tools` 过滤器。

如果将安全服务器选择、请求头和审批视为连接前配置，托管 MCP 设置会更容易理解。在 `RealtimeSession.connect()` 打开传输连接之前，SDK 会解析当前智能体的托管 MCP 工具定义，并在发送给 Realtime API 的初始会话配置中包含受支持的 MCP 字段。

这一时机对于浏览器 WebRTC 应用尤其重要。临时客户端密钥始终在服务器上生成，因此，任何必须保密的托管 MCP 凭证或自定义 `headers` 都应作为初始 `session` 载荷的一部分，附加到服务器端的 `POST /v1/realtime/client_secrets` 请求中。不要将长期有效的凭证放入浏览器代码，也不要计划在 `connect()` 启动后再添加这些凭证。

在 Realtime API 层，后续的 `session.update` 调用仍然可以更改工具及其他可变会话字段；当当前智能体发生变化时，SDK 本身也会发送 `session.update`。不过，在浏览器应用中，应将安全的托管 MCP 初始化视为服务器端、连接前的事项，并确保浏览器端的 `RealtimeSession` 配置与服务器生成密钥时使用的配置保持一致。

#### 后台结果

工具执行期间，智能体无法处理用户的新请求。一种改善体验的方法是，让智能体在即将执行工具时进行说明，或说出特定短语，为工具执行争取一些时间。

如果函数工具完成后不应立即触发另一个模型响应，请从 `@openai/agents/realtime` 返回 `backgroundResult(output)`。这样会将工具输出发回会话，同时保留由您控制响应触发的能力。

#### 超时

函数工具的超时选项（`timeoutMs`、`timeoutBehavior`、`timeoutErrorFunction`）在 Realtime 会话中的工作方式相同。使用默认的 `error_as_result` 时，超时消息会作为工具输出发送。使用 `raise_exception` 时，会话会发出包含 [`ToolTimeoutError`](/openai-agents-js/openai/agents-core/classes/tooltimeouterror) 的 `error` 事件，并且不会为该调用发送工具输出。

#### 对话历史记录访问

除了智能体调用特定工具时传入的参数外，您还可以访问 Realtime 会话所追踪的当前对话历史记录快照。如果需要根据对话的当前状态执行更复杂的操作，或计划使用[工具进行委派](#delegation-through-tools)，此功能会很有用。

#### 工具执行前审批

如果使用 `needsApproval: true` 定义工具，智能体会在执行工具之前发出 `tool_approval_requested` 事件。

通过监听此事件，可以向用户显示界面，让其批准或拒绝工具调用。

使用 `await session.approve(request.approvalItem)` 或 `await session.reject(request.approvalItem)` 处理请求。对于函数工具，可以传入 `{ alwaysApprove: true }` 或 `{ alwaysReject: true }`，以便在会话剩余时间内对重复调用复用同一决定；还可以使用 `session.reject(request.approvalItem, { message: '...' })`，针对该次调用向模型返回自定义拒绝消息。托管 MCP 审批不支持持续生效的批准或拒绝；请改用托管 MCP 的 `allowedTools` 配置来限制这些工具。

如果未传入单次调用的拒绝 `message`，会话会回退到 `toolErrorFormatter`（如果已配置），然后再回退到 SDK 默认的拒绝文本。

默认情况下，函数工具输入护栏会在审批后、工具执行前立即运行。如果向 `new RealtimeSession(...)` 传入 `toolExecution: { preApprovalInputGuardrails: true }`，本地函数工具输入护栏也会在会话发出 `tool_approval_requested` 之前运行。护栏拒绝会将拒绝消息作为工具输出返回，并跳过审批事件。如果护栏允许调用，审批事件仍会发出；在调用 `session.approve(...)` 后、执行工具之前，护栏还会再次运行。

<Code lang="typescript" code={toolApprovalEventExample} />

<Aside type="note">语音智能体等待工具调用审批时，无法处理用户的新请求。</Aside>

### 护栏

护栏可用于监控智能体所说的内容是否违反一组规则，并立即中止响应。这些检查会针对智能体响应的转录文本流运行。在音频会话中，SDK 使用输出音频的转录文本和转录增量，因此关键前提是转录文本可用，而不是存在单独的文本输出模态。

随着模型响应逐步返回，您提供的护栏会异步运行，从而可以根据预定义的分类触发条件中止响应，例如"提及特定禁用词"。

护栏触发时，会话会发出 `guardrail_tripped` 事件。该事件还会提供一个 `details` 对象，其中包含触发护栏的 `itemId`。

<Code lang="typescript" code={guardrailsExample} />

默认情况下，护栏每生成 100 个字符运行一次，并在最终转录文本可用时再次运行。由于朗读文本通常比生成转录文本耗时更长，因此护栏往往能在用户听到不安全输出之前将其中止。

如果要修改此行为，可以向会话传入 `outputGuardrailSettings` 对象。

如果只想在响应结束时对完整生成的转录文本评估一次，请设置 `debounceTextLength: -1`。

## 对话状态与委派

### 对话历史记录管理

`RealtimeSession` 会自动维护本地 `history` 快照，用于追踪用户消息、助手输出、工具调用和截断状态。您可以在 UI 中渲染它、在工具内部检查它，或在需要更正或移除条目时更新它。

随着对话变化，会话会发出 `history_updated`。如果需要请求更改历史记录，请使用 `updateHistory()`。它会要求传输层比较当前历史记录的差异，并发送必要的删除/创建事件；当相应的对话事件返回时，本地 `session.history` 视图也会更新。

#### 限制

1. 目前无法在事后编辑函数工具调用。
2. 历史记录中的助手文本取决于可用的转录文本，其中包括 `output_audio.transcript`。
3. 因中断而截断的响应不会保留最终转录文本。
4. 最好将输入音频转录视为用户所说内容的粗略参考，而不是模型如何理解该音频的精确副本。
