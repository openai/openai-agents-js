- `OpenAIRealtimeWebRTC` 是最简单的浏览器方案，并会为您处理音频输入和输出。

## 会话设置

### 音频处理

某些传输层（如默认的 `OpenAIRealtimeWebRTC`）会自动为您处理音频输入和输出。对于其他传输机制（如 `OpenAIRealtimeWebSocket`），您需要自行处理会话音频：

当底层传输支持时，`session.muted` 会报告当前静音状态，`session.mute(true | false)` 可切换麦克风采集开关。`OpenAIRealtimeWebSocket` 不实现静音功能：`session.muted` 返回 `null`，且 `session.mute()` 会抛出异常，因此对于 websocket 方案，您应在本地暂停采集，并在麦克风需要重新启用之前停止调用 `sendAudio()`。

### 会话配置

在创建 [`RealtimeSession`](/openai-agents-js/openai/agents-realtime/classes/realtimesession/) 时配置会话本身，通常通过 `model` 选项和 `config` 对象完成。`connect(...)` 用于处理连接时的事项，例如凭据、端点 URL 和 SIP 通话附加，而不是任意会话字段。

<Code lang="typescript" code={configureSessionExample} />

在底层，SDK 会将此配置规范化为 Realtime [`session.update`](https://developers.openai.com/api/reference/resources/realtime/client-events/#session.update) 的结构。如果您需要 [RealtimeSessionConfig](/openai-agents-js/openai/agents-realtime/type-aliases/realtimesessionconfig/) 中没有对应属性的原始会话字段，请使用 `providerData`，或通过 `session.transport.sendEvent(...)` 发送原始 `session.update`。

优先使用较新的 SDK 配置结构，即 `outputModalities`、`audio.input` 和 `audio.output`。旧版 SDK 别名如 `modalities`、`inputAudioFormat`、`outputAudioFormat`、`inputAudioTranscription` 和 `turnDetection` 仍会为向后兼容而被规范化，但新代码应使用此处展示的嵌套 `audio` 结构。

对于支持推理的 Realtime 模型（如 `gpt-realtime-2`），请在会话配置中设置 `reasoning.effort`。更高的推理强度可能会增加延迟和 token 使用量。您也可以设置 `parallelToolCalls`，以控制模型是否可以并行调用多个工具。

对于语音到语音会话，通常选择 `outputModalities: ['audio']`，这样会获得音频输出以及转录文本。只有在您想要纯文本响应时，才切换为 `['text']`。

对于较新的参数，如果 [RealtimeSessionConfig](/openai-agents-js/openai/agents-realtime/type-aliases/realtimesessionconfig/) 中还没有对应参数，可以使用 `providerData`。传入 `providerData` 的任何内容都会作为原始 `session` 对象的一部分被转发。

可在构造时设置的其他 `RealtimeSession` 选项：

| 选项 | 类型 | 用途 |
| --- | --- | --- |
| `context` | `TContext` | 合并到会话上下文中的额外本地上下文。 |
| `historyStoreAudio` | `boolean` | 在本地历史快照中存储音频数据（默认禁用）。 |
| `outputGuardrails` | `RealtimeOutputGuardrail[]` | 会话的输出护栏（参见[护栏](#guardrails)）。 |
| `outputGuardrailSettings` | `{ debounceTextLength?: number }` | 护栏执行频率。默认值为 `100`；使用 `-1` 表示仅在完整文本可用时运行一次。 |
| `tracingDisabled` | `boolean` | 为会话禁用追踪。 |
| `groupId` | `string` | 在多个会话或后端运行之间对追踪进行分组。需要 `workflowName`。 |
| `traceMetadata` | `Record<string, any>` | 附加到会话追踪的自定义元数据。需要 `workflowName`。 |
| `workflowName` | `string` | 追踪工作流的友好名称。 |
| `automaticallyTriggerResponseForMcpToolCalls` | `boolean` | 在 MCP 工具调用完成后自动触发模型响应（默认：`true`）。 |

| 选项 | 类型 | 用途 |
| --- | --- | --- |

| `url` | `string` | 可选的自定义 Realtime 端点 URL。 |
| `callId` | `string` | 附加到现有的 SIP 发起通话/会话。 |

### 对话生命周期

`RealtimeSession` 构建在长生命周期的 Realtime 连接之上。它维护一份本地对话历史副本，监听传输事件，运行工具和输出护栏，并使活动智能体配置与传输保持同步。

底层 API 的行为仍然很重要：

- 成功连接后会先收到 `session.created` 事件，之后的配置更改会产生 `session.updated`。
- 大多数会话属性都可以随时间变化，但 `model` 不能在对话中途更改，`voice` 只能在会话产生音频输出之前更改，而追踪应当预先决定，因为 Realtime API 不允许在启用后修改追踪。
- Realtime API 当前将单个会话限制为 60 分钟。
- 输入音频转录是异步的，因此最新一次发言的转录可能会在响应生成已开始之后才到达。

在 SDK 层，`await session.connect()` 表示"传输已准备好，可以开始对话"，但具体时机会因传输方式而异：

- 在默认的浏览器 WebRTC 传输中，SDK 会在数据通道打开后立即发送初始 `session.update`，并尝试等待对应的 `session.updated` 事件后再解析 `connect()`。这样可以避免在您的 instructions、工具和模态应用之前，音频先到达服务器。如果该确认始终未到达，`connect()` 会在短暂超时后回退为直接完成。
- 在默认的服务端 WebSocket 传输中，`connect()` 会在 socket 打开且初始配置已发送后完成。因此，对应的 `session.updated` 事件可能会在 `connect()` 已完成后才到达。

如果您需要原始事件模型，请结合本页一起阅读官方的 [Realtime conversations guide](https://developers.openai.com/api/docs/guides/realtime-conversations/)。

## 交互流程

### 轮次检测与语音活动检测

默认情况下，Realtime 会话使用内置的语音活动检测（VAD），因此 API 可以判断用户何时开始或停止说话，以及何时创建响应。SDK 通过 `audio.input.turnDetection` 暴露这一能力。

<Code lang="typescript" code={turnDetectionExample} />

两种常见模式是：

- `semantic_vad`，旨在实现更自然的轮次边界，并且当用户听起来尚未说完时会多等待一会儿。
- `server_vad`，更依赖阈值驱动，并暴露 `threshold`、`prefixPaddingMs`、`silenceDurationMs` 和 `idleTimeoutMs` 等设置。

如果您想自行管理轮次边界，请将 `audio.input.turnDetection` 设为 `null`。官方的[语音活动检测指南](https://developers.openai.com/api/docs/guides/realtime-vad/)和 [Realtime conversations guide](https://developers.openai.com/api/docs/guides/realtime-conversations/#voice-activity-detection) 对底层行为有更详细的说明。

启用 VAD 后，在智能体说话时插话可以打断当前响应。在 WebSocket 传输上，SDK 会监听 `input_audio_buffer.speech_started`，将助手音频截断到用户实际听到的部分，并发出 `audio_interrupted` 事件。对于在 WebSocket 方案中自行管理播放的情况，这个事件尤其有用。

### 手动响应控制

在更高层的 SDK 中，`sendMessage()` 和 `addImage()` 默认会为您触发响应。当您处理原始传输事件、按住说话流程或自定义审核/校验步骤时，手动响应控制就很重要。

<Code lang="typescript" code={transportEventsExample} />

常见情况有两种：

1. 如果您通过 `audio.input.turnDetection = null` 完全禁用 VAD，则由您负责提交音频轮次，然后发送 `response.create`。
2. 如果您保持 VAD 启用，但将 `turnDetection.interruptResponse = false` 和 `turnDetection.createResponse = false`，API 仍会检测轮次，但将响应创建交由您控制。

第二种模式适用于您希望在模型响应前先检查或审核用户输入的场景。这与官方 [Realtime conversations 中关于禁用自动响应的说明](https://developers.openai.com/api/docs/guides/realtime-conversations/#keep-vad-but-disable-automatic-responses) 一致。

## 智能体能力

### 交接

与常规智能体类似，您可以使用交接将智能体拆分为多个智能体，并在它们之间进行编排，以提升性能并更好地限定问题范围。

<Code lang="typescript" code={multiAgentsExample} />

与常规智能体不同，交接在 Realtime Agents 中的行为略有不同。执行交接时，进行中的会话会原地更新为新的智能体配置。因此，新智能体会自动获得进行中的对话历史访问权限，并且当前不会应用输入过滤器。

由于会话保持实时进行，该会话使用的模型在交接期间不会改变。语音变更遵循底层 Realtime API 规则：只有在会话尚未产生音频输出之前才会生效。Realtime 交接主要用于在同一会话上切换不同的 `RealtimeAgent` 配置；如果您需要使用不同的模型，例如像 `gpt-5.4` 这样的推理模型，或委派给非实时后端智能体，请使用[通过工具委派](#delegation-through-tools)。

### 工具

与常规智能体一样，Realtime Agents 可以调用工具来执行操作。Realtime 支持**函数工具**（本地执行）和**托管 MCP 工具**（由 Realtime API 远程执行）。您可以使用与常规智能体相同的 `tool()` 辅助函数来定义函数工具。

<Code lang="typescript" code={defineToolExample} />

#### 函数工具

函数工具会在与您的 `RealtimeSession` 相同的环境中运行。这意味着，如果您的会话运行在浏览器中，工具也会在浏览器中执行。如果您需要执行敏感操作，请在工具内部调用后端，并让服务器执行特权工作。

这使浏览器侧工具可以充当通往服务端逻辑的轻量回传通道。例如，[`examples/realtime-next`](https://github.com/openai/openai-agents-js/tree/main/examples/realtime-next) 在浏览器中定义了一个 `refundBackchannel` 工具，将请求和当前对话历史转发到服务器上的 `handleRefundRequest(...)`，在那里，单独的 `Runner` 可以使用不同的智能体或模型评估退款请求，然后再将结果返回给语音会话。

#### 托管 MCP 工具

托管 MCP 工具可通过 `hostedMcpTool` 进行配置，并远程执行。当 MCP 工具可用性变化时，会话会发出 `mcp_tools_changed`。如果要防止会话在 MCP 工具调用完成后自动触发模型响应，请设置 `automaticallyTriggerResponseForMcpToolCalls: false`。

当前过滤后的 MCP 工具列表也可通过 `session.availableMcpTools` 获取。该属性和 `mcp_tools_changed` 事件都只反映活动智能体上启用的托管 MCP 服务器，并且已经应用了智能体配置中的任何 `allowed_tools` 过滤器。

如果将安全的服务器选择、headers 和审批视为连接前配置，托管 MCP 的设置会更容易理解。在 `RealtimeSession.connect()` 打开传输之前，SDK 会解析活动智能体的托管 MCP 工具定义，并将支持的 MCP 字段包含到发送给 Realtime API 的初始会话配置中。

这一时机在浏览器 WebRTC 应用中尤为重要。临时客户端密钥始终在您的服务器上签发，因此任何必须保密的托管 MCP 凭据或自定义 `headers`，都应作为初始 `session` 负载的一部分附加在服务端的 `POST /v1/realtime/client_secrets` 请求中。不要将长期凭据放在浏览器代码中，也不要计划在 `connect()` 开始后再添加。

在 Realtime API 层面，后续的 `session.update` 调用仍可更改工具和其他可变会话字段，而当活动智能体变化时，SDK 本身也会发送 `session.update`。不过，在浏览器应用中，您应将安全的 Hosted MCP 初始化视为服务端、连接前的事项，并让浏览器侧 `RealtimeSession` 配置与服务器签发的内容保持一致。

#### 后台结果

在工具执行期间，智能体将无法处理来自用户的新请求。一种改善体验的方法是指示智能体在即将执行工具时进行提示，或说一些特定短语，为工具执行争取时间。

如果某个函数工具应在完成后不立即触发另一次模型响应，请从 `@openai/agents/realtime` 返回 `backgroundResult(output)`。这样会将工具输出发送回会话，同时将响应触发权保留给您控制。

#### 超时

函数工具的超时选项（`timeoutMs`、`timeoutBehavior`、`timeoutErrorFunction`）在 Realtime 会话中的工作方式相同。使用默认的 `error_as_result` 时，超时消息会作为工具输出发送。使用 `raise_exception` 时，会话会发出带有 [`ToolTimeoutError`](/openai-agents-js/openai/agents-core/classes/tooltimeouterror) 的 `error` 事件，并且不会为该调用发送工具输出。

#### 工具执行前审批

如果您在定义工具时设置 `needsApproval: true`，智能体会在执行工具前发出 `tool_approval_requested` 事件。

通过监听该事件，您可以向用户显示一个 UI，以批准或拒绝该工具调用。

使用 `await session.approve(request.approvalItem)` 或 `await session.reject(request.approvalItem)` 处理该请求。对于函数工具，您可以传入 `{ alwaysApprove: true }` 或 `{ alwaysReject: true }`，以便在本次会话的后续重复调用中复用相同决策；也可以使用 `session.reject(request.approvalItem, { message: '...' })` 将自定义拒绝消息返回给模型，仅针对该次调用。托管 MCP 审批不支持粘性批准/拒绝；请改为使用托管 MCP 的 `allowedTools` 配置限制这些工具。

如果您没有传入单次调用的拒绝 `message`，会话会回退到 `toolErrorFormatter`（如果已配置），再回退到 SDK 默认的拒绝文本。

### 护栏

护栏提供了一种方式，用于监控智能体说出的内容是否违反了一组规则，并立即截断响应。这些检查会针对智能体响应的转录流运行。在音频会话中，SDK 使用输出音频转录及转录增量，因此关键前提是转录可用，而不是单独的文本输出模态。

您提供的护栏会在模型响应返回时异步运行，从而允许您基于预定义的分类触发器截断响应，例如"提到某个特定的禁用词"。

当护栏被触发时，会话会发出 `guardrail_tripped` 事件。该事件还会提供一个 `details` 对象，其中包含触发护栏的 `itemId`。

<Code lang="typescript" code={guardrailsExample} />

默认情况下，护栏每 100 个字符运行一次，并在最终转录可用时再运行一次。由于说出文本通常比生成转录花费更长时间，这通常可以让护栏在用户听到之前就截断不安全输出。

如果您想修改此行为，可以向会话传入一个 `outputGuardrailSettings` 对象。

当您只想在响应结束时、对完整生成的转录评估一次时，请设置 `debounceTextLength: -1`。

## 对话状态与委派

### 对话历史管理

`RealtimeSession` 会自动维护一个本地 `history` 快照，用于跟踪用户消息、助手输出、工具调用和截断状态。您可以在 UI 中渲染它，在工具中检查它，或在需要纠正或删除条目时更新它。

随着对话变化，会话会发出 `history_updated`。如果您需要请求历史变更，请使用 `updateHistory()`。它会要求传输对当前历史进行差异计算，并发送必要的删除/创建事件；当相应的对话事件返回时，本地的 `session.history` 视图也会更新。

<Code lang="typescript" code={updateHistoryExample} />

#### 限制

1. 当前您无法在事后编辑函数工具调用。
2. 历史中的助手文本取决于可用的转录内容，包括 `output_audio.transcript`。
3. 被打断而截断的响应不会保留最终转录。
4. 输入音频转录最好视为对用户所说内容的粗略指引，而不是模型如何理解该音频的精确副本。
