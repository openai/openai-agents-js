请尽早确定您的架构：

- `OpenAIRealtimeWebRTC` 是最简单的浏览器方案，并会为您处理音频输入/输出。
- `OpenAIRealtimeWebSocket` 提供更多控制能力，但您必须自行管理音频采集和播放。

- Realtime 交接会保持同一个实时会话模型。语音变更仅在会话尚未产生音频输出前生效。如果您需要不同的后端模型，请通过工具委派，而不是使用交接。

当底层传输支持时，`session.muted` 会报告当前静音状态，`session.mute(true | false)` 可切换麦克风采集。`OpenAIRealtimeWebSocket` 不实现静音：`session.muted` 返回 `null`，且 `session.mute()` 会抛错，因此在 websocket 场景下，您应在本地暂停采集，并停止调用 `sendAudio()`，直到需要再次开启麦克风。

<Code lang="typescript" code={configureSessionExample} />

在底层，SDK 会将该配置标准化为 Realtime [`session.update`](https://developers.openai.com/api/reference/resources/realtime/client-events/#session.update) 结构。如果您需要 [RealtimeSessionConfig](/openai-agents-js/openai/agents-realtime/type-aliases/realtimesessionconfig/) 中没有对应属性的原始会话字段，可使用 `providerData`，或通过 `session.transport.sendEvent(...)` 发送原始 `session.update`。

建议优先使用较新的 SDK 配置结构：`outputModalities`、`audio.input` 和 `audio.output`。旧的 SDK 别名（如 `modalities`、`inputAudioFormat`、`outputAudioFormat`、`inputAudioTranscription`、`turnDetection`）仍会为向后兼容而被标准化，但新代码应使用此处展示的嵌套 `audio` 结构。

对于语音到语音会话，通常选择 `outputModalities: ['audio']`，这会提供音频输出和转写。仅当您希望纯文本响应时，才切换为 `['text']`。

对于较新的参数，如果 [RealtimeSessionConfig](/openai-agents-js/openai/agents-realtime/type-aliases/realtimesessionconfig/) 中没有对应参数，可使用 `providerData`。传入 `providerData` 的内容都会作为原始 `session` 对象的一部分转发。

### 会话生命周期

`RealtimeSession` 位于长期 Realtime 连接之上。它维护会话历史的本地副本，监听传输事件，运行工具和输出护栏，并让当前智能体配置与传输保持同步。

底层 API 行为仍然重要：

- 连接成功会先收到 `session.created` 事件，后续配置变更会产生 `session.updated`。
- 大多数会话属性可随时间更改，但 `model` 不能在会话中途更改，`voice` 只能在会话产生音频输出前更改；追踪应在一开始就决定，因为 Realtime API 在启用后不允许修改追踪。
- Realtime API 当前将单个会话限制为 60 分钟。
- 输入音频转写是异步的，因此最近一次发言的转写可能在响应生成已经开始后才到达。

在 SDK 层，`await session.connect()` 表示“传输已足够就绪，可以开始会话”，但不同传输的具体时点不同：

- 在默认浏览器 WebRTC 传输中，SDK 会在数据通道打开后立即发送初始 `session.update`，并尽量在 `connect()` resolve 前等待对应的 `session.updated` 事件。这样可避免在您的 instructions、工具和模态应用前，音频已到达服务器。如果该确认迟迟未到，`connect()` 会在短暂超时后回退为直接 resolve。
- 在默认服务端 WebSocket 传输中，`connect()` 会在 socket 打开且初始配置已发送后 resolve。因此对应的 `session.updated` 事件可能在 `connect()` resolve 之后才到达。

如果您需要原始事件模型，请结合本页阅读官方 [Realtime conversations guide](https://developers.openai.com/api/docs/guides/realtime-conversations/)。

## 交互流程

### 轮次检测与语音活动检测

默认情况下，Realtime 会话使用内置语音活动检测（VAD），由 API 判断用户何时开始/停止说话以及何时创建响应。SDK 通过 `audio.input.turnDetection` 暴露该能力。

<Code lang="typescript" code={turnDetectionExample} />

两种常见模式：

- `semantic_vad`：目标是更自然的轮次边界，当用户听起来还没说完时会多等一会儿。
- `server_vad`：更偏阈值驱动，暴露 `threshold`、`prefixPaddingMs`、`silenceDurationMs`、`idleTimeoutMs` 等设置。

如果您想自行管理轮次边界，将 `audio.input.turnDetection` 设为 `null`。官方 [voice activity detection guide](https://developers.openai.com/api/docs/guides/realtime-vad/) 和 [Realtime conversations guide](https://developers.openai.com/api/docs/guides/realtime-conversations/#voice-activity-detection) 对底层行为有更详细说明。

### 打断机制

启用 VAD 时，用户在智能体说话过程中插话可打断当前响应。在 WebSocket 传输上，SDK 会监听 `input_audio_buffer.speech_started`，将助手音频截断为用户实际听到的部分，并发出 `audio_interrupted` 事件。对于在 WebSocket 场景下自行管理播放的情况，这个事件尤其有用。

### 手动响应控制

在更高层的 SDK 中，`sendMessage()` 和 `addImage()` 默认会为您触发响应。若您使用原始传输事件、按住说话流程或自定义审核/校验步骤，手动响应控制就很重要。

<Code lang="typescript" code={transportEventsExample} />

常见有两种情况：

1. 如果您通过 `audio.input.turnDetection = null` 完全禁用 VAD，则需要自行提交音频轮次并发送 `response.create`。
2. 如果您保留 VAD，但设置 `turnDetection.interruptResponse = false` 且 `turnDetection.createResponse = false`，API 仍会检测轮次，但响应创建由您控制。

第二种模式适用于您想在模型响应前先检查或审核用户输入的场景。它与官方 [Realtime conversations guidance on disabling automatic responses](https://developers.openai.com/api/docs/guides/realtime-conversations/#keep-vad-but-disable-automatic-responses) 一致。

## 智能体能力

### 交接

与常规智能体类似，您可以使用 handoffs 将智能体拆分为多个智能体，并在它们之间编排，以提升性能并更好地界定问题范围。

<Code lang="typescript" code={multiAgentsExample} />

与常规智能体不同，Realtime Agents 中的 handoffs 行为略有差异。执行交接后，进行中的会话会更新为新的智能体配置。因此，新智能体会自动访问当前会话历史，且目前不会应用输入过滤器。

由于会话保持在线，该会话的模型在交接期间不会变化。语音变更遵循底层 Realtime API 规则：仅在会话尚未产生音频输出前生效。Realtime 交接主要用于在同一会话上切换 `RealtimeAgent` 配置；若您需要使用不同模型（例如推理模型 `gpt-5.2`），或委派给非实时后端智能体，请使用[通过工具委派](#delegation-through-tools)。

### 工具

与常规智能体一样，Realtime Agents 可以调用工具执行操作。Realtime 支持**函数工具**（本地执行）和**托管 MCP 工具**（由 Realtime API 远程执行）。您可以使用与常规智能体相同的 `tool()` 辅助方法定义函数工具。

<Code lang="typescript" code={defineToolExample} />

#### 函数工具

函数工具与您的 `RealtimeSession` 在同一环境中运行。这意味着如果会话在浏览器中运行，工具也在浏览器中执行。若需执行敏感操作，请在工具内部调用后端，让服务器执行特权工作。

这样浏览器侧工具可作为服务器侧逻辑的轻量回传通道。例如，[`examples/realtime-next`](https://github.com/openai/openai-agents-js/tree/main/examples/realtime-next) 在浏览器中定义了 `refundBackchannel` 工具，它会把请求和当前会话历史转发到服务器的 `handleRefundRequest(...)`，再由单独的 `Runner` 使用不同智能体或模型评估退款，最后把结果返回语音会话。

#### 托管 MCP 工具

托管 MCP 工具可通过 `hostedMcpTool` 配置并远程执行。当 MCP 工具可用性变化时，会话会发出 `mcp_tools_changed`。若要防止在 MCP 工具调用完成后自动触发模型响应，请设置 `automaticallyTriggerResponseForMcpToolCalls: false`。

当前过滤后的 MCP 工具列表也可通过 `session.availableMcpTools` 获取。该属性和 `mcp_tools_changed` 事件都只反映活动智能体启用的托管 MCP 服务器，并已应用智能体配置中的 `allowed_tools` 过滤。

如果将安全的服务器选择、headers 和审批视为连接前配置，托管 MCP 的设置会更易理解。在 `RealtimeSession.connect()` 打开传输前，SDK 会解析活动智能体的托管 MCP 工具定义，并将支持的 MCP 字段包含在发送给 Realtime API 的初始会话配置中。

这一时序在浏览器 WebRTC 应用中尤其关键。临时 client secret 始终在您的服务器上签发，因此任何必须保密的托管 MCP 凭据或自定义 `headers`，都应在服务端 `POST /v1/realtime/client_secrets` 请求中作为初始 `session` 负载附加。不要把长期凭据放在浏览器代码里，也不要计划在 `connect()` 开始后再补加。

在 Realtime API 层，后续 `session.update` 仍可更改工具和其他可变会话字段，且当活动智能体变化时，SDK 自身也会发送 `session.update`。但在浏览器应用中，您应将安全的 Hosted MCP 初始化视为服务端、连接前事项，并保持浏览器侧 `RealtimeSession` 配置与服务器签发内容一致。

#### 后台结果

工具执行期间，智能体将无法处理来自用户的新请求。改善体验的一种方式是让智能体在执行工具前先告知用户，或说一些过渡语，为工具执行争取时间。

如果函数工具应在完成后不立即触发下一次模型响应，请从 `@openai/agents/realtime` 返回 `backgroundResult(output)`。这样会把工具输出回传到会话，同时将响应触发权交由您控制。

#### 超时

函数工具超时选项（`timeoutMs`、`timeoutBehavior`、`timeoutErrorFunction`）在 Realtime 会话中的行为相同。默认 `error_as_result` 下，超时消息会作为工具输出发送。使用 `raise_exception` 时，会话会发出携带 [`ToolTimeoutError`](/openai-agents-js/openai/agents-core/classes/tooltimeouterror) 的 `error` 事件，并且该次调用不会发送工具输出。

### 护栏

护栏可用于监控智能体输出是否违反一组规则，并立即截断响应。这些检查会针对智能体响应的转写流运行。在音频会话中，SDK 使用输出音频转写及转写增量，因此关键前提是转写可用，而不是单独的文本输出模态。

您提供的护栏会在模型响应返回过程中异步运行，使您可基于预定义分类触发器截断响应，例如“提到某个特定禁用词”。

当护栏触发时，会话会发出 `guardrail_tripped` 事件。事件还会提供 `details` 对象，其中包含触发护栏的 `itemId`。

<Code lang="typescript" code={guardrailsExample} />

如果您想修改此行为，可向会话传入 `outputGuardrailSettings` 对象。

## 会话状态与委派

### 会话历史管理

`RealtimeSession` 会自动维护本地 `history` 快照，用于跟踪用户消息、助手输出、工具调用和截断状态。您可在 UI 中渲染它、在工具中检查它，或在需要更正/删除条目时更新它。

会话变化时会发出 `history_updated`。若需请求历史变更，请使用 `updateHistory()`。它会让传输对当前历史做差异比较并发送必要的删除/创建事件；当对应会话事件返回后，本地 `session.history` 视图会更新。

<Code lang="typescript" code={updateHistoryExample} />

#### 限制

1. 目前无法在事后编辑函数工具调用。
2. 历史中的助手文本依赖可用转写，包括 `output_audio.transcript`。
3. 被打断而截断的响应不会保留最终转写。
4. 输入音频转写更适合作为用户所述内容的粗略参考，而非模型如何解释音频的精确副本。
