- `OpenAIRealtimeWebRTC` 是最简单的浏览器路径，并且会为您处理音频输入/输出。

当底层传输支持时，`session.muted` 会报告当前静音状态，`session.mute(true | false)` 会切换麦克风采集。`OpenAIRealtimeWebSocket` 不实现静音：`session.muted` 返回 `null`，且 `session.mute()` 会抛出异常，因此在 websocket 场景下，您应在本地暂停采集，并在麦克风需要恢复前停止调用 `sendAudio()`。

<Code lang="typescript" code={configureSessionExample} />

在底层，SDK 会将该配置标准化为 Realtime [`session.update`](https://developers.openai.com/api/reference/resources/realtime/client-events/#session.update) 结构。如果您需要 [RealtimeSessionConfig](/openai-agents-js/openai/agents-realtime/type-aliases/realtimesessionconfig/) 中没有对应属性的原始会话字段，请使用 `providerData`，或通过 `session.transport.sendEvent(...)` 发送原始 `session.update`。

建议优先使用较新的 SDK 配置结构：`outputModalities`、`audio.input` 和 `audio.output`。较旧的 SDK 别名（如 `modalities`、`inputAudioFormat`、`outputAudioFormat`、`inputAudioTranscription` 和 `turnDetection`）仍会为向后兼容而被标准化，但新代码应使用此处展示的嵌套 `audio` 结构。

对于语音到语音会话，通常选择 `outputModalities: ['audio']`，可获得音频输出和转写。仅在您需要纯文本响应时切换为 `['text']`。

对于较新的参数（在 [RealtimeSessionConfig](/openai-agents-js/openai/agents-realtime/type-aliases/realtimesessionconfig/) 中暂无对应参数），可以使用 `providerData`。传入 `providerData` 的内容会作为原始 `session` 对象的一部分被转发。

### 对话生命周期

`RealtimeSession` 构建在长期存活的 Realtime 连接之上。它会保留对话历史的本地副本、监听传输事件、运行工具和输出护栏，并使当前智能体配置与传输层保持同步。

底层 API 行为仍然重要：

- 连接成功后会先收到 `session.created` 事件，后续配置变更会产生 `session.updated`。
- 大多数会话属性可随时间变化，但 `model` 不能在对话中途更改，`voice` 只能在会话产生音频输出前更改，而追踪应提前决定，因为 Realtime API 不允许在启用后再修改追踪。
- Realtime API 当前将单个会话限制为 60 分钟。
- 输入音频转写是异步的，因此最新一次发言的转写可能会在响应生成开始后才到达。

在 SDK 层，`await session.connect()` 表示“传输层已准备到可开始对话”，但具体时点因传输方式而异：

- 在默认的浏览器 WebRTC 传输中，SDK 会在数据通道打开后立即发送初始 `session.update`，并尽量等待对应的 `session.updated` 事件再解析 `connect()`。这样可避免在 instructions、工具和模态应用前音频已到达服务器。若该确认始终未到达，`connect()` 会在短超时后回退为已解析。
- 在默认的服务端 WebSocket 传输中，`connect()` 会在 socket 打开且初始配置发送后即解析。因此匹配的 `session.updated` 事件可能在 `connect()` 解析后才到达。

如果您需要原始事件模型，请结合本页阅读官方 [Realtime conversations guide](https://developers.openai.com/api/docs/guides/realtime-conversations/)。

## 交互流程

### 轮次检测与语音活动检测

默认情况下，Realtime 会话使用内置语音活动检测（VAD），因此 API 可判断用户何时开始或停止说话，以及何时创建响应。SDK 通过 `audio.input.turnDetection` 暴露该能力。

<Code lang="typescript" code={turnDetectionExample} />

两种常见模式：

- `semantic_vad`：目标是更自然的轮次边界，当用户听起来还没说完时会多等一会儿。
- `server_vad`：更依赖阈值，并暴露 `threshold`、`prefixPaddingMs`、`silenceDurationMs`、`idleTimeoutMs` 等设置。

如果您想自行管理轮次边界，请将 `audio.input.turnDetection` 设为 `null`。官方 [voice activity detection guide](https://developers.openai.com/api/docs/guides/realtime-vad/) 和 [Realtime conversations guide](https://developers.openai.com/api/docs/guides/realtime-conversations/#voice-activity-detection) 对底层行为有更详细说明。

有两种常见情况：

1. 若您通过 `audio.input.turnDetection = null` 完全禁用 VAD，则需自行提交音频轮次并发送 `response.create`。

## 智能体能力

### 交接

与常规智能体类似，您可以使用交接将智能体拆分为多个智能体，并在它们之间进行编排，以提升性能并更好地限定问题范围。

<Code lang="typescript" code={multiAgentsExample} />

与常规智能体不同，交接在 Realtime Agents 中行为略有不同。执行交接时，进行中的会话会就地更新为新的智能体配置。因此，新智能体会自动访问进行中的对话历史，且当前不会应用输入过滤器。

由于会话保持在线，交接期间该会话的模型不会变化。语音切换遵循底层 Realtime API 规则：仅在会话产生音频输出前有效。实时交接主要用于在同一会话中切换 `RealtimeAgent` 配置；如果您需要不同模型（例如推理模型 `gpt-5.2`），或要委派给非实时后端智能体，请使用[通过工具进行委派](#delegation-through-tools)。

### 工具

与常规智能体一样，Realtime Agents 可以调用工具执行操作。Realtime 支持**函数工具**（本地执行）和**托管 MCP 工具**（由 Realtime API 远程执行）。您可以使用与常规智能体相同的 `tool()` 辅助函数定义函数工具。

<Code lang="typescript" code={defineToolExample} />

#### 函数工具

函数工具运行在与 `RealtimeSession` 相同的环境中。这意味着如果会话在浏览器中运行，工具也在浏览器中执行。若需执行敏感操作，请在工具内部调用后端，由服务器完成特权工作。

这使浏览器侧工具可作为服务器侧逻辑的轻量回传通道。例如，[`examples/realtime-next`](https://github.com/openai/openai-agents-js/tree/main/examples/realtime-next) 在浏览器中定义了 `refundBackchannel` 工具，它会将请求和当前对话历史转发到服务器上的 `handleRefundRequest(...)`，然后由独立的 `Runner` 使用不同智能体或模型评估退款，再把结果返回给语音会话。

#### 托管 MCP 工具

托管 MCP 工具可通过 `hostedMcpTool` 配置并远程执行。当 MCP 工具可用性变化时，会话会发出 `mcp_tools_changed`。若要阻止会话在 MCP 工具调用完成后自动触发模型响应，请设置 `automaticallyTriggerResponseForMcpToolCalls: false`。

当前筛选后的 MCP 工具列表也可通过 `session.availableMcpTools` 获取。该属性和 `mcp_tools_changed` 事件都仅反映当前智能体上启用的托管 MCP 服务器，并已应用智能体配置中的 `allowed_tools` 过滤。

托管 MCP 的设置在“连接前配置”思路下最易理解：将安全服务器选择、headers 和审批都视为连接前配置。在 `RealtimeSession.connect()` 打开传输前，SDK 会解析当前智能体的托管 MCP 工具定义，并将支持的 MCP 字段纳入发送给 Realtime API 的初始会话配置。

这一时机在浏览器 WebRTC 应用中尤其关键。临时客户端密钥始终由您的服务器签发，因此任何必须保密的托管 MCP 凭据或自定义 `headers`，都应作为初始 `session` 负载的一部分附加在服务器侧 `POST /v1/realtime/client_secrets` 请求中。不要把长期凭据放在浏览器代码里并计划在 `connect()` 启动后再补加。

在 Realtime API 层面，后续 `session.update` 仍可更改工具和其他可变会话字段，且 SDK 在活动智能体变化时也会发送 `session.update`。但在浏览器应用中，您应将安全的 Hosted MCP 初始化视为服务器侧、连接前事项，并让浏览器侧 `RealtimeSession` 配置与服务器签发内容保持一致。

#### 后台结果

工具执行期间，智能体将无法处理用户的新请求。提升体验的一种方式是让智能体在即将执行工具时进行提示，或说一些特定短语为工具执行争取时间。

如果函数工具应在完成后不立即触发下一次模型响应，请从 `@openai/agents/realtime` 返回 `backgroundResult(output)`。这会将工具输出回传到会话，同时将响应触发权交由您控制。

#### 超时

函数工具超时选项（`timeoutMs`、`timeoutBehavior`、`timeoutErrorFunction`）在 Realtime 会话中的行为相同。默认 `error_as_result` 下，超时消息会作为工具输出发送；`raise_exception` 下，会话会发出带有 [`ToolTimeoutError`](/openai-agents-js/openai/agents-core/classes/tooltimeouterror) 的 `error` 事件，且不会为该次调用发送工具输出。

#### 工具执行前审批

如果您在定义工具时设置 `needsApproval: true`，智能体会在执行工具前发出 `tool_approval_requested` 事件。

监听该事件后，您可以向用户展示 UI 以批准或拒绝该工具调用。

使用 `await session.approve(request.approvalItem)` 或 `await session.reject(request.approvalItem)` 处理该请求。对于函数工具，您可以传入 `{ alwaysApprove: true }` 或 `{ alwaysReject: true }`，以在本次会话余下时间复用相同决策；还可以使用 `session.reject(request.approvalItem, { message: '...' })` 为该次调用向模型返回自定义拒绝消息。托管 MCP 审批不支持粘性批准/拒绝；请改用托管 MCP 的 `allowedTools` 配置来限制这些工具。

### 护栏

护栏可用于监控智能体输出是否违反规则，并立即截断响应。这些检查针对智能体响应的转写流执行。在音频会话中，SDK 使用输出音频转写及转写增量，因此关键前提是转写可用，而非单独的文本输出模态。

您提供的护栏会在模型响应返回时异步运行，使您可基于预定义分类触发器截断响应，例如“提到某个禁用词”。

当护栏触发时，会话会发出 `guardrail_tripped` 事件。事件还会提供 `details` 对象，其中包含触发护栏的 `itemId`。

如果您想调整此行为，可向会话传入 `outputGuardrailSettings` 对象。

## 对话状态与委派

### 对话历史管理

`RealtimeSession` 会自动维护本地 `history` 快照，用于跟踪用户消息、助手输出、工具调用和截断状态。您可以在 UI 中渲染它、在工具内检查它，或在需要修正/删除条目时更新它。

随着对话变化，会话会发出 `history_updated`。若需请求历史变更，请使用 `updateHistory()`。它会要求传输层对当前历史做差异比较，并发送必要的删除/创建事件；当对应对话事件返回后，本地 `session.history` 视图会更新。

<Code lang="typescript" code={updateHistoryExample} />

#### 限制

1. 目前无法在事后编辑函数工具调用。
2. 历史中的助手文本依赖可用转写，包括 `output_audio.transcript`。
3. 被打断而截断的响应不会保留最终转写。
4. 输入音频转写更适合作为用户发言的大致参考，而非模型对音频理解的精确拷贝。
