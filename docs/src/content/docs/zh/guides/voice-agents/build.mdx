请尽早选择您的架构：

- `OpenAIRealtimeWebRTC` 是最简单的浏览器路径，并会为您处理音频输入/输出。

## 会话设置

### 音频处理

某些传输层（例如默认的 `OpenAIRealtimeWebRTC`）会自动为您处理音频输入和输出。对于其他传输机制（例如 `OpenAIRealtimeWebSocket`），您需要自行处理会话音频：

当底层传输支持时，`session.muted` 会报告当前静音状态，`session.mute(true | false)` 会切换麦克风采集。`OpenAIRealtimeWebSocket` 不实现静音：`session.muted` 返回 `null`，`session.mute()` 会抛出异常。因此，对于 websocket 设置，您应在自己这侧暂停采集，并停止调用 `sendAudio()`，直到麦克风需要再次开启。

### 会话配置

创建 [`RealtimeSession`](/openai-agents-js/openai/agents-realtime/classes/realtimesession/) 时配置会话本身，通常通过 `model` 选项和 `config` 对象进行配置。`connect(...)` 用于连接时相关事项，例如凭据、端点 URL 和 SIP 通话附加，而不是用于任意会话字段。

<Code lang="typescript" code={configureSessionExample} />

在底层，SDK 会将此配置规范化为 Realtime [`session.update`](https://developers.openai.com/api/reference/resources/realtime/client-events/#session.update) 形状。如果您需要某个原始会话字段，但该字段在 [RealtimeSessionConfig](/openai-agents-js/openai/agents-realtime/type-aliases/realtimesessionconfig/) 中没有对应属性，请使用 `providerData`，或通过 `session.transport.sendEvent(...)` 发送原始 `session.update`。

优先使用较新的 SDK 配置形状，包括 `outputModalities`、`audio.input` 和 `audio.output`。较旧的 SDK 别名（例如 `modalities`、`inputAudioFormat`、`outputAudioFormat`、`inputAudioTranscription` 和 `turnDetection`）仍会为了向后兼容而被规范化，但新代码应使用此处展示的嵌套 `audio` 结构。

对于具备推理能力的 Realtime 模型（例如 `gpt-realtime-2`），请在会话配置中设置 `reasoning.effort`。更高的推理强度可能会增加延迟和 token 使用量。您也可以在需要控制模型是否可以并行调用多个工具时设置 `parallelToolCalls`。

对于语音到语音会话，通常选择 `outputModalities: ['audio']`，它会为您提供音频输出和转写。仅当您需要纯文本响应时，才切换到 `['text']`。

对于较新的参数，如果在 [RealtimeSessionConfig](/openai-agents-js/openai/agents-realtime/type-aliases/realtimesessionconfig/) 中没有对应参数，您可以使用 `providerData`。传入 `providerData` 的任何内容都会作为原始 `session` 对象的一部分转发。

您可以在构造时设置的其他 `RealtimeSession` 选项：

### 对话生命周期

`RealtimeSession` 位于长连接 Realtime 连接之上。它会保留对话历史的本地副本，监听传输事件，运行工具和输出护栏，并让活跃智能体配置与传输保持同步。

底层 API 行为仍然很重要：

- 成功连接会以 `session.created` 事件开始，后续配置变更会产生 `session.updated`。
- 大多数会话属性都可以随时间变更，但 `model` 不能在对话中途变更，`voice` 只能在会话产生音频输出之前变更，而且追踪应提前决定，因为 Realtime API 不允许在启用追踪后修改它。
- Realtime API 目前将单个会话限制为 60 分钟。
- 输入音频转写是异步的，因此最新话语的转写可能会在响应生成已经开始之后才到达。

在 SDK 层，`await session.connect()` 表示"传输已准备到足以开始对话"，但确切时间点因传输而异：

- 在默认的浏览器 WebRTC 传输中，SDK 会在数据通道打开后立即发送初始 `session.update`，并尝试等待对应的 `session.updated` 事件后再解析 `connect()`。这样做是为了避免音频在您的 instructions、tools 和 modalities 应用之前到达服务器。如果该确认从未到达，`connect()` 会在短暂超时后回退为解析。
- 在默认的服务器端 WebSocket 传输中，`connect()` 会在 socket 打开且初始配置已发送后解析。因此，匹配的 `session.updated` 事件可能会在 `connect()` 已经解析之后才到达。

如果您需要原始事件模型，请将官方 [Realtime 对话指南](https://developers.openai.com/api/docs/guides/realtime-conversations/) 与本页一起阅读。

## 交互流程

### 轮次检测和语音活动检测

默认情况下，Realtime 会话使用内置语音活动检测（VAD），以便 API 可以决定用户何时开始或停止说话，以及何时创建响应。SDK 通过 `audio.input.turnDetection` 暴露此功能。

<Code lang="typescript" code={turnDetectionExample} />

两种常见模式是：

- `semantic_vad`，旨在提供更自然的轮次边界，并且当用户听起来还没说完时，可以多等待一会儿。
- `server_vad`，更依赖阈值，并暴露 `threshold`、`prefixPaddingMs`、`silenceDurationMs` 和 `idleTimeoutMs` 等设置。

如果您想自行管理轮次边界，请将 `audio.input.turnDetection` 设置为 `null`。官方[语音活动检测指南](https://developers.openai.com/api/docs/guides/realtime-vad/)和 [Realtime 对话指南](https://developers.openai.com/api/docs/guides/realtime-conversations/#voice-activity-detection)更详细地描述了底层行为。

### 手动响应控制

在较高的 SDK 层，`sendMessage()` 和 `addImage()` 默认会为您触发响应。当您处理原始传输事件、按键通话流程或自定义审核/验证步骤时，手动响应控制很重要。

有两种常见情况：

1. 如果您通过 `audio.input.turnDetection = null` 完全禁用 VAD，则需要负责提交音频轮次，然后发送 `response.create`。

## 智能体能力

### 交接

与常规智能体类似，您可以使用交接将智能体拆分为多个智能体，并在它们之间进行编排，以提升性能并更好地限定问题范围。

<Code lang="typescript" code={multiAgentsExample} />

与常规智能体不同，交接在实时智能体中的行为略有不同。执行交接时，正在进行的会话会被原地更新为新的智能体配置。因此，新智能体会自动访问正在进行的对话历史，并且输入过滤器目前不会应用。

由于会话保持活动状态，该会话的模型在交接期间不会变化。语音变更遵循底层 Realtime API 规则：它们仅在会话尚未产生音频输出之前生效。Realtime 交接主要用于在同一会话上切换不同的 `RealtimeAgent` 配置；如果您需要使用不同模型，例如像 `gpt-5.4` 这样的推理模型，或委托给非实时后端智能体，请使用[通过工具委托](#delegation-through-tools)。

### 工具

与常规智能体一样，实时智能体可以调用工具来执行操作。Realtime 支持**函数工具**（本地执行）和**托管 MCP 工具**（由 Realtime API 远程执行）。您可以使用与常规智能体相同的 `tool()` 辅助函数定义函数工具。

<Code lang="typescript" code={defineToolExample} />

#### 函数工具

函数工具会在与您的 `RealtimeSession` 相同的环境中运行。这意味着，如果您的会话在浏览器中运行，工具就在浏览器中执行。如果您需要执行敏感操作，请在工具内部调用您的后端，并让服务器执行特权工作。

这允许浏览器侧工具充当通往服务器端逻辑的轻量级后台通道。例如，[`examples/realtime-next`](https://github.com/openai/openai-agents-js/tree/main/examples/realtime-next) 在浏览器中定义了一个 `refundBackchannel` 工具，它会将请求和当前对话历史转发到服务器上的 `handleRefundRequest(...)`，在那里，单独的 `Runner` 可以使用不同的智能体或模型来评估退款，然后将结果返回给语音会话。

#### 托管 MCP 工具

托管 MCP 工具可以使用 `hostedMcpTool` 配置，并会被远程执行。当 MCP 工具可用性发生变化时，会话会发出 `mcp_tools_changed`。如果要防止会话在 MCP 工具调用完成后自动触发模型响应，请设置 `automaticallyTriggerResponseForMcpToolCalls: false`。

当前经过过滤的 MCP 工具列表也可以通过 `session.availableMcpTools` 获取。该属性和 `mcp_tools_changed` 事件都只反映活跃智能体上启用的托管 MCP 服务器，并且会应用智能体配置中的任何 `allowed_tools` 过滤器。

如果您将安全服务器选择、headers 和审批视为连接前配置，托管 MCP 设置会最容易理解。在 `RealtimeSession.connect()` 打开传输之前，SDK 会解析活跃智能体的托管 MCP 工具定义，并将支持的 MCP 字段包含在发送给 Realtime API 的初始会话配置中。

这个时机在浏览器 WebRTC 应用中最为重要。临时客户端密钥始终在您的服务器上签发，因此任何必须保密的托管 MCP 凭据或自定义 `headers` 都应作为初始 `session` 负载的一部分，附加到该服务器端 `POST /v1/realtime/client_secrets` 请求中。不要将长期凭据放入浏览器代码中，并计划在 `connect()` 启动后再添加它们。

在 Realtime API 层面，后续 `session.update` 调用仍可以更改工具和其他可变会话字段，SDK 本身也会在活跃智能体变化时发送 `session.update`。不过，在浏览器应用中，您应将安全的托管 MCP 初始化视为服务器端、连接前事项，并让浏览器端 `RealtimeSession` 配置与服务器签发的内容保持一致。

#### 后台结果

工具执行期间，智能体将无法处理来自用户的新请求。改善体验的一种方式是让智能体在即将执行工具时进行说明，或说出特定短语，为智能体执行工具争取一些时间。

如果某个函数工具应在完成后不立即触发另一个模型响应，请从 `@openai/agents/realtime` 返回 `backgroundResult(output)`。这会将工具输出发送回会话，同时将响应触发留给您控制。

#### 超时

函数工具超时选项（`timeoutMs`、`timeoutBehavior`、`timeoutErrorFunction`）在 Realtime 会话中的工作方式相同。使用默认的 `error_as_result` 时，超时消息会作为工具输出发送。使用 `raise_exception` 时，会话会发出带有 [`ToolTimeoutError`](/openai-agents-js/openai/agents-core/classes/tooltimeouterror) 的 `error` 事件，并且不会为该调用发送工具输出。

#### 工具执行前审批

如果您使用 `needsApproval: true` 定义工具，智能体会在执行工具之前发出 `tool_approval_requested` 事件。

通过监听此事件，您可以向用户展示 UI，以批准或拒绝该工具调用。

使用 `await session.approve(request.approvalItem)` 或 `await session.reject(request.approvalItem)` 解决该请求。对于函数工具，您可以传入 `{ alwaysApprove: true }` 或 `{ alwaysReject: true }`，以便在会话剩余期间对重复调用复用相同决策；也可以使用 `session.reject(request.approvalItem, { message: '...' })` 将该特定调用的自定义拒绝消息发送回模型。托管 MCP 审批不支持粘性批准/拒绝；请改用托管 MCP 的 `allowedTools` 配置限制这些工具。

如果您没有传入单次调用的拒绝 `message`，会话会回退到 `toolErrorFormatter`（如果已配置），然后回退到 SDK 默认拒绝文本。

默认情况下，函数工具输入护栏会在审批之后、工具执行之前立即运行。如果您向 `new RealtimeSession(...)` 传入 `toolExecution: { preApprovalInputGuardrails: true }`，本地函数工具输入护栏也会在会话发出 `tool_approval_requested` 之前运行。护栏拒绝会将拒绝消息作为工具输出发送回去，并跳过审批事件。如果护栏允许该调用，审批事件仍会发出，并且护栏会在 `session.approve(...)` 之后、执行之前再次运行。

### 护栏

护栏提供了一种方式，用于监控智能体所说的内容是否违反一组规则，并立即截断响应。这些检查会针对智能体响应的转写流运行。在音频会话中，SDK 使用输出音频转写和转写增量，因此重要的前提是转写可用，而不是单独的文本输出模态。

您提供的护栏会在模型响应返回过程中异步运行，使您可以基于预定义的分类触发条件截断响应，例如"提到了某个特定禁用词"。

当护栏触发时，会话会发出 `guardrail_tripped` 事件。该事件还会提供一个 `details` 对象，其中包含触发护栏的 `itemId`。

<Code lang="typescript" code={guardrailsExample} />

默认情况下，护栏每 100 个字符运行一次，并在最终转写可用时再次运行。由于说出文本通常比生成转写耗时更长，这通常能让护栏在用户听到不安全输出之前将其截断。

如果您想修改此行为，可以向会话传入 `outputGuardrailSettings` 对象。

当您只想在响应结束时对完全生成的转写评估一次时，请设置 `debounceTextLength: -1`。

## 对话状态和委托

### 对话历史管理

`RealtimeSession` 会自动维护本地 `history` 快照，用于跟踪用户消息、assistant 输出、工具调用和截断状态。您可以在 UI 中渲染它，在工具内部检查它，或在需要修正或移除条目时更新它。

随着对话变化，会话会发出 `history_updated`。如果您需要请求历史变更，请使用 `updateHistory()`。它会要求传输对当前历史进行差异比较，并发送必要的删除/创建事件；本地 `session.history` 视图会在对应的对话事件从传输返回时更新。

#### 限制

1. 您目前无法在事后编辑函数工具调用。
2. 历史记录中的 assistant 文本取决于可用的转写，包括 `output_audio.transcript`。
3. 被打断截断的响应不会保留最终转写。
4. 输入音频转写最好被视为用户所说内容的粗略指南，而不是模型如何解读音频的精确副本。
