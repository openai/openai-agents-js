## 模型选择

### 默认模型

当您在初始化 `Agent` 时未指定模型，将使用默认模型。当前默认值是 [`gpt-4.1`](https://developers.openai.com/api/docs/models/gpt-4.1)，以保证兼容性和低延迟。如果您有访问权限，我们建议将智能体设置为 [`gpt-5.4`](https://developers.openai.com/api/docs/models/gpt-5.4) 以获得更高质量，同时显式设置 `modelSettings`。

如果您想切换到其他模型（如 [`gpt-5.4`](https://developers.openai.com/api/docs/models/gpt-5.4)），有两种方式可为智能体配置。

第一种，如果您希望所有未设置自定义模型的智能体都统一使用某个特定模型，请在运行智能体之前设置 `OPENAI_DEFAULT_MODEL` 环境变量。

```bash
export OPENAI_DEFAULT_MODEL=gpt-5.4
node my-awesome-agent.js
```

第二种，您可以为 `Runner` 实例设置默认模型。如果某个智能体未设置模型，则会使用该 `Runner` 的默认模型。

## OpenAI provider 配置

### OpenAI provider

默认 `ModelProvider` 使用 OpenAI API 解析名称。它支持两个不同端点：

| API | 用法 | 调用 `setOpenAIAPI()` |
| --- | --- | --- |
| Chat Completions | 标准聊天与函数调用 | `setOpenAIAPI('chat_completions')` |
| Responses | 新的流式优先生成式 API（工具调用、灵活输出） | `setOpenAIAPI('responses')` _（默认）_ |

#### Responses WebSocket 传输

当您将 OpenAI provider 与 Responses API 一起使用时，可以通过 WebSocket 传输发送请求，而不是默认的 HTTP 传输。

可通过 `setOpenAIResponsesTransport('websocket')` 全局启用，或按 provider 启用：`new OpenAIProvider({ useResponses: true, useResponsesWebSocket: true })`。

仅为了使用 WebSocket 传输，您不需要 `withResponsesWebSocketSession(...)` 或自定义 `OpenAIProvider`。如果每次运行/请求都重新连接是可接受的，那么启用 `setOpenAIResponsesTransport('websocket')` 后，现有的 `run()` / `Runner.run()` 用法仍可继续工作。

传输选择遵循模型解析路径：

- `setOpenAIResponsesTransport('websocket')` 仅影响那些后续会通过 OpenAI provider 解析、且使用 Responses API 的字符串模型名。
- 如果您向 `Agent` 或 `Runner` 传入具体 `Model` 实例，会按原样使用该实例。`OpenAIResponsesWSModel` 保持 WebSocket，`OpenAIResponsesModel` 保持 HTTP，`OpenAIChatCompletionsModel` 保持 Chat Completions。
- 如果您提供了自己的 `modelProvider`，则由该 provider 控制模型解析。请在其中启用 WebSocket，而不是依赖全局设置器。
- 如果您通过代理、网关或其他 OpenAI 兼容端点路由，目标端必须支持 WebSocket `/responses` 端点。您可能还需要显式设置 `websocketBaseURL`。

仅当您希望优化连接复用并更明确地管理 websocket provider 生命周期时，才使用 `withResponsesWebSocketSession(...)` 或自定义 `OpenAIProvider` / `Runner`：

- `withResponsesWebSocketSession(...)`：便捷的作用域生命周期，回调结束后自动清理。
- 自定义 `OpenAIProvider` / `Runner`：在您自己的应用架构中显式控制生命周期（包括关闭清理）。

尽管名字里有 Session，`withResponsesWebSocketSession(...)` 是传输生命周期辅助工具，与[会话指南](/openai-agents-js/zh/guides/sessions)中描述的内存 `Session` 接口无关。

如果您使用 websocket 代理或网关，请在 `OpenAIProvider` 上配置 `websocketBaseURL`，或设置 `OPENAI_WEBSOCKET_BASE_URL`。

如果您自行实例化 `OpenAIProvider`，请注意：由 websocket 支持的 Responses 模型包装器默认会被缓存以复用连接。请在关闭时调用 `await provider.close()` 释放这些缓存连接。`withResponsesWebSocketSession(...)` 在很大程度上就是为您管理该生命周期：它会创建启用 websocket 的 provider 和 runner，传给您的回调，并在之后始终关闭 provider。对临时 provider 使用 `providerOptions`，对回调作用域 runner 默认值使用 `runnerConfig`。

完整的流式传输 + 人工干预示例（使用 Responses WebSocket 传输）请参见 [`examples/basic/stream-ws.ts`](https://github.com/openai/openai-agents-js/tree/main/examples/basic/stream-ws.ts)。

#### 仅 Responses 的延迟工具加载

`toolSearchTool()`、`toolNamespace()`，以及设置了 `deferLoading: true` 的函数工具或远程 MCP 服务器工具，都要求使用 OpenAI Responses API。Chat Completions provider 会拒绝命名空间或延迟加载的函数工具，AI SDK 适配器也不支持延迟 Responses 工具加载流程。需要工具搜索时，请直接使用 Responses 模型。

工具搜索仅在 GPT-5.4 及更新且在 Responses API 中支持该能力的模型版本上可用。

当一次运行包含延迟工具时，请将 `toolSearchTool()` 添加到同一个智能体，并将 `modelSettings.toolChoice` 保持为 `'auto'`。SDK 不允许您按名称强制内置 `tool_search` 工具或延迟函数工具，因为模型需要自行决定何时加载这些定义。完整配置请参见[工具指南](/openai-agents-js/zh/guides/tools#deferred-tool-loading-with-tool-search)和官方 [OpenAI tool search guide](https://developers.openai.com/api/docs/guides/tools-tool-search)。

## 模型行为与提示

### ModelSettings

`ModelSettings` 映射 OpenAI 参数，但与 provider 无关。

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `temperature` | `number` | 创造性与确定性之间的权衡。 |
| `topP` | `number` | 核采样。 |
| `frequencyPenalty` | `number` | 惩罚重复 token。 |
| `presencePenalty` | `number` | 鼓励生成新 token。 |
| `toolChoice` | `'auto' \| 'required' \| 'none' \| string` | 参见[强制使用工具](/openai-agents-js/zh/guides/agents#forcing-tool-use)。在 OpenAI Responses 上，`toolChoice: 'computer'` 会在可用时强制使用 GA 内置 computer 工具。 |
| `parallelToolCalls` | `boolean` | 在支持时允许并行函数调用。 |
| `truncation` | `'auto' \| 'disabled'` | token 截断策略。 |
| `maxTokens` | `number` | 响应中的最大 token 数。 |
| `store` | `boolean` | 持久化响应，用于检索 / RAG 工作流。 |
| `promptCacheRetention` | `'in-memory' \| '24h' \| null` | 在支持时控制 provider 的提示缓存保留策略。 |
| `reasoning.effort` | `'none' \| 'minimal' \| 'low' \| 'medium' \| 'high' \| 'xhigh'` | gpt-5.x 模型的推理强度。 |
| `reasoning.summary` | `'auto' \| 'concise' \| 'detailed'` | 控制模型返回的推理摘要详细程度。 |
| `text.verbosity` | `'low' \| 'medium' \| 'high'` | gpt-5.x 等模型的文本详细程度。 |
| `providerData` | `Record<string, any>` | provider 特定透传选项，会转发到底层模型。 |

### Prompt

智能体可通过 `prompt` 参数配置，表示使用服务器端存储的 prompt 配置来控制智能体行为。目前此选项仅在使用 OpenAI [Responses API](https://platform.openai.com/docs/api-reference/responses) 时受支持。

`prompt` 可以是静态对象，也可以是运行时返回对象的函数。回调形态请参见[动态提示](/openai-agents-js/zh/guides/agents#dynamic-prompts)。

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `promptId` | `string` | prompt 的唯一标识符。 |
| `version` | `string` | 您要使用的 prompt 版本。 |
| `variables` | `object` | 要替换进 prompt 的键值变量。值可以是字符串，或文本、图像、文件等内容输入类型。 |

<Code lang="typescript" code={promptIdExample} title="带 prompt 的智能体" />

任何额外的智能体配置（如工具或 instructions）都会覆盖您在已存储 prompt 中配置的值。

当已存储 prompt 已定义模型时，SDK 不会发送智能体默认模型，除非您显式覆盖它。这对 `computerTool()` 很关键：由 prompt 管理的运行默认保持旧版预览 wire 形态以兼容。若要在 prompt 管理的运行中启用 GA Responses computer 工具，请显式设置 `modelSettings.toolChoice: 'computer'`，或发送显式模型（如 [`gpt-5.4`](https://developers.openai.com/api/docs/models/gpt-5.4)）。有关计算机操作的更多细节，请参见[工具](/openai-agents-js/zh/guides/tools#computer-tool-specifics)。
