## 模型选择

### 默认模型

当您在初始化 `Agent` 时未指定模型，将使用默认模型。当前默认值是 [`gpt-5.4-mini`](https://developers.openai.com/api/docs/models/gpt-5.4-mini)，并使用 `reasoning.effort: "none"` 和 `text.verbosity: "low"`，以平衡质量与延迟。

如果您想切换到其他模型，例如 [`gpt-5.5`](https://developers.openai.com/api/docs/models/gpt-5.5)，有两种方式可以为您的智能体进行配置。

首先，如果您希望所有未设置自定义模型的智能体统一使用某个特定模型，请在运行智能体之前设置 `OPENAI_DEFAULT_MODEL` 环境变量。

```bash
export OPENAI_DEFAULT_MODEL=gpt-5.5
node my-awesome-agent.js
```

其次，您可以为 `Runner` 实例设置默认模型。如果您没有为某个智能体设置模型，则会使用该 `Runner` 的默认模型。

当您以这种方式使用任意 GPT-5.x 模型（例如 [`gpt-5.4`](https://developers.openai.com/api/docs/models/gpt-5.4)）时，SDK 会应用默认的 `modelSettings`。它会设置最适合大多数场景的参数。若要调整默认模型的推理强度，请传入您自己的 `modelSettings`：

#### 非 GPT-5 模型

如果您传入的是非 GPT-5 的模型名称，且未提供自定义 `modelSettings`，SDK 会回退为与任何模型兼容的通用 `modelSettings`。

## OpenAI 提供方配置

### OpenAI 提供方

默认的 `ModelProvider` 使用 OpenAI API 解析名称。它支持两个不同的端点：

| API | 用法 | 调用 `setOpenAIAPI()` |
| --- | --- | --- |
| Chat Completions | 标准聊天与 Function Calling | `setOpenAIAPI('chat_completions')` |
| Responses | 新的以流式传输优先的生成式 API（工具调用、灵活输出） | `setOpenAIAPI('responses')` _(默认)_ |

#### Responses WebSocket 传输

当您将 OpenAI 提供方与 Responses API 一起使用时，可以通过 WebSocket 传输发送请求，而不是使用默认的 HTTP 传输。

您可以通过 `setOpenAIResponsesTransport('websocket')` 全局启用它，也可以通过 `new OpenAIProvider({ useResponses: true, useResponsesWebSocket: true })` 为单个提供方启用它。

仅为了使用 WebSocket 传输，您不需要 `withResponsesWebSocketSession(...)` 或自定义 `OpenAIProvider`。如果每次运行/请求都重新连接是可以接受的，那么在启用 `setOpenAIResponsesTransport('websocket')` 后，您现有的 `run()` / `Runner.run()` 用法仍可继续工作。

传输方式的选择遵循模型解析流程：

- `setOpenAIResponsesTransport('websocket')` 只会影响那些之后通过 OpenAI 提供方、并在使用 Responses API 时解析的字符串模型名称。
- 如果您向 `Agent` 或 `Runner` 传入一个具体的 `Model` 实例，则会原样使用该实例。`OpenAIResponsesWSModel` 保持使用 WebSocket，`OpenAIResponsesModel` 保持使用 HTTP，`OpenAIChatCompletionsModel` 保持使用 Chat Completions。
- 如果您提供了自己的 `modelProvider`，则该提供方会控制模型解析。此时应在其中启用 WebSocket，而不是依赖全局设置器。
- 如果您通过代理、网关或其他兼容 OpenAI 的端点进行路由，则目标端必须支持 WebSocket `/responses` 端点。您也可能需要显式设置 `websocketBaseURL`。

只有在您希望优化连接复用，并更显式地管理 websocket 提供方生命周期时，才需要使用 `withResponsesWebSocketSession(...)` 或自定义 `OpenAIProvider` / `Runner`：

- `withResponsesWebSocketSession(...)`：便捷的作用域生命周期管理，回调结束后自动清理。
- 自定义 `OpenAIProvider` / `Runner`：在您自己的应用架构中显式控制生命周期（包括关闭时的清理）。

尽管名字中带有 session，`withResponsesWebSocketSession(...)` 实际上是一个传输生命周期辅助工具，与[会话](/openai-agents-js/zh/guides/sessions)指南中描述的记忆 `Session` 接口无关。

如果您使用 websocket 代理或网关，请在 `OpenAIProvider` 上配置 `websocketBaseURL`，或设置 `OPENAI_WEBSOCKET_BASE_URL`。

如果您自行实例化 `OpenAIProvider`，请记住：由 websocket 支持的 Responses 模型包装器默认会被缓存，以便复用连接。请在关闭时调用 `await provider.close()` 释放这些缓存连接。`withResponsesWebSocketSession(...)` 很大程度上就是为了帮您管理这一生命周期：它会创建启用 websocket 的 provider 和 runner，将它们传给您的回调，并在结束后始终关闭 provider。对临时 provider 使用 `providerOptions`，对回调作用域内的 runner 默认值使用 `runnerConfig`。

完整的流式传输 + HITL 示例，以及如何使用 Responses WebSocket 传输，请参阅 [`examples/basic/stream-ws.ts`](https://github.com/openai/openai-agents-js/tree/main/examples/basic/stream-ws.ts)。

#### 仅限 Responses 的延迟工具加载

`toolSearchTool()`、`toolNamespace()`，以及设置了 `deferLoading: true` 的函数工具或远程 MCP 服务器工具，都要求使用 OpenAI Responses API。Chat Completions 提供方会拒绝命名空间或延迟加载的函数工具，而 AI SDK 适配器也不支持延迟的 Responses 工具加载流程。当您需要工具搜索时，请直接使用 Responses 模型。

工具搜索仅在 GPT-5.4 及更高版本、且在 Responses API 中支持该功能的模型发布版本上可用。

当一次运行包含延迟工具时，请将 `toolSearchTool()` 添加到同一个智能体中，并将 `modelSettings.toolChoice` 保持为 `'auto'`。SDK 不允许您按名称强制使用内置的 `tool_search` 工具或某个延迟函数工具，因为模型需要自行决定何时加载这些定义。完整配置请参阅[工具](/openai-agents-js/zh/guides/tools#deferred-tool-loading-with-tool-search)指南和官方 [OpenAI tool search guide](https://developers.openai.com/api/docs/guides/tools-tool-search)。

## 模型行为与提示

### ModelSettings

`ModelSettings` 映射 OpenAI 参数，但与具体提供方无关。

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `temperature` | `number` | 创造性与确定性之间的权衡。 |
| `topP` | `number` | 核采样。 |
| `frequencyPenalty` | `number` | 惩罚重复 token。 |
| `presencePenalty` | `number` | 鼓励生成新 token。 |
| `toolChoice` | `'auto' \| 'required' \| 'none' \| string` | 参见[强制使用工具](/openai-agents-js/zh/guides/agents#forcing-tool-use)。在 OpenAI Responses 中，`toolChoice: 'computer'` 会在可用时强制使用 GA 内置计算机工具。 |
| `parallelToolCalls` | `boolean` | 在支持时允许并行函数调用。 |
| `truncation` | `'auto' \| 'disabled'` | Token 截断策略。 |
| `maxTokens` | `number` | 响应中的最大 token 数。 |
| `store` | `boolean` | 持久化响应，以便用于检索 / RAG 工作流。 |
| `promptCacheRetention` | `'in-memory' \| '24h' \| null` | 在支持时控制提供方的提示缓存保留策略。 |
| `contextManagement` | `ModelSettingsContextManagement` | 控制提供方的上下文管理，例如服务器端压缩。 |
| `reasoning.effort` | `'none' \| 'minimal' \| 'low' \| 'medium' \| 'high' \| 'xhigh'` | gpt-5.x 模型的推理强度。 |
| `reasoning.summary` | `'auto' \| 'concise' \| 'detailed'` | 控制模型返回多少推理摘要。 |
| `text.verbosity` | `'low' \| 'medium' \| 'high'` | gpt-5.x 等模型的文本详细程度。 |
| `providerData` | `Record<string, any>` | 传递给底层模型的提供方特定透传选项。 |
| `retry` | `ModelRetrySettings` | 仅运行时启用的重试配置。参见[模型重试](#model-retries)。 |

您可以在任一层级附加这些设置：

### 模型重试

重试仅在运行时生效，且为可选启用。除非您配置了 `modelSettings.retry` 且策略返回重试决定，否则 SDK 不会重试模型请求。

<Code lang="typescript" code={retryExample} title="启用模型重试" />

`ModelRetrySettings` 有三个字段：

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `maxRetries` | `number` | 初始请求之后允许的重试次数。 |
| `backoff` | `{ initialDelayMs?, maxDelayMs?, multiplier?, jitter? }` | 当策略决定重试但未返回 `delayMs` 时使用的默认延迟策略。 |
| `policy` | `RetryPolicy` | 决定是否重试的回调。该函数仅在运行时存在，不会被序列化到持久化的运行状态中。 |

重试策略会收到一个 `RetryPolicyContext`，其中包含：

- `attempt` 和 `maxRetries`，便于您根据尝试次数做决策。
- `stream`，便于区分流式和非流式行为。
- `error`，用于原始检查。
- `normalized` 信息，例如 `statusCode`、`retryAfterMs`、`errorCode`、`isNetworkError` 和 `isAbort`。
- 当底层模型/提供方可提供重试建议时的 `providerAdvice`。

策略可以返回以下之一：

- `true` / `false`，表示简单的重试决定。
- `{ retry, delayMs?, reason? }`，当您想覆盖延迟或附加用于日志记录的诊断原因时使用。

SDK 在 `retryPolicies` 上导出了现成的辅助方法：

| 辅助方法 | 行为 |
| --- | --- |
| `retryPolicies.never()` | 始终不重试。 |
| `retryPolicies.providerSuggested()` | 在可用时遵循提供方的重试建议。 |
| `retryPolicies.networkError()` | 匹配临时性的传输/连接失败。 |
| `retryPolicies.httpStatus([..])` | 匹配指定的 HTTP 状态码。 |
| `retryPolicies.retryAfter()` | 仅当存在 retry-after 提示时才重试，并使用该延迟。 |
| `retryPolicies.any(...)` | 任一嵌套策略启用重试时即重试。 |
| `retryPolicies.all(...)` | 仅当所有嵌套策略都启用重试时才重试。 |

组合策略时，`providerSuggested()` 是最安全的首选构件，因为当提供方能够区分否决与可安全重放批准时，它会保留这些信息。

#### 安全边界

某些失败永远不会被自动重试：

- Abort 错误。
- 在已经发出任何可见事件或原始模型事件之后的流式运行。
- 提供方建议将重放标记为不安全的情况。

使用 `previousResponseId` 或 `conversationId` 的有状态后续请求也会被更保守地处理。对于这些请求，仅使用诸如 `networkError()` 或 `httpStatus([500])` 之类的非提供方谓词还不够。重试策略必须包含来自提供方的重放安全批准，通常通过 `retryPolicies.providerSuggested()` 实现。

#### Runner 与 agent 的合并行为

在 runner 级和 agent 级 `modelSettings` 之间，`retry` 会进行深度合并：

- Agent 可以只覆盖 `retry.maxRetries`，同时继承 runner 的 `policy`。
- Agent 可以只覆盖 `retry.backoff` 的一部分，并保留 runner 中同级的其他 backoff 字段。
- 如果您需要移除继承的 `policy` 或 `backoff`，请显式将该字段设置为 `undefined`。

更完整的带日志示例，请参阅 [`examples/basic/retry.ts`](https://github.com/openai/openai-agents-js/tree/main/examples/basic/retry.ts) 和 [`examples/ai-sdk/retry.ts`](https://github.com/openai/openai-agents-js/tree/main/examples/ai-sdk/retry.ts)。

### Prompt

智能体可以通过 `prompt` 参数进行配置，用于指定应使用的服务器端存储 prompt 配置，以控制 Agent 的行为。目前，该选项仅在使用 OpenAI [Responses API](https://platform.openai.com/docs/api-reference/responses) 时受支持。

`prompt` 可以是静态对象，也可以是运行时返回该对象的函数。关于回调形式，请参见[动态提示](/openai-agents-js/zh/guides/agents#dynamic-prompts)。

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `promptId` | `string` | prompt 的唯一标识符。 |
| `version` | `string` | 您希望使用的 prompt 版本。 |
| `variables` | `object` | 用于替换到 prompt 中的键值对变量。值可以是字符串，也可以是文本、图像或文件等内容输入类型。 |

<Code lang="typescript" code={promptIdExample} title="带有 prompt 的 Agent" />

任何额外的 agent 配置（如工具或 instructions）都会覆盖您在已存储 prompt 中可能配置的值。

当已存储的 prompt 已经定义了模型时，除非您显式覆盖，否则 SDK 不会发送 agent 的默认模型。这一点对于 `computerTool()` 很重要：为兼容起见，由 prompt 管理的运行默认保留旧版预览的线缆格式。若要在由 prompt 管理的运行中启用 GA Responses 计算机工具，请显式设置 `modelSettings.toolChoice: 'computer'`，或发送一个显式模型，例如 [`gpt-5.4`](https://developers.openai.com/api/docs/models/gpt-5.4)。有关计算机操作的更多细节，请参见[工具](/openai-agents-js/zh/guides/tools#computer-tool-specifics)。
