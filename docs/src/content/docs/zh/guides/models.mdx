## 模型选择

### 默认模型

当您在初始化 `Agent` 时未指定模型，将使用默认模型。当前默认值为 [`gpt-4.1`](https://developers.openai.com/api/docs/models/gpt-4.1)，以兼顾兼容性和低延迟。如果您有权限，我们建议将智能体设置为 [`gpt-5.4`](https://developers.openai.com/api/docs/models/gpt-5.4) 以获得更高质量，同时显式设置 `modelSettings`。

如果您想切换到 [`gpt-5.4`](https://developers.openai.com/api/docs/models/gpt-5.4) 等其他模型，有两种方式可以配置智能体。

第一种，如果您希望所有未设置自定义模型的智能体统一使用某个特定模型，请在运行智能体前设置 `OPENAI_DEFAULT_MODEL` 环境变量。

```bash
export OPENAI_DEFAULT_MODEL=gpt-5.4
node my-awesome-agent.js
```

第二种，您可以为 `Runner` 实例设置默认模型。如果智能体未设置模型，将使用该 `Runner` 的默认模型。

#### GPT-5.x 模型

当您以这种方式使用任意 GPT-5.x 模型（例如 [`gpt-5.4`](https://developers.openai.com/api/docs/models/gpt-5.4)）时，SDK 会应用默认 `modelSettings`。它会设置在大多数场景下表现最佳的值。若要调整默认模型的推理强度，请传入您自己的 `modelSettings`：

如果您传入非 GPT-5 的模型名称且未提供自定义 `modelSettings`，SDK 会回退到与任意模型兼容的通用 `modelSettings`。

## OpenAI 提供方配置

### OpenAI 提供方

默认 `ModelProvider` 使用 OpenAI API 解析名称。它支持两个不同端点：

| API | 用法 | 调用 `setOpenAIAPI()` |
| --- | --- | --- |
| Chat Completions | 标准聊天与函数调用 | `setOpenAIAPI('chat_completions')` |
| Responses | 新一代以流式传输优先的生成式 API（工具调用、灵活输出） | `setOpenAIAPI('responses')` _（默认）_ |

#### Responses WebSocket 传输

当您在 OpenAI 提供方中使用 Responses API 时，可通过 WebSocket 传输发送请求，而不是默认的 HTTP 传输。

可通过 `setOpenAIResponsesTransport('websocket')` 全局启用，或在单个提供方中通过 `new OpenAIProvider({ useResponses: true, useResponsesWebSocket: true })` 启用。

若仅需使用 WebSocket 传输，您不需要 `withResponsesWebSocketSession(...)` 或自定义 `OpenAIProvider`。如果每次运行/请求都重新连接是可接受的，启用 `setOpenAIResponsesTransport('websocket')` 后，现有 `run()` / `Runner.run()` 用法仍可继续工作。

传输方式选择遵循模型解析规则：

- `setOpenAIResponsesTransport('websocket')` 只会影响后续通过 OpenAI 提供方并使用 Responses API 解析的字符串模型名。
- 若您向 `Agent` 或 `Runner` 传入具体的 `Model` 实例，则该实例会原样使用。`OpenAIResponsesWSModel` 继续使用 WebSocket，`OpenAIResponsesModel` 继续使用 HTTP，`OpenAIChatCompletionsModel` 继续使用 Chat Completions。
- 若您提供了自己的 `modelProvider`，则由该提供方控制模型解析。请在其中启用 WebSocket，而不是依赖全局设置器。
- 若您通过代理、网关或其他 OpenAI 兼容端点路由，请确保目标支持 WebSocket `/responses` 端点。您也可能需要显式设置 `websocketBaseURL`。

仅当您希望优化连接复用并更显式地管理 websocket 提供方生命周期时，才使用 `withResponsesWebSocketSession(...)` 或自定义 `OpenAIProvider` / `Runner`：

- `withResponsesWebSocketSession(...)`：便捷的作用域生命周期管理，回调结束后自动清理。
- 自定义 `OpenAIProvider` / `Runner`：在您自己的应用架构中进行显式生命周期控制（包括关闭时清理）。

尽管名称里有 Session，`withResponsesWebSocketSession(...)` 是传输生命周期辅助工具，与[会话](/openai-agents-js/zh/guides/sessions)中描述的内存 `Session` 接口无关。

如果您使用 websocket 代理或网关，请在 `OpenAIProvider` 上配置 `websocketBaseURL`，或设置 `OPENAI_WEBSOCKET_BASE_URL`。

如果您自行实例化 `OpenAIProvider`，请注意：基于 websocket 的 Responses 模型包装器默认会被缓存以复用连接。请在关闭时调用 `await provider.close()` 释放这些缓存连接。`withResponsesWebSocketSession(...)` 很大程度上就是为您管理该生命周期：它会创建启用 websocket 的 provider 和 runner，传给您的回调，并在之后始终关闭 provider。临时 provider 用 `providerOptions` 配置，回调作用域内的 runner 默认值用 `runnerConfig` 配置。

完整的流式传输 + 人工干预（HITL）示例，请参见 [`examples/basic/stream-ws.ts`](https://github.com/openai/openai-agents-js/tree/main/examples/basic/stream-ws.ts)，其使用了 Responses WebSocket 传输。

#### 仅 Responses 的延迟工具加载

`toolSearchTool()`、`toolNamespace()`，以及设置了 `deferLoading: true` 的函数工具或托管 MCP 工具，都要求使用 OpenAI Responses API。Chat Completions 提供方会拒绝命名空间或延迟加载的函数工具，AI SDK 适配器也不支持延迟 Responses 工具加载流程。需要工具搜索时，请直接使用 Responses 模型。

工具搜索仅在 GPT-5.4 及更高且在 Responses API 中支持该能力的模型版本上可用。

当一次运行包含延迟工具时，请在同一智能体上添加 `toolSearchTool()`，并将 `modelSettings.toolChoice` 保持为 `'auto'`。SDK 不允许您通过名称强制内置 `tool_search` 工具或延迟函数工具，因为模型需要自行决定何时加载这些定义。完整配置请参阅[工具](/openai-agents-js/zh/guides/tools#deferred-tool-loading-with-tool-search)以及官方 [OpenAI tool search guide](https://developers.openai.com/api/docs/guides/tools-tool-search)。

## 模型行为与提示

### ModelSettings

`ModelSettings` 映射 OpenAI 参数，但与提供方无关。

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `temperature` | `number` | 创造性与确定性的平衡。 |
| `topP` | `number` | 核采样。 |
| `frequencyPenalty` | `number` | 惩罚重复 token。 |
| `presencePenalty` | `number` | 鼓励生成新 token。 |
| `toolChoice` | `'auto' \| 'required' \| 'none' \| string` | 参见[强制使用工具](/openai-agents-js/zh/guides/agents#forcing-tool-use)。 |
| `parallelToolCalls` | `boolean` | 在支持时允许并行函数调用。 |
| `truncation` | `'auto' \| 'disabled'` | token 截断策略。 |
| `maxTokens` | `number` | 响应中的最大 token 数。 |
| `store` | `boolean` | 持久化响应以用于检索 / RAG 工作流。 |
| `promptCacheRetention` | `'in-memory' \| '24h' \| null` | 在支持时控制提供方提示缓存保留策略。 |
| `reasoning.effort` | `'none' \| 'minimal' \| 'low' \| 'medium' \| 'high' \| 'xhigh'` | gpt-5.x 模型的推理强度。 |
| `reasoning.summary` | `'auto' \| 'concise' \| 'detailed'` | 控制模型返回多少推理摘要。 |
| `text.verbosity` | `'low' \| 'medium' \| 'high'` | gpt-5.x 等模型的文本详略程度。 |
| `providerData` | `Record<string, any>` | 透传给底层模型的提供方特定选项。 |

可在任一层级附加设置：

### 提示

智能体可通过 `prompt` 参数进行配置，表示应使用服务器端存储的提示配置来控制智能体行为。目前该选项仅在使用 OpenAI [Responses API](https://platform.openai.com/docs/api-reference/responses) 时受支持。

`prompt` 可以是静态对象，也可以是运行时返回对象的函数。关于回调形状，请参见[动态提示](/openai-agents-js/zh/guides/agents#dynamic-prompts)。

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `promptId` | `string` | 提示的唯一标识符。 |
| `version` | `string` | 要使用的提示版本。 |
| `variables` | `object` | 用于替换到提示中的键值对变量。值可以是字符串或文本、图像、文件等内容输入类型。 |

任何额外的智能体配置（如工具或 instructions）都会覆盖您在已存储提示中配置的值。
