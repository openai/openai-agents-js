現時点での主なユースケースは、機密性の高いツール実行に対する承認要求です。

### フロー

1. エージェントがツール（1 つまたは複数）を呼び出す判断をすると、 `needsApproval` を評価してそのツールに承認が必要かどうかを確認します。
2. 承認が必要な場合、エージェントは承認がすでに付与済みか拒否済みかを確認します。
   - 承認も拒否もされていない場合、ツールは「このツール呼び出しは実行できない」という固定メッセージをエージェントに返します。
   - 承認 / 拒否が未処理の場合、ツール承認リクエストがトリガーされます。
3. エージェントはすべてのツール承認リクエストを収集し、実行を中断します。
4. 中断がある場合、 [エージェントの実行結果](/openai-agents-js/ja/guides/results) には保留中のステップを説明する `interruptions` 配列が含まれます。ツール呼び出しに確認が必要なときは、 `type: "tool_approval_item"` を持つ `ToolApprovalItem` が現れます。
5. `result.state.approve(interruption)` または `result.state.reject(interruption)` を呼び出して、ツール呼び出しを承認または拒否できます。同じツールをその後の実行中ずっと承認または拒否のままにするには、 `{ alwaysApprove: true }` または `{ alwaysReject: true }` を渡します。
6. すべての中断を処理したあと、 `result.state` を `runner.run(agent, state)` に戻して渡すことで実行を再開できます。ここで `agent` は全体の実行を開始した元のエージェントです。
7. フローはステップ 1 から再び始まります。

## 長時間の承認待ちへの対応

人間の介入（HITL） フローは、サーバーを起動し続けなくても長時間中断できるように設計されています。リクエストをいったん終了して後で続ける必要がある場合は、状態をシリアライズして後で再開できます。

状態は `result.state.toString()` （または `JSON.stringify(result.state)` ）でシリアライズでき、後で `RunState.fromString(agent, serializedState)` にシリアライズ済み状態を渡して再開できます。ここで `agent` は全体の実行を開始したエージェントのインスタンスです。

再開したプロセスで新しいコンテキストオブジェクトを注入する必要がある場合は、代わりに `RunState.fromStringWithContext(agent, serializedState, context, { contextStrategy })` を使用します。

- `contextStrategy: 'merge'` （デフォルト）は、シリアライズ済みの承認状態を保持しつつ、提供された `RunContext` にマージします
- `contextStrategy: 'replace'` は、提供された `RunContext` をそのまま使って実行を再構築します

デフォルトでは、誤って秘密情報を永続化しないよう、シリアライズされた状態からトレーシング API キーは除外されます。状態と一緒にトレーシング認証情報を意図的に移動する必要がある場合にのみ、 `result.state.toString({ includeTracingApiKey: true })` を渡してください。

これにより、シリアライズした状態をデータベースに保存したり、リクエストと一緒に保持したりできます。

### 保留タスクのバージョニング

<Aside>
  これは主に、エージェントを変更しながらシリアライズ済み状態を長期間保存しようとしている場合に当てはまります。
</Aside>

承認リクエストに長い時間がかかり、エージェント定義に意味のあるバージョン管理をしたい場合、または Agents SDK のバージョンを上げる場合は、現時点では package alias を使って 2 つの Agents SDK バージョンを並行インストールし、独自の分岐ロジックを実装することを推奨します。

実運用では、独自コードにバージョン番号を付与してシリアライズ済み状態と一緒に保存し、デシリアライズ時に正しいバージョンのコードへ誘導する形になります。
