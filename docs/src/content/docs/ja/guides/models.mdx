すべての Agent は最終的に LLM を呼び出します。 SDK は、モデルを 2 つの軽量なインターフェースの背後に抽象化しています。

- [`Model`](/openai-agents-js/openai/agents/interfaces/model) – 特定の API に対して _1 回の_ リクエストを行う方法を認識しています

## モデルの選択

### デフォルトモデル

`Agent` の初期化時にモデルを指定しない場合は、デフォルトモデルが使用されます。 現在のデフォルトは [`gpt-5.4-mini`](https://developers.openai.com/api/docs/models/gpt-5.4-mini) で、`reasoning.effort: "none"` および `text.verbosity: "low"` が設定されており、品質とレイテンシのバランスを取っています。

[`gpt-5.5`](https://developers.openai.com/api/docs/models/gpt-5.5) のような別のモデルに切り替えたい場合、エージェントを設定する方法は 2 つあります。

まず、カスタムモデルを設定していないすべてのエージェントで一貫して特定のモデルを使いたい場合は、エージェントを実行する前に `OPENAI_DEFAULT_MODEL` 環境変数を設定します。

```bash
export OPENAI_DEFAULT_MODEL=gpt-5.5
node my-awesome-agent.js
```

次に、`Runner` インスタンスにデフォルトモデルを設定できます。 エージェントにモデルを設定しない場合は、この `Runner` のデフォルトモデルが使用されます。

カスタム `modelSettings` なしで GPT-5 以外のモデル名を渡した場合、SDK は任意のモデルと互換性のある汎用的な `modelSettings` に戻します。

## OpenAI provider の設定

### OpenAI provider

デフォルトの `ModelProvider` は、OpenAI API を使って名前を解決します。 2 つの異なるエンドポイントをサポートしています。

| API | 使用用途 | `setOpenAIAPI()` の呼び出し |
| --- | --- | --- |
| Chat Completions | 標準的なチャットと function calls | `setOpenAIAPI('chat_completions')` |
| Responses | 新しい streaming-first の生成 API（ツール呼び出し、柔軟な出力） | `setOpenAIAPI('responses')` _(デフォルト)_ |

カスタムのネットワーク設定が必要な場合は、`setDefaultOpenAIClient(client)` を使って独自の `OpenAI` クライアントを組み込むこともできます。

#### Responses WebSocket トランスポート

Responses API とともに OpenAI provider を使う場合、デフォルトの HTTP トランスポートの代わりに WebSocket トランスポートでリクエストを送信できます。

グローバルに有効化するには `setOpenAIResponsesTransport('websocket')` を使うか、provider ごとに `new OpenAIProvider({ useResponses: true, useResponsesWebSocket: true })` で有効化します。

WebSocket トランスポートを使うだけであれば、`withResponsesWebSocketSession(...)` やカスタム `OpenAIProvider` は不要です。 実行やリクエストごとの再接続を許容できるなら、`setOpenAIResponsesTransport('websocket')` を有効化したあとも、既存の `run()` / `Runner.run()` の使い方をそのまま継続できます。

トランスポートの選択は、モデル解決に従います。

- `setOpenAIResponsesTransport('websocket')` が影響するのは、Responses API を使う際に後から OpenAI provider を通じて解決される文字列のモデル名だけです
- 具体的な `Model` インスタンスを `Agent` または `Runner` に渡した場合、そのインスタンスがそのまま使われます。 `OpenAIResponsesWSModel` は WebSocket のまま、`OpenAIResponsesModel` は HTTP のまま、`OpenAIChatCompletionsModel` は Chat Completions のままです
- 独自の `modelProvider` を指定した場合、モデル解決はその provider が制御します。 グローバル setter に頼るのではなく、そちらで WebSocket を有効化してください
- プロキシ、ゲートウェイ、またはその他の OpenAI 互換エンドポイントを経由する場合、接続先が WebSocket の `/responses` エンドポイントをサポートしている必要があります。 `websocketBaseURL` を明示的に設定する必要がある場合もあります

`withResponsesWebSocketSession(...)` またはカスタム `OpenAIProvider` / `Runner` を使うのは、接続再利用を最適化し、websocket provider のライフサイクルをより明示的に管理したい場合だけです。

- `withResponsesWebSocketSession(...)`: コールバック後に自動クリーンアップされる、便利なスコープ付きライフサイクル
- カスタム `OpenAIProvider` / `Runner`: 独自のアプリ構成での明示的なライフサイクル制御（シャットダウン時のクリーンアップを含む）

名前に反して、`withResponsesWebSocketSession(...)` はトランスポートのライフサイクルを補助するヘルパーであり、[セッションガイド](/openai-agents-js/ja/guides/sessions) で説明しているメモリ `Session` インターフェースとは無関係です。

websocket のプロキシやゲートウェイを使う場合は、`OpenAIProvider` で `websocketBaseURL` を設定するか、`OPENAI_WEBSOCKET_BASE_URL` を設定してください。

`OpenAIProvider` を自分でインスタンス化する場合は、websocket ベースの Responses モデルラッパーが接続再利用のためにデフォルトでキャッシュされることに注意してください。 シャットダウン時には、キャッシュされた接続を解放するために `await provider.close()` を呼び出します。 `withResponsesWebSocketSession(...)` は主にこのライフサイクル管理のために存在しており、websocket を有効にした provider と runner を作成してコールバックに渡し、その後に必ず provider を閉じます。 一時的な provider には `providerOptions` を、コールバックにスコープされた runner のデフォルトには `runnerConfig` を使ってください。

Responses WebSocket トランスポートを使った、ストリーミング + HITL の完全な例については [`examples/basic/stream-ws.ts`](https://github.com/openai/openai-agents-js/tree/main/examples/basic/stream-ws.ts) を参照してください。

#### Responses 専用の遅延ツール読み込み

`toolSearchTool()`, `toolNamespace()`, および `deferLoading: true` を設定した関数ツールまたは Hosted MCP tools は、OpenAI Responses API を必要とします。 Chat Completions provider は、名前空間付きまたは遅延された関数ツールを拒否し、AI SDK アダプターは遅延された Responses ツール読み込みフローをサポートしていません。 ツール検索が必要な場合は、Responses モデルを直接使用してください。

ツール検索がサポートされるのは、Responses API でこれをサポートする GPT-5.4 以降のモデルリリースのみです。

実行に遅延ツールが含まれる場合は、同じエージェントに `toolSearchTool()` を追加し、`modelSettings.toolChoice` は `'auto'` のままにしてください。 モデルがそれらの定義をいつ読み込むかを判断する必要があるため、SDK では組み込みの `tool_search` ツールや遅延関数ツールを名前で強制することはできません。 完全な設定については、[ツールガイド](/openai-agents-js/ja/guides/tools#deferred-tool-loading-with-tool-search) と公式の [OpenAI tool search guide](https://developers.openai.com/api/docs/guides/tools-tool-search) を参照してください。

## モデルの挙動とプロンプト

### ModelSettings

`ModelSettings` は OpenAI のパラメーターを反映しますが、provider 非依存です。

| Field | Type | Notes |
| --- | --- | --- |
| `temperature` | `number` | 創造性と決定性のバランス |
| `topP` | `number` | Nucleus sampling |
| `frequencyPenalty` | `number` | 繰り返されるトークンにペナルティを与える |
| `presencePenalty` | `number` | 新しいトークンを促す |
| `toolChoice` | `'auto' \| 'required' \| 'none' \| string` | [ツール使用の強制](/openai-agents-js/ja/guides/agents#forcing-tool-use) を参照してください。 OpenAI Responses では、`toolChoice: 'computer'` により、利用可能な場合は GA の組み込み computer tool を強制します |
| `parallelToolCalls` | `boolean` | サポートされている場合に並列 function calls を許可する |
| `truncation` | `'auto' \| 'disabled'` | トークン切り詰め戦略 |
| `maxTokens` | `number` | レスポンス内の最大トークン数 |
| `store` | `boolean` | 取得 / RAG ワークフロー用にレスポンスを永続化する |
| `promptCacheRetention` | `'in-memory' \| '24h' \| null` | サポートされている場合に provider の prompt cache 保持を制御する |
| `contextManagement` | `ModelSettingsContextManagement` | サーバー側 compaction などの provider のコンテキスト管理を制御する |
| `reasoning.effort` | `'none' \| 'minimal' \| 'low' \| 'medium' \| 'high' \| 'xhigh'` | gpt-5.x モデルの reasoning effort |
| `reasoning.summary` | `'auto' \| 'concise' \| 'detailed'` | モデルが返す reasoning summary の量を制御する |
| `text.verbosity` | `'low' \| 'medium' \| 'high'` | gpt-5.x などのテキスト詳細度 |
| `providerData` | `Record<string, any>` | 基盤モデルに転送される provider 固有のパススルーオプション |
| `retry` | `ModelRetrySettings` | 実行時のみのオプトイン型リトライ設定。 [モデルのリトライ](#model-retries) を参照してください |

設定はどちらのレベルにも付与できます。

### モデルのリトライ

リトライは実行時のみで、明示的に有効化する必要があります。 `modelSettings.retry` を設定し、かつポリシーがリトライを返さない限り、SDK はモデルリクエストをリトライしません。

<Code lang="typescript" code={retryExample} title="モデルのリトライを有効化" />

`ModelRetrySettings` には 3 つのフィールドがあります。

| Field | Type | Notes |
| --- | --- | --- |
| `maxRetries` | `number` | 初回リクエスト後に許可されるリトライ試行回数 |
| `backoff` | `{ initialDelayMs?, maxDelayMs?, multiplier?, jitter? }` | ポリシーが `delayMs` を返さずにリトライした場合のデフォルト遅延戦略。 `backoff.maxDelayMs` が制限するのは、この計算済みバックオフ遅延のみであり、ポリシーが返した明示的な `delayMs` 値や retry-after ヒントは制限しません |
| `policy` | `RetryPolicy` | リトライするかどうかを判断するコールバック。 この関数は実行時のみであり、永続化された実行状態にはシリアライズされません |

リトライポリシーは、次を含む `RetryPolicyContext` を受け取ります。

- `attempt` と `maxRetries`。試行回数を考慮した判断ができます
- `stream`。ストリーミング時と非ストリーミング時で分岐できます
- `error`。生の内容を確認できます
- `statusCode`, `retryAfterMs`, `errorCode`, `isNetworkError`, `isAbort` などの正規化された情報である `normalized`
- 基盤のモデル / provider がリトライ指針を提供できる場合の `providerAdvice`

ポリシーは次のいずれかを返せます。

- 単純なリトライ判断としての `true` / `false`
- 遅延を上書きしたり、ログ記録用の診断理由を付加したい場合の `{ retry, delayMs?, reason? }`

SDK は `retryPolicies` に、すぐ使えるヘルパーをエクスポートしています。

| Helper | Behavior |
| --- | --- |
| `retryPolicies.never()` | 常に無効化する |
| `retryPolicies.providerSuggested()` | 利用可能な場合は provider のリトライ指針に従う |
| `retryPolicies.networkError()` | 一時的なトランスポート / 接続障害に一致する |
| `retryPolicies.httpStatus([..])` | 指定した HTTP ステータスコードに一致する |
| `retryPolicies.retryAfter()` | retry-after ヒントが利用可能な場合にのみリトライし、そのヒントを `backoff.maxDelayMs` で制限されない明示的遅延として使う |
| `retryPolicies.any(...)` | ネストされたポリシーのいずれかが有効化した場合にリトライする |
| `retryPolicies.all(...)` | ネストされたすべてのポリシーが有効化した場合にのみリトライする |

ポリシーを組み合わせる場合、`providerSuggested()` は最初の構成要素として最も安全です。 provider がそれらを区別できる場合に、provider による拒否と replay-safe な承認を保持するためです。

#### 安全境界

一部の失敗は自動的にリトライされません。

- Abort エラー
- 何らかの可視イベントまたは生のモデルイベントがすでに出力された後のストリーミング実行
- リプレイが安全ではないと示す provider の指針

`previousResponseId` または `conversationId` を使う状態付きの後続リクエストも、より保守的に扱われます。 これらのリクエストでは、`networkError()` や `httpStatus([500])` のような provider 以外の述語だけでは不十分です。 リトライポリシーには、通常は `retryPolicies.providerSuggested()` を通じた provider からの replay-safe 承認を含める必要があります。

#### Runner と agent のマージ動作

`retry` は runner レベルと agent レベルの `modelSettings` 間で深くマージされます。

- agent は `retry.maxRetries` だけを上書きし、runner の `policy` は継承できます
- agent は `retry.backoff` の一部だけを上書きし、他の sibling backoff フィールドは runner から保持できます
- 継承した `policy` や `backoff` を削除する必要がある場合は、そのフィールドを明示的に `undefined` に設定してください

より完全なログ付きの例については、[`examples/basic/retry.ts`](https://github.com/openai/openai-agents-js/tree/main/examples/basic/retry.ts) と [`examples/ai-sdk/retry.ts`](https://github.com/openai/openai-agents-js/tree/main/examples/ai-sdk/retry.ts) を参照してください。

### プロンプト

エージェントは `prompt` パラメーターで設定でき、Agent の挙動を制御するために使用するサーバー保存済みのプロンプト設定を示します。 現在、このオプションがサポートされるのは、OpenAI の [Responses API](https://platform.openai.com/docs/api-reference/responses) を使用する場合のみです。

`prompt` は静的オブジェクトでも、実行時にそれを返す関数でもかまいません。 コールバックの形については、[動的プロンプト](/openai-agents-js/ja/guides/agents#dynamic-prompts) を参照してください。

| Field | Type | Notes |
| --- | --- | --- |
| `promptId` | `string` | プロンプトの一意識別子 |
| `version` | `string` | 使用したいプロンプトのバージョン |
| `variables` | `object` | プロンプトに代入する変数のキー / 値の組。 値には文字列、またはテキスト・画像・ファイルのような content input types を指定できます |

ツールや instructions のような追加の agent 設定は、保存済みプロンプト内で設定されている値を上書きします。

保存済みプロンプトがすでにモデルを定義している場合、明示的に上書きしない限り、SDK は agent のデフォルトモデルを送信しません。 これは `computerTool()` に関係します。プロンプト管理された実行では、互換性のためにデフォルトで従来の preview wire shape が維持されます。 プロンプト管理された実行で GA の Responses computer tool を有効にするには、`modelSettings.toolChoice: 'computer'` を明示的に設定するか、[`gpt-5.4`](https://developers.openai.com/api/docs/models/gpt-5.4) のような明示的なモデルを送信してください。 コンピュータ操作に関する周辺の詳細は [ツール](/openai-agents-js/ja/guides/tools#computer-tool-specifics) を参照してください。
