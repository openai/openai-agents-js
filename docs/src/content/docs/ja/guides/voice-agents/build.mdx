## セッション設定

### 音声処理

デフォルトの `OpenAIRealtimeWebRTC` のような一部のトランスポート層では、音声入力と出力が自動で処理されます。`OpenAIRealtimeWebSocket` のようなほかのトランスポート方式では、セッション音声を自分で処理する必要があります:

基盤トランスポートが対応している場合、`session.muted` は現在のミュート状態を示し、`session.mute(true | false)` はマイクキャプチャを切り替えます。`OpenAIRealtimeWebSocket` はミュートを実装していません: `session.muted` は `null` を返し、`session.mute()` は例外を投げます。そのため WebSocket 構成では、再度マイクを有効にすべきタイミングまで、クライアント側でキャプチャを一時停止し、`sendAudio()` の呼び出しを止めてください。

### セッション構成

[`RealtimeSession`](/openai-agents-js/openai/agents-realtime/classes/realtimesession/) を作成するときに、通常は `model` オプションと `config` オブジェクトを通じてセッション自体を構成します。`connect(...)` は任意のセッションフィールドではなく、認証情報、エンドポイント URL、SIP 通話アタッチなどの接続時の関心事に使います。

内部では、SDK はこの構成を Realtime の [`session.update`](https://developers.openai.com/api/reference/resources/realtime/client-events/#session.update) 形式に正規化します。[RealtimeSessionConfig](/openai-agents-js/openai/agents-realtime/type-aliases/realtimesessionconfig/) に対応するプロパティがない生のセッションフィールドが必要な場合は、`providerData` を使うか、`session.transport.sendEvent(...)` で生の `session.update` を送信してください。

新しい SDK 構成形式である `outputModalities`、`audio.input`、`audio.output` の使用を推奨します。`modalities`、`inputAudioFormat`、`outputAudioFormat`、`inputAudioTranscription`、`turnDetection` などの旧 SDK エイリアスも後方互換のために正規化されますが、新規コードではここで示すネストされた `audio` 構造を使ってください。

音声対音声セッションでは、一般的に `outputModalities: ['audio']` を選びます。これにより音声出力と文字起こしが得られます。テキストのみの応答が欲しい場合のみ `['text']` に切り替えてください。

新しいパラメーターで [RealtimeSessionConfig](/openai-agents-js/openai/agents-realtime/type-aliases/realtimesessionconfig/) に対応項目がない場合は、`providerData` を使えます。`providerData` に渡した内容はすべて生の `session` オブジェクトの一部として転送されます。

### 会話ライフサイクル

`RealtimeSession` は長寿命の Realtime 接続の上に構築されています。会話履歴のローカルコピーを保持し、トランスポートイベントを監視し、ツールと出力ガードレールを実行し、アクティブなエージェント構成をトランスポートと同期し続けます。

基盤 API の挙動も重要です:

- 接続成功時は `session.created` イベントで始まり、その後の構成変更で `session.updated` が生成されます
- 多くのセッションプロパティは時間経過で変更できますが、`model` は会話途中で変更不可、`voice` はセッションが音声出力を生成する前のみ変更可能、トレーシングは有効化後に Realtime API で変更できないため事前に決める必要があります
- Realtime API は現在、単一セッションを 60 分に制限しています
- 入力音声文字起こしは非同期のため、最新発話の文字起こしは応答生成開始後に到着することがあります

SDK 層では、`await session.connect()` は「会話開始に十分な準備がトランスポートで整った」を意味しますが、正確なタイミングはトランスポートにより異なります:

- デフォルトのブラウザ WebRTC トランスポートでは、データチャネルが開いた直後に SDK が初期 `session.update` を送信し、`connect()` 解決前に対応する `session.updated` イベントを待機しようとします。これは、instructions、ツール、モダリティが適用される前に音声がサーバーへ届くのを防ぐためです。この確認応答が来ない場合、`connect()` は短いタイムアウト後にフォールバックで解決します
- デフォルトのサーバー側 WebSocket トランスポートでは、ソケットオープンと初期構成送信が完了すると `connect()` が解決します。そのため対応する `session.updated` は `connect()` 解決後に到着し得ます

生イベントモデルが必要な場合は、このページとあわせて公式の [Realtime conversations guide](https://developers.openai.com/api/docs/guides/realtime-conversations/) を確認してください。

## インタラクションフロー

### ターン検出と音声アクティビティ検出

デフォルトでは Realtime セッションは組み込みの音声アクティビティ検出（ VAD ）を使い、API がユーザーの発話開始 / 終了や応答作成タイミングを判断します。SDK ではこれを `audio.input.turnDetection` で公開しています。

<Code lang="typescript" code={turnDetectionExample} />

一般的な 2 つのモード:

- `semantic_vad`: より自然なターン境界を目指し、ユーザーがまだ話し終えていないように聞こえる場合は少し長く待機します
- `server_vad`: しきい値ベースで、`threshold`、`prefixPaddingMs`、`silenceDurationMs`、`idleTimeoutMs` などの設定を公開します

ターン境界を自分で管理したい場合は、`audio.input.turnDetection` を `null` に設定してください。基盤挙動の詳細は、公式の [voice activity detection guide](https://developers.openai.com/api/docs/guides/realtime-vad/) および [Realtime conversations guide](https://developers.openai.com/api/docs/guides/realtime-conversations/#voice-activity-detection) を参照してください。

VAD が有効なとき、エージェントの発話に重ねて話すと現在の応答を中断できます。WebSocket トランスポートでは、SDK は `input_audio_buffer.speech_started` を監視し、ユーザーが実際に聞いた分までアシスタント音声を切り詰め、`audio_interrupted` イベントを発行します。このイベントは WebSocket 構成で再生を自前管理する場合に特に有用です。

よくあるケースは 2 つです:

1. `audio.input.turnDetection = null` で VAD を完全に無効化した場合、音声ターンのコミットと `response.create` 送信はあなたの責任です

## エージェント機能

### ハンドオフ

通常のエージェントと同様に、ハンドオフを使ってエージェントを複数に分割し、それらをオーケストレーションすることで、性能向上や問題スコープの明確化が可能です。

<Code lang="typescript" code={multiAgentsExample} />

通常のエージェントと異なり、Realtime Agents ではハンドオフの挙動が少し異なります。ハンドオフ実行時、進行中セッションは新しいエージェント構成で更新されます。このため、新しいエージェントは進行中会話履歴へ自動アクセスでき、入力フィルターは現在適用されません。

セッションはライブのままなので、ハンドオフ中にそのセッションのモデルは変わりません。音声変更は基盤 Realtime API ルールに従い、セッションが音声出力を生成する前にのみ機能します。Realtime ハンドオフは主に同一セッション上で `RealtimeAgent` 構成を切り替える用途です。異なるモデル（例: `gpt-5.2` のような推論モデル）を使う必要がある場合や、非 realtime バックエンドエージェントへ委譲する場合は、[delegation through tools](#delegation-through-tools) を使ってください。

### ツール

通常のエージェントと同様に、Realtime Agents はアクション実行のためにツールを呼び出せます。Realtime は **function tools**（ローカル実行）と **hosted MCP tools**（Realtime API によるリモート実行）をサポートします。関数ツールは通常エージェントと同じ `tool()` ヘルパーで定義できます。

<Code lang="typescript" code={defineToolExample} />

#### 関数ツール

関数ツールは `RealtimeSession` と同じ環境で実行されます。つまり、セッションをブラウザで動かしている場合、ツールもブラウザで実行されます。機密性の高い処理が必要な場合は、ツール内からバックエンドを呼び出し、権限が必要な処理はサーバー側で実行してください。

これにより、ブラウザ側ツールをサーバー側ロジックへの薄いバックチャネルとして使えます。例えば [`examples/realtime-next`](https://github.com/openai/openai-agents-js/tree/main/examples/realtime-next) では、ブラウザで `refundBackchannel` ツールを定義し、リクエストと現在の会話履歴をサーバー側の `handleRefundRequest(...)` へ転送します。サーバー側では別の `Runner` が異なるエージェントやモデルを使って返金可否を評価し、結果を音声セッションに返します。

#### Hosted MCP ツール

Hosted MCP ツールは `hostedMcpTool` で構成でき、リモートで実行されます。MCP ツール利用可能性が変わると、セッションは `mcp_tools_changed` を発行します。MCP ツール呼び出し完了後にセッションがモデル応答を自動トリガーしないようにするには、`automaticallyTriggerResponseForMcpToolCalls: false` を設定してください。

現在のフィルタ済み MCP ツール一覧は `session.availableMcpTools` でも取得できます。このプロパティと `mcp_tools_changed` イベントはどちらも、アクティブエージェントで有効な hosted MCP サーバーのうち、エージェント構成の `allowed_tools` フィルター適用後のみを反映します。

Hosted MCP の設定は、安全なサーバー選択、ヘッダー、承認を接続前構成として扱うと最も整理しやすくなります。`RealtimeSession.connect()` がトランスポートを開く前に、SDK はアクティブエージェントの Hosted MCP ツール定義を解決し、サポートされる MCP フィールドを初期セッション構成に含めて Realtime API へ送信します。

このタイミングはブラウザ WebRTC アプリで特に重要です。エフェメラルクライアントシークレットは常にサーバーで発行されるため、秘匿すべき Hosted MCP 認証情報やカスタム `headers` は、初期 `session` ペイロードの一部としてサーバー側 `POST /v1/realtime/client_secrets` リクエストに含める必要があります。長期認証情報をブラウザコードに置き、`connect()` 開始後に追加する前提にはしないでください。

Realtime API レベルでは、その後の `session.update` 呼び出しでツールやほかの可変セッションフィールドを変更できますし、SDK 自体もアクティブエージェント変更時に `session.update` を送ります。ただしブラウザアプリでは、安全な Hosted MCP 初期化をサーバー側の接続前関心事として扱い、ブラウザ側 `RealtimeSession` 構成をサーバーが発行した内容と整合させてください。

#### バックグラウンド結果

ツール実行中、エージェントはユーザーからの新規リクエストを処理できません。体験改善の方法の 1 つは、ツール実行前にエージェントが告知する、または時間稼ぎ用の特定フレーズを話すよう指示することです。

関数ツールが直ちに別のモデル応答をトリガーせずに完了すべき場合、`@openai/agents/realtime` の `backgroundResult(output)` を返してください。これにより、ツール出力をセッションへ返しつつ、応答トリガーはあなたの制御下に置けます。

#### タイムアウト

関数ツールのタイムアウトオプション（`timeoutMs`、`timeoutBehavior`、`timeoutErrorFunction`）は Realtime セッションでも同様に動作します。デフォルトの `error_as_result` では、タイムアウトメッセージがツール出力として送られます。`raise_exception` では、セッションは [`ToolTimeoutError`](/openai-agents-js/openai/agents-core/classes/tooltimeouterror) を伴う `error` イベントを発行し、その呼び出しではツール出力を送信しません。

#### ツール実行前の承認

`needsApproval: true` でツールを定義すると、ツール実行前にエージェントは `tool_approval_requested` イベントを発行します。

このイベントを監視することで、ツール呼び出しを承認 / 拒否する UI をユーザーに表示できます。

`await session.approve(request.approvalItem)` または `await session.reject(request.approvalItem)` でリクエストを解決します。関数ツールでは `{ alwaysApprove: true }` または `{ alwaysReject: true }` を渡すことで、セッション残り時間中の繰り返し呼び出しに同じ判断を再利用できます。Hosted MCP の承認は固定承認 / 拒否をサポートしないため、代わりに hosted MCP の `allowedTools` 構成でそれらのツールを制限してください。

### ガードレール

ガードレールは、エージェント発話がルールセットに違反したかを監視し、応答を即時遮断する方法を提供します。これらのチェックはエージェント応答の文字起こしストリームに対して実行されます。音声セッションでは SDK は出力音声文字起こしと文字起こし差分を使うため、重要な前提は別個のテキスト出力モダリティではなく文字起こしの可用性です。

提供したガードレールはモデル応答返却中に非同期で実行され、たとえば「特定の禁止語に言及した」などの事前定義分類トリガーに基づいて応答を遮断できます。

ガードレールが発火すると、セッションは `guardrail_tripped` イベントを発行します。このイベントは、ガードレールを発火させた `itemId` を含む `details` オブジェクトも提供します。

デフォルトではガードレールは 100 文字ごとに実行され、最終文字起こしが利用可能になると再度実行されます。通常、テキストを話す時間は文字起こし生成より長いため、これによりユーザーが聞く前に安全でない出力を遮断できることが多くなります。

## 会話状態と委譲

### 会話履歴管理

`RealtimeSession` は、ユーザーメッセージ、アシスタント出力、ツール呼び出し、切り詰め状態を追跡するローカル `history` スナップショットを自動維持します。これを UI に表示したり、ツール内で調査したり、項目修正 / 削除が必要な場合に更新したりできます。

会話が変化すると、セッションは `history_updated` を発行します。履歴変更を要求する必要がある場合は `updateHistory()` を使ってください。これはトランスポートに現在履歴との差分計算と必要な削除 / 作成イベント送信を要求し、対応する会話イベントが戻るとローカル `session.history` ビューが更新されます。

#### 制限事項

1. 現在、関数ツール呼び出しを事後編集することはできません
2. 履歴内のアシスタントテキストは、`output_audio.transcript` を含む利用可能な文字起こしに依存します
3. 割り込みで切り詰められた応答には最終文字起こしが保持されません
4. 入力音声文字起こしは、モデルが音声をどう解釈したかの正確なコピーではなく、ユーザー発話の大まかな目安として扱うのが適切です
