## セッション設定

### 音声処理

デフォルトの `OpenAIRealtimeWebRTC` のような一部のトランスポートレイヤーは、音声入力と出力を自動で処理します。`OpenAIRealtimeWebSocket` のような他のトランスポートでは、セッション音声を自分で処理する必要があります:

基盤トランスポートが対応している場合、`session.muted` は現在のミュート状態を返し、`session.mute(true | false)` はマイクキャプチャを切り替えます。`OpenAIRealtimeWebSocket` はミュートを実装していません。`session.muted` は `null` を返し、`session.mute()` は例外を投げます。そのため WebSocket 構成では、あなたの側でキャプチャを一時停止し、マイクを再度有効にするまで `sendAudio()` の呼び出しを止めてください。

### セッション構成

[`RealtimeSession`](/openai-agents-js/openai/agents-realtime/classes/realtimesession/) を作成するときに、通常は `model` オプションと `config` オブジェクトを通じてセッション自体を設定します。`connect(...)` は任意のセッションフィールドではなく、認証情報、エンドポイント URL、SIP 通話のアタッチといった接続時の関心事のためにあります。

内部では、SDK はこの構成を Realtime の [`session.update`](https://developers.openai.com/api/reference/resources/realtime/client-events/#session.update) 形式に正規化します。[RealtimeSessionConfig](/openai-agents-js/openai/agents-realtime/type-aliases/realtimesessionconfig/) に対応プロパティがない生のセッションフィールドが必要な場合は、`providerData` を使うか、`session.transport.sendEvent(...)` で生の `session.update` を送信してください。

`outputModalities`、`audio.input`、`audio.output` を使う新しい SDK 構成形式を推奨します。`modalities`、`inputAudioFormat`、`outputAudioFormat`、`inputAudioTranscription`、`turnDetection` などの旧 SDK エイリアスも後方互換性のために引き続き正規化されますが、新しいコードではここで示すネストされた `audio` 構造を使ってください。

speech-to-speech セッションでは、一般的に `outputModalities: ['audio']` を選びます。これにより音声出力と文字起こしが得られます。テキストのみの応答が必要な場合にだけ `['text']` に切り替えてください。

新しいパラメーターで [RealtimeSessionConfig](/openai-agents-js/openai/agents-realtime/type-aliases/realtimesessionconfig/) に対応項目がない場合は、`providerData` を使えます。`providerData` に渡したものはすべて、生の `session` オブジェクトの一部として転送されます。

### 会話ライフサイクル

`RealtimeSession` は長寿命の Realtime 接続の上にあります。会話履歴のローカルコピーを保持し、トランスポートイベントを監視し、ツールと出力ガードレールを実行し、アクティブエージェント設定をトランスポートと同期し続けます。

基盤 API の挙動も引き続き重要です:

- 接続成功時は `session.created` イベントで始まり、後続の構成変更では `session.updated` が生成されます
- 多くのセッションプロパティは時間経過で変更可能ですが、`model` は会話途中で変更できません。`voice` はセッションが音声出力を生成する前にのみ変更可能です。トレーシングは Realtime API で有効化後の変更ができないため、事前に決めるべきです
- Realtime API は現在、単一セッションを 60 分に制限しています
- 入力音声の文字起こしは非同期のため、最新発話の文字起こしが応答生成開始後に到着することがあります

SDK レイヤーでは、`await session.connect()` は「会話を開始できる程度にトランスポートの準備ができた」を意味しますが、正確な時点はトランスポートにより異なります:

- デフォルトのブラウザ WebRTC トランスポートでは、データチャネルが開いたらすぐに初期 `session.update` を送信し、`connect()` を解決する前に対応する `session.updated` イベントを待とうとします。これは instructions、tools、modalities が適用される前に音声がサーバーへ届くことを避けるためです。確認応答が来ない場合、`connect()` は短いタイムアウト後にフォールバックで解決します
- デフォルトのサーバーサイド WebSocket トランスポートでは、ソケットが開き初期構成が送信されると `connect()` が解決します。そのため対応する `session.updated` は `connect()` 解決後に到着する場合があります

生イベントモデルが必要な場合は、このページと併せて公式の [Realtime conversations guide](https://developers.openai.com/api/docs/guides/realtime-conversations/) を参照してください。

## インタラクションフロー

### ターン検出と音声アクティビティ検出

デフォルトでは、Realtime セッションは組み込みの voice activity detection（VAD）を使用し、ユーザーの発話開始 / 終了や応答作成タイミングを API が判断します。SDK では `audio.input.turnDetection` で公開されています。

<Code lang="typescript" code={turnDetectionExample} />

一般的な 2 つのモード:

- `semantic_vad` はより自然なターン境界を目指し、ユーザーがまだ話し終えていないように聞こえる場合は少し長く待てます
- `server_vad` はしきい値主導で、`threshold`、`prefixPaddingMs`、`silenceDurationMs`、`idleTimeoutMs` などの設定を公開します

ターン境界を自分で管理したい場合は、`audio.input.turnDetection` を `null` にしてください。基盤動作の詳細は、公式の [voice activity detection guide](https://developers.openai.com/api/docs/guides/realtime-vad/) と [Realtime conversations guide](https://developers.openai.com/api/docs/guides/realtime-conversations/#voice-activity-detection) を参照してください。

1. `audio.input.turnDetection = null` で VAD を完全に無効化する場合、音声ターンのコミットと `response.create` の送信はあなたの責任です

## エージェント機能

### ハンドオフ

通常のエージェントと同様に、ハンドオフを使ってエージェントを複数に分割し、それらの間をオーケストレーションすることで、性能向上と問題スコープの適切化ができます。

<Code lang="typescript" code={multiAgentsExample} />

通常のエージェントとは異なり、Realtime Agents ではハンドオフ挙動がやや異なります。ハンドオフ実行時、進行中セッションは新しいエージェント構成で更新されます。このため、新しいエージェントは進行中の会話履歴に自動的にアクセスでき、現在は入力フィルターは適用されません。

セッションはライブのまま維持されるため、ハンドオフ中にそのセッションのモデルは変わりません。音声変更は基盤 Realtime API ルールに従い、セッションが音声出力を生成する前にのみ機能します。Realtime ハンドオフは主に同一セッション内で `RealtimeAgent` 構成を切り替えるためのものです。たとえば `gpt-5.2` のような推論モデルなど別モデルを使う必要がある場合や、非リアルタイムのバックエンドエージェントへ委譲する場合は、[delegation through tools](#delegation-through-tools) を使ってください。

### ツール

通常のエージェントと同様に、Realtime Agents はアクション実行のためにツール呼び出しができます。Realtime は **関数ツール**（ローカル実行）と **hosted MCP ツール**（Realtime API によるリモート実行）をサポートします。通常のエージェントと同じ `tool()` ヘルパーで関数ツールを定義できます。

<Code lang="typescript" code={defineToolExample} />

#### 関数ツール

関数ツールは `RealtimeSession` と同じ環境で実行されます。つまり、セッションをブラウザで実行しているならツールもブラウザで実行されます。機密性の高い操作が必要な場合は、ツール内からバックエンドを呼び出し、特権処理はサーバーに実行させてください。

これにより、ブラウザ側ツールをサーバー側ロジックへの薄いバックチャネルとして使えます。たとえば [`examples/realtime-next`](https://github.com/openai/openai-agents-js/tree/main/examples/realtime-next) では、ブラウザ内の `refundBackchannel` ツールがリクエストと現在の会話履歴をサーバーの `handleRefundRequest(...)` に転送します。サーバーでは別の `Runner` が別エージェントまたは別モデルを使って返金判定を行い、結果を音声セッションへ返します。

#### Hosted MCP ツール

Hosted MCP ツールは `hostedMcpTool` で設定でき、リモート実行されます。MCP ツールの利用可否が変化すると、セッションは `mcp_tools_changed` を発行します。MCP ツール呼び出し完了後のモデル応答自動トリガーを防ぐには、`automaticallyTriggerResponseForMcpToolCalls: false` を設定してください。

現在のフィルター済み MCP ツール一覧は `session.availableMcpTools` としても取得できます。このプロパティと `mcp_tools_changed` イベントはどちらも、アクティブエージェントで有効化された hosted MCP サーバーのみを反映し、エージェント構成の `allowed_tools` フィルター適用後の状態です。

Hosted MCP 設定は、安全なサーバー選択・ヘッダー・承認を接続前構成として扱うと理解しやすくなります。`RealtimeSession.connect()` がトランスポートを開く前に、SDK はアクティブエージェントの hosted MCP ツール定義を解決し、対応する MCP フィールドを初期セッション構成へ含めて Realtime API に送信します。

このタイミングはブラウザ WebRTC アプリで特に重要です。一時的なクライアントシークレットは常にサーバーで発行されるため、秘匿すべき hosted MCP 認証情報やカスタム `headers` は、サーバー側の `POST /v1/realtime/client_secrets` リクエストに初期 `session` ペイロードの一部として付与してください。長期認証情報をブラウザコードに置いて、`connect()` 開始後に追加する運用は避けてください。

Realtime API レベルでは、後続の `session.update` 呼び出しでツールや他の可変セッションフィールドを変更できます。また SDK 自体もアクティブエージェント変更時に `session.update` を送信します。ただしブラウザアプリでは、安全な Hosted MCP 初期化はサーバー側の接続前関心事として扱い、ブラウザ側 `RealtimeSession` 構成をサーバーで発行した内容に合わせてください。

#### バックグラウンド結果

ツール実行中、エージェントはユーザーからの新規リクエストを処理できません。体験を改善する 1 つの方法は、ツール実行前にそれを案内する発話や、実行時間を稼ぐ定型句をエージェントに指示することです。

関数ツールを、即座に次のモデル応答をトリガーせず完了させたい場合は、`@openai/agents/realtime` の `backgroundResult(output)` を返してください。これによりツール出力はセッションへ返しつつ、応答トリガーはあなたが制御できます。

#### タイムアウト

関数ツールのタイムアウトオプション（`timeoutMs`、`timeoutBehavior`、`timeoutErrorFunction`）は Realtime セッションでも同様に動作します。デフォルトの `error_as_result` では、タイムアウトメッセージはツール出力として送信されます。`raise_exception` では、セッションは [`ToolTimeoutError`](/openai-agents-js/openai/agents-core/classes/tooltimeouterror) を伴う `error` イベントを発行し、その呼び出しのツール出力は送信しません。

#### 会話履歴へのアクセス

#### ツール実行前の承認

ツールを `needsApproval: true` で定義すると、ツール実行前にエージェントは `tool_approval_requested` イベントを発行します。

このイベントを監視することで、ユーザーにツール呼び出しを承認 / 拒否させる UI を表示できます。

`await session.approve(request.approvalItem)` または `await session.reject(request.approvalItem)` でリクエストを解決します。関数ツールでは、セッション残り期間の繰り返し呼び出しに同じ判断を再利用するため `{ alwaysApprove: true }` または `{ alwaysReject: true }` を渡せます。また、特定呼び出しに対して `session.reject(request.approvalItem, { message: '...' })` を使ってモデルへカスタム拒否メッセージを返せます。Hosted MCP 承認は sticky な承認 / 拒否をサポートしないため、代わりに hosted MCP の `allowedTools` 構成でこれらのツールを制限してください。

呼び出しごとの拒否 `message` を渡さない場合、セッションは `toolErrorFormatter`（設定されている場合）を使い、その後 SDK のデフォルト拒否テキストにフォールバックします。

### ガードレール

ガードレールは、エージェント発話がルールセットに違反したかを監視し、応答を即時に遮断する方法を提供します。これらのチェックはエージェント応答の文字起こしストリームに対して実行されます。音声セッションでは、SDK は出力音声文字起こしと文字起こし差分を使うため、重要な前提は別個のテキスト出力モダリティではなく文字起こしの可用性です。

提供したガードレールはモデル応答の返却中に非同期実行されるため、たとえば「特定の禁止語を含む」のような事前定義分類トリガーに基づいて応答を遮断できます。

ガードレールが発火すると、セッションは `guardrail_tripped` イベントを発行します。イベントには、ガードレールを発火させた `itemId` を含む `details` オブジェクトも含まれます。

## 会話状態と委譲

### 会話履歴管理

`RealtimeSession` は、ユーザーメッセージ、アシスタント出力、ツール呼び出し、切り詰め状態を追跡するローカル `history` スナップショットを自動管理します。これを UI に表示したり、ツール内で参照したり、項目の修正 / 削除が必要なときに更新したりできます。

会話変化時、セッションは `history_updated` を発行します。履歴変更を要求するには `updateHistory()` を使ってください。これはトランスポートに現在履歴との差分計算と必要な delete / create イベント送信を依頼し、対応する会話イベントが戻るとローカルの `session.history` ビューが更新されます。

#### 制限事項

1. 現時点では関数ツール呼び出しを事後編集できません
2. 履歴内のアシスタントテキストは、`output_audio.transcript` を含む利用可能な文字起こしに依存します
3. 割り込みで切り詰められた応答には最終文字起こしが保持されません
4. 入力音声文字起こしは、モデルが音声をどう解釈したかの正確な複製ではなく、ユーザー発話の大まかなガイドとして扱うのが適切です
