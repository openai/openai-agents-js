에이전트는 OpenAI Agents SDK의 핵심 구성 요소입니다. **Agent**는 다음으로 구성된 대규모 언어 모델(LLM)입니다:

- **Instructions** – 모델에 _자신이 누구인지_ 와 _어떻게 응답해야 하는지_ 알려주는 시스템 프롬프트
- **Model** – 호출할 OpenAI 모델과 선택적 모델 튜닝 매개변수
- **Tools** – 작업 수행을 위해 LLM이 호출할 수 있는 함수 또는 API 목록

## Agent 기본 사항

### 기본 설정

`Agent` 생성자는 하나의 설정 객체를 받습니다. 가장 자주 사용하는 속성은 아래와 같습니다.

| 속성 | 필수 | 설명 |
| --- | --- | --- |
| `name` | 예 | 사람이 읽기 쉬운 짧은 식별자 |
| `instructions` | 예 | 시스템 프롬프트(문자열 **또는** 함수 – [동적 instructions](#dynamic-instructions) 참고) |
| `prompt` | 아니요 | OpenAI Responses API 프롬프트 설정. 정적 프롬프트 객체 또는 함수를 받을 수 있습니다. [Prompt](/openai-agents-js/ko/guides/models#prompt) 참고 |
| `handoffDescription` | 아니요 | 이 에이전트가 핸드오프 도구로 제공될 때 사용하는 짧은 설명 |
| `handoffs` | 아니요 | 대화를 전문 에이전트로 위임합니다. [구성 패턴](#composition-patterns) 및 [핸드오프 가이드](/openai-agents-js/ko/guides/handoffs) 참고 |
| `model` | 아니요 | 모델 이름 **또는** 커스텀 [`Model`](/openai-agents-js/openai/agents/interfaces/model/) 구현 |
| `modelSettings` | 아니요 | 튜닝 매개변수(temperature, top_p 등). [모델](/openai-agents-js/ko/guides/models#modelsettings) 참고. 필요한 속성이 최상위에 없다면 `providerData` 아래에 포함할 수 있습니다 |
| `tools` | 아니요 | 모델이 호출할 수 있는 [`Tool`](/openai-agents-js/openai/agents/type-aliases/tool/) 인스턴스 배열. [도구](/openai-agents-js/ko/guides/tools) 참고 |
| `mcpServers` | 아니요 | 에이전트를 위한 MCP 기반 도구. [모델 컨텍스트 프로토콜 (MCP) 가이드](/openai-agents-js/ko/guides/mcp) 참고 |
| `inputGuardrails` | 아니요 | 이 에이전트 체인의 첫 사용자 입력에 적용되는 가드레일. [가드레일](/openai-agents-js/ko/guides/guardrails) 참고 |
| `outputGuardrails` | 아니요 | 이 에이전트의 최종 출력에 적용되는 가드레일. [가드레일](/openai-agents-js/ko/guides/guardrails) 참고 |
| `outputType` | 아니요 | 일반 텍스트 대신 structured outputs 반환. [출력 타입](#output-types) 및 [실행 결과](/openai-agents-js/ko/guides/results#final-output) 참고 |
| `toolUseBehavior` | 아니요 | 함수 도구 결과를 모델로 다시 보낼지 실행을 종료할지 제어. [도구 사용 강제](#forcing-tool-use) 참고 |
| `resetToolChoice` | 아니요 | 도구 호출 후 `toolChoice`를 기본값(기본: `true`)으로 재설정해 도구 사용 루프를 방지합니다. [도구 사용 강제](#forcing-tool-use) 참고 |
| `handoffOutputTypeWarningEnabled` | 아니요 | 핸드오프 출력 타입이 다를 때 경고를 발생시킵니다(기본: `true`). [실행 결과](/openai-agents-js/ko/guides/results#final-output) 참고 |

### 컨텍스트

에이전트는 **컨텍스트 타입에 대해 제네릭**입니다. 즉 `Agent<TContext, TOutput>`입니다. *컨텍스트*는 사용자가 생성해 `Runner.run()`에 전달하는 의존성 주입 객체입니다. 모든 도구, 가드레일, 핸드오프 등에 전달되며 상태 저장이나 공용 서비스(데이터베이스 연결, 사용자 메타데이터, 기능 플래그 등) 제공에 유용합니다.

### OpenAI 플랫폼 매핑

일부 에이전트 개념은 OpenAI 플랫폼 개념과 직접 매핑되며, 일부는 에이전트 정의 시점이 아니라 실행 시점에 설정됩니다.

| SDK 개념 | OpenAI 가이드 | 중요한 시점 |
| --- | --- | --- |
| `outputType` | [Structured Outputs](https://developers.openai.com/api/docs/guides/structured-outputs) | 에이전트가 텍스트 대신 타입이 지정된 JSON 또는 Zod 검증 객체를 반환해야 할 때 |
| `tools` / hosted tools | [Tools guide](https://developers.openai.com/api/docs/guides/tools) | 모델이 검색, 조회, 코드 실행 또는 함수/도구 호출을 해야 할 때 |
| `conversationId` / `previousResponseId` | [Conversation state](https://developers.openai.com/api/docs/guides/conversation-state) | 턴 간 대화 상태를 OpenAI에서 유지하거나 연결하고 싶을 때 |

`conversationId`와 `previousResponseId`는 `Agent` 생성자 필드가 아니라 런타임 제어 항목입니다. 해당 SDK 진입점이 필요하면 [에이전트 실행](/openai-agents-js/ko/guides/running-agents)을 사용하세요.

## 구성 패턴

에이전트가 더 큰 워크플로에 참여할 때 가장 자주 사용하는 SDK 진입점은 두 가지입니다:

1. **관리자(Agents as tools)** – 중앙 에이전트가 대화를 소유하고, 도구로 노출된 전문 에이전트를 호출합니다
2. **핸드오프** – 초기 에이전트가 사용자 요청을 식별하면 전체 대화를 전문 에이전트에 위임합니다

이 두 접근 방식은 상호 보완적입니다. 관리자는 가드레일이나 속도 제한을 적용할 단일 지점을 제공하고, 핸드오프는 각 에이전트가 대화 제어를 유지하지 않고도 단일 작업에 집중할 수 있게 합니다. 설계상 트레이드오프와 각 패턴의 선택 기준은 [에이전트 오케스트레이션](/openai-agents-js/ko/guides/multi-agent)을 참고하세요.

#### 관리자(Agents as tools)

이 패턴에서 관리자는 제어를 넘기지 않습니다. LLM이 도구를 사용하고 관리자가 최종 답변을 요약합니다. 자세한 내용은 [도구 가이드](/openai-agents-js/ko/guides/tools#agents-as-tools)를 참고하세요.

#### 핸드오프

핸드오프에서는 트리아지 에이전트가 요청을 라우팅하지만, 핸드오프가 발생하면 전문 에이전트가 최종 출력을 생성할 때까지 대화를 소유합니다. 이렇게 하면 프롬프트를 짧게 유지하고 각 에이전트를 독립적으로 다루기 쉬워집니다. 자세한 내용은 [핸드오프 가이드](/openai-agents-js/ko/guides/handoffs)를 참고하세요.

### 라이프사이클 훅

고급 사용 사례에서는 이벤트 리스닝으로 Agent 라이프사이클을 관찰할 수 있습니다.

`Agent` 인스턴스는 해당 인스턴스의 라이프사이클 이벤트를 발생시키고, `Runner`는 전체 실행에 걸쳐 동일한 이벤트 이름을 단일 스트림으로 발생시킵니다. 이는 핸드오프와 도구 호출을 한 곳에서 관찰하려는 멀티 에이전트 워크플로에서 유용합니다.

공통 이벤트 이름은 다음과 같습니다:

| 이벤트 | Agent 훅 인자 | Runner 훅 인자 |
| --- | --- | --- |
| `agent_start` | `(context, agent, turnInput?)` | `(context, agent, turnInput?)` |
| `agent_end` | `(context, output)` | `(context, agent, output)` |
| `agent_handoff` | `(context, nextAgent)` | `(context, fromAgent, toAgent)` |
| `agent_tool_start` | `(context, tool, { toolCall })` | `(context, agent, tool, { toolCall })` |
| `agent_tool_end` | `(context, tool, result, { toolCall })` | `(context, agent, tool, result, { toolCall })` |

### 도구 사용 강제

도구를 제공해도 LLM이 반드시 호출하는 것은 아닙니다. `modelSettings.toolChoice`로 도구 사용을 **강제**할 수 있습니다:

1. `'auto'` (기본값) – LLM이 도구 사용 여부를 결정
2. `'required'` – LLM이 반드시 도구를 호출(어떤 도구인지는 선택 가능)
3. `'none'` – LLM이 도구를 **호출하지 않아야 함**
4. 특정 도구 이름(예: `'calculator'`) – LLM이 해당 도구를 반드시 호출

#### 무한 루프 방지

도구 호출 후 SDK는 자동으로 `toolChoice`를 `'auto'`로 재설정합니다. 이렇게 하면 모델이 도구를 반복 호출하려는 무한 루프에 빠지는 것을 방지할 수 있습니다. `resetToolChoice` 플래그 또는 `toolUseBehavior` 설정으로 이 동작을 재정의할 수 있습니다:

- `'run_llm_again'` (기본값) – 도구 결과를 포함해 LLM을 다시 실행
- `'stop_on_first_tool'` – 첫 번째 도구 결과를 최종 답변으로 처리
- `{ stopAtToolNames: ['my_tool'] }` – 목록의 도구 중 하나라도 호출되면 중지
- `(context, toolResults) => ...` – 실행 종료 여부를 반환하는 커스텀 함수

```typescript
const agent = new Agent({
  ...,
  toolUseBehavior: 'stop_on_first_tool',
});
```

참고: `toolUseBehavior`는 **함수 도구**에만 적용됩니다. Hosted tools는 항상 처리를 위해 모델로 돌아갑니다.
