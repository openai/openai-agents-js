도구는 에이전트가 **행동을 수행**하도록 합니다. 데이터를 가져오고, 외부 API를 호출하고, 코드를 실행하거나, 심지어 컴퓨터를 사용할 수 있습니다. JavaScript/TypeScript SDK는 여섯 가지 카테고리를 지원합니다:

1. **OpenAI 호스트하는 도구** – OpenAI 서버에서 모델과 함께 실행됨 _(웹 검색, 파일 검색, Code Interpreter, 이미지 생성)_
2. **로컬 내장 도구** – 사용자의 환경에서 실행됨 _(컴퓨터 사용, shell, apply_patch)_
3. **함수 도구** – 로컬 함수를 JSON 스키마로 감싸 LLM이 호출할 수 있게 함
4. **Agents as tools** – 전체 에이전트를 호출 가능한 도구로 노출
5. **MCP servers** – Model context protocol (MCP) 서버(로컬 또는 원격) 연결
6. **실험적: Codex 도구** – Codex SDK를 함수 도구로 감싸 작업공간 인지 작업을 실행

## 2. 로컬 내장 도구

로컬 내장 도구는 사용자의 환경에서 실행되며 구현을 제공해야 합니다:

- **컴퓨터 사용** – `Computer` 인터페이스를 구현하고 `computerTool()`에 전달
- **Shell** – 로컬 `Shell` 구현을 제공하거나, 호스팅된 컨테이너 환경을 구성
- **패치 적용** – `Editor` 인터페이스를 구현하고 `applyPatchTool()`에 전달

컴퓨터 및 패치 적용 도구는 로컬에서 실행되며 OpenAI에서 호스팅되지 **않습니다**. Shell 도구는 `shellTool()` 구성에 따라 로컬 또는 호스팅된 컨테이너 환경에서 실행될 수 있습니다.
도구 호출은 여전히 모델의 응답으로 요청되지만, 호출이 어떻게 실행되는지는 애플리케이션이 제어합니다.

### 옵션 레퍼런스

| 필드               | 필수   | 설명                                                                                                                                                      |
| ------------------ | ------ | --------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `name`             | 아니오 | 기본값은 함수 이름(예: `get_weather`)                                                                                                                     |
| `description`      | 예     | LLM에 표시되는 명확하고 사람이 읽을 수 있는 설명                                                                                                          |
| `parameters`       | 예     | Zod 스키마 또는 원문 JSON 스키마 객체 중 하나. Zod parameters는 자동으로 **strict** 모드를 활성화                                                         |
| `strict`           | 아니오 | `true`(기본값)일 때, 인수가 유효성 검사를 통과하지 못하면 SDK가 모델 오류를 반환. 퍼지 매칭이 필요하면 `false`로 설정                                     |
| `execute`          | 예     | `(args, context) => string \| unknown \| Promise<...>` – 비즈니스 로직. 문자열이 아닌 출력은 모델을 위해 직렬화됨. 두 번째 선택적 매개변수는 `RunContext` |
| `errorFunction`    | 아니오 | 내부 오류를 사용자 가시 문자열로 변환하는 커스텀 핸들러 `(context, error) => string`                                                                      |
| `needsApproval`    | 아니오 | 실행 전 휴먼 승인 요구. 자세한 내용은 [휴먼 인 더 루프 (HITL)](/openai-agents-js/ko/guides/human-in-the-loop) 가이드 참고                                 |
| `isEnabled`        | 아니오 | 실행별로 조건부로 도구 노출. 불리언 또는 프레디킷 허용                                                                                                    |
| `inputGuardrails`  | 아니오 | 도구 실행 전 실행되는 가드레일. 거부하거나 throw 가능. [가드레일](/openai-agents-js/ko/guides/guardrails#tool-guardrails) 참고                            |
| `outputGuardrails` | 아니오 | 도구 실행 후 실행되는 가드레일. 거부하거나 throw 가능. [가드레일](/openai-agents-js/ko/guides/guardrails#tool-guardrails) 참고                            |

## 4. Agents as tools

대화를 완전히 핸드오프하지 않고 한 에이전트가 다른 에이전트를 *보조*하도록 하려면 `agent.asTool()`을 사용하세요:

<Code lang="typescript" code={agentsAsToolsExample} title="Agents as tools" />

내부적으로 SDK는 다음을 수행합니다:

- 단일 `input` 매개변수를 가진 함수 도구를 생성
- 도구가 호출되면 해당 입력으로 하위 에이전트를 실행
- 마지막 메시지 또는 `customOutputExtractor`로 추출한 출력을 반환

에이전트를 도구로 실행하면, Agents SDK는 기본 설정으로 runner를 생성하고 함수 실행 내에서 그 runner로 에이전트를 실행합니다. `runConfig`나 `runOptions`의 속성을 제공하려면 `asTool()`에 전달해 runner 동작을 커스터마이즈할 수 있습니다.

또한 `asTool()` 옵션을 통해 에이전트 도구에 `needsApproval`과 `isEnabled`를 설정하여 휴먼인더루프(HITL) 플로우 및 조건부 도구 가용성과 통합할 수 있습니다.

`agent.asTool()`의 고급 구조화 입력 옵션:

- `inputBuilder`: 구조화된 도구 인수를 중첩 에이전트 입력 페이로드에 매핑
- `includeInputSchema`: 더 강력한 스키마 인지 동작을 위해 중첩 실행에 입력 JSON 스키마 포함
- `resumeState`: 중첩 직렬화 `RunState`를 재개할 때 컨텍스트 정합 전략 제어

### 에이전트 도구의 스트리밍 이벤트

에이전트 도구는 모든 중첩 실행 이벤트를 앱으로 스트리밍할 수 있습니다. 도구 구성 방식에 맞는 훅 스타일을 선택하세요:

<Code
  lang="typescript"
  code={agentsAsToolsStreamingExample}
  title="Streaming agent tools"
/>

- 이벤트 유형은 `RunStreamEvent['type']`와 일치: `raw_model_stream_event`, `run_item_stream_event`, `agent_updated_stream_event`
- `onStream`은 가장 단순한 포괄형이며 도구를 인라인으로 선언할 때 잘 동작함(`tools: [agent.asTool({ onStream })]`). 이벤트별 라우팅이 필요 없다면 이 방식을 사용
- `on(eventName, handler)`는 선택적으로(또는 `'*'`로) 구독할 수 있어 더 세밀한 처리가 필요하거나 생성 후 리스너를 부착하고 싶을 때 적합
- `onStream` 또는 어떤 `on(...)` 핸들러든 제공하면 agent-as-tool은 자동으로 스트리밍 모드로 실행됨. 없으면 비스트리밍 경로 유지
- 핸들러는 병렬로 호출되므로 느린 `onStream` 콜백이 `on(...)` 핸들러를 차단하지 않음(반대도 마찬가지)
- `toolCallId`는 도구가 모델의 도구 호출을 통해 호출되었을 때 제공됨. 직접 `invoke()` 호출 또는 공급자 특성에 따라 생략될 수 있음

## 6. 실험적: Codex 도구

`@openai/agents-extensions/experimental/codex`는 `codexTool()`을 제공하며, 모델 도구 호출을 Codex SDK로 라우팅하여 에이전트가 작업공간 범위의 작업(shell, 파일 편집, MCP 도구)을 자율적으로 실행할 수 있게 합니다. 이 표면은 실험적이며 변경될 수 있습니다.

먼저 의존성을 설치하세요:

```bash
npm install @openai/agents-extensions @openai/codex-sdk
```

빠른 시작:

<Code
  lang="typescript"
  code={codexToolExample}
  title="Experimental Codex tool"
/>

알아둘 점:

- 인증: `CODEX_API_KEY`(권장) 또는 `OPENAI_API_KEY`를 제공하거나 `codexOptions.apiKey`를 전달
- 입력: 엄격한 스키마 — `inputs`에는 최소 하나의 `{ type: 'text', text }` 또는 `{ type: 'local_image', path }`가 포함되어야 함
- 안전: `sandboxMode`를 `workingDirectory`와 함께 사용. 디렉터리가 Git 저장소가 아니면 `skipGitRepoCheck` 설정
- 스레딩: `useRunContextThreadId: true`는 최신 thread id를 `runContext.context`에서 읽고 저장하며, 앱 상태에서 턴 간 재사용에 유용
- 스레드 ID 우선순위: 도구 호출의 `threadId`(스키마에 포함된 경우)가 우선, 그다음 런 컨텍스트 thread id, 그다음 `codexTool({ threadId })`
- 런 컨텍스트 키: 기본값은 `name: 'codex'`일 때 `codexThreadId`, `name: 'engineer'`와 같은 이름일 때는 `codexThreadId_<suffix>`(정규화 후 `codex_engineer`)
- 변경 가능한 컨텍스트 요구사항: `useRunContextThreadId` 사용 시 `run(..., { context })`에 변경 가능한 객체 또는 `Map` 전달
- 네이밍: 도구 이름은 `codex` 네임스페이스로 정규화됨(`engineer`는 `codex_engineer`가 됨). 하나의 에이전트 내 중복 Codex 도구 이름은 거부
- 스트리밍: `onStream`은 Codex 이벤트(추론, 명령 실행, MCP 도구 호출, 파일 변경, 웹 검색)를 미러링하여 진행 상황을 로깅하거나 트레이싱 가능
- 출력: 도구 결과에는 `response`, `usage`, `threadId`가 포함되며, Codex 토큰 사용량은 `RunContext`에 기록됨
- 구조: `outputSchema`는 디스크립터, JSON 스키마 객체, 또는 Zod 객체일 수 있음. JSON 객체 스키마의 경우 `additionalProperties`는 `false`여야 함

## 모범 사례

- **짧고 명시적인 설명** – 도구가 _무엇을_ 하는지와 *언제 사용하는지*를 설명
- **입력 검증** – 가능하면 엄격한 JSON 검증을 위해 Zod 스키마 사용
- **에러 핸들러의 부작용 회피** – `errorFunction`은 유용한 문자열을 반환해야 하며 throw하지 않아야 함
- **도구당 하나의 책임** – 작고 합성 가능한 도구가 더 나은 모델 추론으로 이어짐
