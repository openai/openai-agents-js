에이전트를 실행하면 최종 출력과 실행 전체 기록을 포함하는 [결과](/openai-agents-js/ko/guides/results) 객체를 받게 됩니다.

## 에이전트 루프

Runner에서 run 메서드를 사용할 때 시작 에이전트와 입력을 전달합니다. 입력은 문자열(사용자 메시지로 간주) 또는 OpenAI Responses API의 항목과 동일한 입력 항목 목록일 수 있습니다.

그 다음 러너는 다음 루프를 실행합니다:

1. 현재 입력으로 현재 에이전트의 모델을 호출합니다.
2. LLM 응답을 확인합니다.
   - **최종 출력** → 반환
   - **핸드오프** → 새로운 에이전트로 전환, 누적된 대화 기록 유지, 1로 이동
   - **도구 호출** → 도구 실행, 결과를 대화에 추가, 1로 이동
3. `maxTurns`에 도달하면 [`MaxTurnsExceededError`](/openai-agents-js/openai/agents-core/classes/maxturnsexceedederror)를 발생시킵니다.

<Aside type="note">
  LLM 출력이 "최종 출력"으로 간주되는 규칙은 원하는 타입의 텍스트 출력을
  생성하고, 도구 호출이 없는 경우입니다.
</Aside>

### Runner 수명 주기

앱이 시작될 때 `Runner`를 생성하고 요청마다 재사용하세요. 이 인스턴스는 모델 제공자와 트레이싱 옵션 같은 전역 설정을 저장합니다. 완전히 다른 구성이 필요할 때만 다른 `Runner`를 만드세요. 간단한 스크립트의 경우 내부적으로 기본 러너를 사용하는 `run()`을 호출할 수도 있습니다.

## 실행 인수

`run()` 메서드의 입력은 실행을 시작할 초기 에이전트, 실행 입력, 그리고 옵션 집합입니다.

입력은 문자열(사용자 메시지로 간주), [입력 항목](/openai-agents-js/openai/agents-core/type-aliases/agentinputitem) 목록, 또는 [휴먼 인 더 루프](/openai-agents-js/ko/guides/human-in-the-loop) 에이전트를 구성하는 경우 [`RunState`](/openai-agents-js/openai/agents-core/classes/runstate) 객체가 될 수 있습니다.

추가 옵션은 다음과 같습니다:

| Option                  | Default | Description                                                                                                                                       |
| ----------------------- | ------- | ------------------------------------------------------------------------------------------------------------------------------------------------- |
| `stream`                | `false` | `true`인 경우 `StreamedRunResult`를 반환하고 모델에서 도착하는 이벤트를 스트리밍으로 내보냅니다.                                                  |
| `context`               | –       | 모든 tool / guardrail / handoff에 전달되는 컨텍스트 객체입니다. [컨텍스트 가이드](/openai-agents-js/ko/guides/context)에서 자세히 알아보세요.     |
| `maxTurns`              | `10`    | 안전 제한 – 도달 시 [`MaxTurnsExceededError`](/openai-agents-js/openai/agents-core/classes/maxturnsexceedederror) 발생                            |
| `signal`                | –       | 취소를 위한 `AbortSignal`                                                                                                                         |
| `session`               | –       | 세션 지속성 구현. [세션 가이드](/openai-agents-js/ko/guides/sessions) 참고                                                                        |
| `sessionInputCallback`  | –       | 세션 기록과 새 입력을 병합하는 커스텀 로직; 모델 호출 전 실행됩니다. [sessions](/openai-agents-js/ko/guides/sessions) 참고                        |
| `callModelInputFilter`  | –       | 모델 호출 직전에 모델 입력(항목 + 선택적 instructions)을 편집하는 훅. [Call model input filter](#call-model-input-filter) 참고                    |
| `toolErrorFormatter`    | –       | 모델에 반환되는 도구 승인 거부 메시지를 커스터마이즈하는 훅. [Tool error formatter](#tool-error-formatter) 참고                                   |
| `reasoningItemIdPolicy` | –       | 이전 실행 항목을 모델 입력으로 되돌릴 때 reasoning-item `id`를 보존할지 생략할지 제어. [Reasoning item ID policy](#reasoning-item-id-policy) 참고 |
| `tracing`               | –       | 실행별 트레이싱 구성 재정의(예: export API key)                                                                                                   |
| `errorHandlers`         | –       | 지원되는 런타임 오류에 대한 핸들러(현재 `maxTurns`). [Error handlers](#error-handlers) 참고                                                       |
| `conversationId`        | –       | 서버 측 대화 재사용(OpenAI Responses API + Conversations API 전용)                                                                                |
| `previousResponseId`    | –       | 대화를 만들지 않고 이전 Responses API 호출에서 계속하기(OpenAI Responses API 전용)                                                                |

## 스트리밍

스트리밍을 사용하면 LLM 실행 중 스트리밍 이벤트를 추가로 받을 수 있습니다. 스트림이 시작되면 `StreamedRunResult`에는 모든 새 출력물을 포함해 실행에 대한 완전한 정보가 담깁니다. `for await` 루프로 스트리밍 이벤트를 순회할 수 있습니다. 자세한 내용은 [스트리밍 가이드](/openai-agents-js/ko/guides/streaming)를 참고하세요.

## Call model input filter

`callModelInputFilter`를 사용해 모델이 호출되기 바로 직전에 모델 입력을 편집합니다. 이 훅은 현재 에이전트, 컨텍스트, 결합된 입력 항목(세션 기록 포함 시 포함)을 받습니다. 민감한 데이터 마스킹, 오래된 메시지 제거, 추가 시스템 가이드 삽입 등을 위해 업데이트된 `input` 배열과 선택적 `instructions`를 반환하세요.

실행별로 `runner.run(..., { callModelInputFilter })`에 설정하거나, `Runner` 구성(`RunConfig`의 `callModelInputFilter`)에서 기본값으로 설정할 수 있습니다.

## Tool error formatter

도구 호출이 거부될 때 모델에 다시 전송되는 승인 거부 메시지를 커스터마이즈하려면 `toolErrorFormatter`를 사용하세요. 이를 통해 SDK 기본 메시지 대신 도메인별 문구(예: 컴플라이언스 가이드)를 반환할 수 있습니다.

포매터는 실행별(`runner.run(..., { toolErrorFormatter })`)로 설정하거나 `RunConfig`에서 전역(`new Runner(...)`의 `toolErrorFormatter`)으로 설정할 수 있습니다.

포매터는 현재 `approval_rejected` 이벤트에 대해 실행되며 다음을 전달받습니다:

- `kind` (현재 항상 `'approval_rejected'`)
- `toolType` (`'function'`, `'computer'`, `'shell'`, 또는 `'apply_patch'`)
- `toolName`
- `callId`
- `defaultMessage` (SDK 기본 메시지)
- `runContext`

메시지를 재정의하려면 문자열을 반환하고, SDK 기본 메시지를 유지하려면 `undefined`를 반환하세요. 포매터가 예외를 던지거나 문자열이 아닌 값을 반환하면, SDK는 경고를 기록하고 기본 승인 거부 메시지로 폴백합니다.

## Reasoning item ID 정책

`reasoningItemIdPolicy`를 사용해, SDK가 이전에 생성된 실행 항목을 나중의 모델 입력을 위한 `AgentInputItem[]`로 변환할 때 reasoning 항목이 `id` 필드를 유지할지 제어합니다.

이는 다음과 같이 생성된 모델 항목을 입력으로 재생하는 경우에 영향을 줍니다:

- 동일 실행 내 후속 모델 호출(예: 도구 실행 후)
- 생성된 항목을 입력/기록으로 재사용하는 이후 턴
- 저장된 `RunState`에서 재개된 실행
- `result.history` / `result.output` 같은 파생 결과 뷰(모델 입력 형태의 배열)

- `'preserve'`(기본값)는 reasoning-item ID를 유지합니다.
- `'omit'`은 입력으로 다시 보내기 전에 reasoning 항목에서 `id` 필드를 제거합니다.
- reasoning이 아닌 항목은 영향을 받지 않습니다.

다음에는 영향을 주지 않습니다:

- 원문 모델 응답(`result.rawResponses`)
- 실행 항목(`result.newItems`)
- 제공자가 반환하는 모델의 현재 턴 출력

즉, 이 정책은 SDK가 이전에 생성된 항목으로부터 다음 입력을 구성할 때 적용됩니다.

정책은 실행별로 설정할 수 있습니다(`runner.run(..., { reasoningItemIdPolicy: 'omit' })`) 또는 러너 기본값으로 설정할 수 있습니다(`new Runner({ reasoningItemIdPolicy: 'omit', ... })`). 저장된 `RunState`에서 재개할 때, 이전에 결정된 정책이 재사용되며, 재정의하지 않는 한 그대로 유지됩니다.

### `callModelInputFilter`와의 상호 작용

`reasoningItemIdPolicy`는 `callModelInputFilter` 전에 적용됩니다. 커스텀 동작이 필요하면, `callModelInputFilter`에서 준비된 입력을 검사하고 모델 호출 전에 reasoning ID를 다시 도입하거나 제거할 수 있습니다.

### `'omit'`을 사용할 때

재생된 reasoning 항목을 ID 없이 정규화하고 싶을 때 `'omit'`을 사용하세요(예: 전달/재생된 모델 입력을 더 단순하게 유지하거나 앱 파이프라인의 통합 요구 사항에 맞추기 위해).

또한 백엔드/제공자가 재생된 reasoning 항목을 요청 검증 오류로 거부하는 경우(예: 후속 입력의 reasoning item ID와 관련된 HTTP `400` 오류), 문제 해결 옵션으로도 유용합니다. 이러한 경우 `'omit'`으로 재생된 reasoning ID를 제거하면 백엔드가 새 요청에 대해 유효하지 않다고 처리하는 ID 전송을 피할 수 있습니다.

재생된 입력을 통해 reasoning-item ID를 SDK가 전달하기를 원하고, 통합이 이를 허용하는 경우에는 `'preserve'`를 유지하세요.

## 오류 핸들러

`errorHandlers`를 사용하면 지원되는 런타임 오류를 throw하는 대신 최종 출력으로 변환할 수 있습니다. 현재는 `maxTurns`만 지원됩니다.

- `errorHandlers.maxTurns`는 최대 턴 오류만 처리합니다.
- `errorHandlers.default`는 지원되는 종류에 대한 폴백으로 사용됩니다.
- 핸들러는 `{ error, context, runData }`를 받고 `{ finalOutput, includeInHistory? }`를 반환할 수 있습니다.

- [`GuardrailExecutionError`](/openai-agents-js/openai/agents-core/classes/guardrailexecutionerror) – 가드레일 실행 실패
- [`ToolTimeoutError`](/openai-agents-js/openai/agents-core/classes/tooltimeouterror) – 함수 도구가 `timeoutMs`를 초과했고 `timeoutBehavior: 'raise_exception'`을 사용
- [`ToolCallError`](/openai-agents-js/openai/agents-core/classes/toolcallerror) – 타임아웃 이외의 이유로 함수 도구 실행 실패
- [`UserError`](/openai-agents-js/openai/agents-core/classes/usererror) – 구성 또는 사용자 입력에 기반해 발생한 모든 오류

모두 기본 `AgentsError` 클래스를 확장하며, 여기에는 현재 실행 상태에 접근할 수 있는 `state` 속성이 제공될 수 있습니다.

입력 vs. 출력 재시도:

- 입력 가드레일은 실행의 첫 번째 사용자 입력에서만 실행되므로, 동일한 입력/컨텍스트로 새 실행을 시작해야 재시도할 수 있습니다. 저장된 `state`를 전달해도 입력 가드레일은 다시 트리거되지 않습니다.
- 출력 가드레일은 모델 응답 후에 실행되므로, `GuardrailExecutionError`의 저장된 `state`를 재사용해 모델을 다시 호출하지 않고 출력 가드레일만 다시 실행할 수 있습니다.
