[**Model Context Protocol (MCP)**](https://modelcontextprotocol.io)은 애플리케이션이 LLM에 도구와 컨텍스트를 제공하는 방식을 표준화하는 개방형 프로토콜입니다. MCP 문서의 설명은 다음과 같습니다.

> MCP는 애플리케이션이 LLM에 컨텍스트를 제공하는 방식을 표준화하는 개방형 프로토콜입니다. MCP를 AI 애플리케이션용 USB-C 포트라고 생각할 수 있습니다. USB-C가 기기를 다양한 주변 장치와 액세서리에 연결하는 표준화된 방식을 제공하는 것처럼, MCP는 AI 모델을 다양한 데이터 소스와 도구에 연결하는 표준화된 방식을 제공합니다.

이 SDK는 세 가지 유형의 MCP 서버를 지원합니다.

1. **호스티드 MCP 서버 도구** – [OpenAI Responses API](https://platform.openai.com/docs/guides/tools-remote-mcp)가 도구로 사용하는 원격 MCP 서버
2. **스트리밍 가능한 HTTP MCP 서버** – [스트리밍 가능한 HTTP 전송 방식](https://modelcontextprotocol.io/docs/concepts/transports#streamable-http)을 구현하는 로컬 또는 원격 서버
3. **Stdio MCP 서버** – 표준 입력/출력을 통해 접근하는 서버(가장 간단한 옵션)

> 참고: SDK에는 레거시 Server-Sent Events 전송 방식을 위한 `MCPServerSSE`도 포함되어 있지만, MCP 프로젝트에서는 SSE가 더 이상 권장되지 않습니다. 새로운 연동에는 스트리밍 가능한 HTTP 또는 stdio를 사용하는 것이 좋습니다.

사용 사례에 따라 서버 유형을 선택하세요.

| 필요한 기능 | 권장 옵션 |
| --- | --- |
| 기본 OpenAI Responses 모델을 사용하여 공개적으로 접근할 수 있는 원격 서버 호출 | **1. 호스티드 MCP 도구** |
| 공개적으로 접근할 수 있는 원격 서버를 사용하되 도구 호출은 로컬에서 트리거 | **2. 스트리밍 가능한 HTTP** |
| 로컬에서 실행되는 스트리밍 가능한 HTTP 서버 사용 | **2. 스트리밍 가능한 HTTP** |
| OpenAI Responses 이외의 모델에서 스트리밍 가능한 HTTP 서버 사용 | **2. 스트리밍 가능한 HTTP** |
| 표준 I/O 프로토콜만 지원하는 로컬 MCP 서버 사용 | **3. Stdio** |

### 호스티드 MCP 옵션 참고 자료

`hostedMcpTool(...)`은 MCP 서버 URL과 커넥터 기반 서버를 모두 지원합니다.

| 옵션 | 유형 | 참고 |
| --- | --- | --- |
| `serverLabel` | `string` | 이벤트와 트레이스에서 호스티드 MCP 서버를 식별하는 필수 레이블입니다. |
| `serverUrl` | `string` | 원격 MCP 서버 URL입니다. 일반 호스티드 MCP 서버에 사용합니다. |
| `connectorId` | `string` | OpenAI 커넥터 ID입니다. 커넥터 기반 호스티드 서버에서는 `serverUrl` 대신 사용합니다. |
| `authorization` | `string` | 호스티드 MCP 백엔드로 전송되는 선택적 인증 토큰입니다. |
| `headers` | `Record<string, string>` | 선택적 추가 요청 헤더입니다. |
| `allowedTools` | `string[] \| object` | 모델에 노출할 도구 이름의 허용 목록입니다. `string[]` 또는 `{ toolNames?: string[] }`을 전달합니다. |
| `allowedCallers` | `('direct' \| 'programmatic')[]` | 호스티드 MCP 도구를 직접 호출할지, 프로그래매틱 도구 호출을 통해 호출할지, 또는 두 방식 모두 허용할지를 제어하는 Responses 전용 비어 있지 않은 목록입니다. |
| `deferLoading` | `boolean` | 호스티드 MCP 도구를 위한 Responses 전용 지연 로딩입니다. 같은 에이전트에 `toolSearchTool()`이 필요합니다. |
| `requireApproval` | `'never' \| 'always' \| object` | 호스티드 MCP 도구 호출에 대한 승인 정책입니다. 도구별로 재정의하려면 객체 형식을 사용합니다. 기본값은 `'never'`입니다. |
| `onApproval` | 승인 콜백 | `requireApproval`에 승인 처리가 필요한 경우 프로그래밍 방식의 승인/거부를 위한 선택적 콜백입니다. |

호스티드 MCP 서버의 도구 정의를 처음부터 노출하는 대신 도구 검색을 통해 필요할 때 모델이 로드하도록 하려면 `deferLoading: true`를 설정합니다. 이 기능은 OpenAI Responses API에서만 작동하고 같은 요청에 `toolSearchTool()`이 필요하며, GPT-5.4 및 이후 지원 모델 릴리스에서 사용해야 합니다. 전체 지연 로딩 설정은 [도구](/openai-agents-js/ko/guides/tools#deferred-tool-loading-with-tool-search)를 참고하세요.

호스티드 MCP 도구를 모델이 생성한 JavaScript에서만 호출해야 한다면 `allowedCallers: ['programmatic']`을 설정하고, 두 경로를 모두 허용하려면 `'direct'`와 `'programmatic'`을 모두 포함합니다. 같은 에이전트에 `programmaticToolCallingTool()`을 추가합니다. 프로그램이 MCP 도구를 호출할 때도 기존 `requireApproval` 및 `onApproval` 정책이 적용됩니다. 전체 설정은 [프로그래매틱 도구 호출](/openai-agents-js/ko/guides/tools#programmatic-tool-calling)을 참고하세요.

`requireApproval` 객체 형식:

```ts
{
  always?: { toolNames: string[] };
  never?: { toolNames: string[] };
}
```

`onApproval` 시그니처:

```ts
async function onApproval(
  context,
  item,
): Promise<{
  approve: boolean;
  reason?: string;
}> {}
```

## 에이전트 수준 MCP 설정

전송 방식을 선택하는 것 외에도 `Agent.mcpConfig`를 설정하여 로컬 MCP 도구의 준비 방식을 조정할 수 있습니다.

```ts
const agent = new Agent({
  name: 'Assistant',
  mcpServers: [server],
  mcpConfig: {
    // Try to convert MCP tool schemas to strict JSON schema.
    convertSchemasToStrict: true,
    // Set to null to raise MCP tool failures instead of returning model-visible error text.
    errorFunction: null,
    // Prefix local MCP tool names with their server name.
    includeServerInToolNames: true,
  },
});
```

참고:

- `convertSchemasToStrict`는 최선형 방식으로 작동합니다. 스키마를 변환할 수 없으면 원래 스키마를 사용합니다.
- `errorFunction`은 MCP 도구 호출 실패를 모델에 표시하는 방식을 제어합니다.
- `errorFunction`을 설정하지 않으면 SDK가 기본 도구 오류 포매터를 사용합니다.
- 서버 수준의 `errorFunction` 값은 해당 서버의 `Agent.mcpConfig.errorFunction`을 재정의합니다.
- `includeServerInToolNames`는 명시적으로 활성화해야 합니다. 활성화하면 각 로컬 MCP 도구가 결정론적인 서버 접두사 이름으로 모델에 노출되므로 여러 MCP 서버에서 같은 이름의 도구를 게시할 때 충돌을 방지하는 데 도움이 됩니다.

- **여러 서버 동시 사용**: 모든 서버에 병렬로 연결하고 에이전트에는 `mcpServers.active`를 사용
- **부분 실패 처리**: `failed` + `errors`를 확인하고 계속 진행할지 재시도할지 결정
- **실패한 서버 재시도**: `mcpServers.reconnect()` 호출(기본적으로 실패한 서버만 재시도)

엄격한 "전부 성공 또는 전부 실패" 연결이나 다른 제한 시간이 필요하면 `connectMcpServers(servers, options)`를 사용하고 환경에 맞게 옵션을 조정합니다.

`connectMcpServers` 옵션:

| 옵션 | 유형 | 기본값 | 참고 |
| --- | --- | --- | --- |
| `connectTimeoutMs` | `number \| null` | `10000` | 각 서버의 `connect()` 제한 시간입니다. 비활성화하려면 `null`을 사용합니다. |
| `closeTimeoutMs` | `number \| null` | `10000` | 각 서버의 `close()` 제한 시간입니다. 비활성화하려면 `null`을 사용합니다. |
| `dropFailed` | `boolean` | `true` | 실패한 서버를 `active`에서 제외합니다. |
| `strict` | `boolean` | `false` | 서버 연결에 하나라도 실패하면 예외를 발생시킵니다. |
| `suppressAbortError` | `boolean` | `true` | 실패한 서버를 계속 추적하면서 중단과 유사한 오류는 무시합니다. |
| `connectInParallel` | `boolean` | `false` | 모든 서버를 순차적으로 연결하는 대신 동시에 연결합니다. |

`mcpServers.reconnect(options)` 지원 옵션:

| 옵션 | 유형 | 기본값 | 참고 |
| --- | --- | --- | --- |
| `failedOnly` | `boolean` | `true` | 실패한 서버만 재시도하거나(`true`) 모든 서버에 다시 연결합니다(`false`). |

### 비동기 해제(선택 사항)

런타임에서 `Symbol.asyncDispose`를 지원하는 경우 `MCPServers`도 `await using` 패턴을 지원합니다. TypeScript에서는 `tsconfig.json`에서 `esnext.disposable`을 활성화합니다.

```json
{
  "compilerOptions": {
    "lib": ["ES2018", "DOM", "esnext.disposable"]
  }
}
```

그런 다음 다음과 같이 작성할 수 있습니다.

```ts
await using mcpServers = await connectMcpServers(servers);
```

## 기타 참고 사항

**스트리밍 가능한 HTTP** 및 **Stdio** 서버에서는 `Agent`가 실행될 때마다 사용 가능한 도구를 검색하기 위해 `list_tools()`를 호출할 수 있습니다. 이러한 왕복 과정은 특히 원격 서버에서 지연 시간을 늘릴 수 있으므로 `MCPServerStdio` 또는 `MCPServerStreamableHttp`에 `cacheToolsList: true`를 전달하여 결과를 메모리에 캐시할 수 있습니다.

도구 목록이 변경되지 않는다고 확신하는 경우에만 활성화하세요. 나중에 캐시를 무효화하려면 서버 인스턴스에서 `invalidateToolsCache()`를 호출합니다. `getAllMcpTools(...)`를 통한 공유 MCP 도구 캐싱을 사용하는 경우 `invalidateServerToolsCache(serverName)`을 사용하여 서버 이름별로 캐시를 무효화할 수도 있습니다.

고급 사용 사례에서는 `getAllMcpTools({ generateMCPToolCacheKey })`를 사용하여 캐시 분할 방식(예: 서버 + 에이전트 + 실행 컨텍스트)을 맞춤 설정할 수 있습니다.

### 서버 접두사가 있는 도구 이름

기본적으로 로컬 MCP 도구는 MCP 서버가 보고한 도구 이름을 유지합니다. 두 로컬 MCP 서버가 같은 도구 이름을 노출하면 모델이 두 도구 중 하나를 안전하게 선택할 수 없으므로 SDK가 중복 도구 이름 오류를 발생시킵니다.

결정론적인 서버 접두사 이름을 사용하려면 에이전트에서 `mcpConfig.includeServerInToolNames: true`를 설정합니다.

```ts
const agent = new Agent({
  name: 'Assistant',
  mcpServers: [docsServer, calendarServer],
  mcpConfig: {
    includeServerInToolNames: true,
  },
});
```

이 설정을 사용하면 `docs` 서버의 `search` 도구는 모델에 `mcp_docs__search`로 노출되고, `calendar` 서버의 `search` 도구는 `mcp_calendar__search`로 노출됩니다. SDK는 원래 서버에서 원래 MCP 도구 이름을 사용하여 계속 호출합니다.

생성된 이름은 ASCII에 안전하고 함수 도구 이름 제한을 준수하며, 같은 에이전트의 로컬 함수 도구 이름 및 활성화된 핸드오프 이름과의 충돌을 방지합니다. 이 설정은 로컬 스트리밍 가능한 HTTP 및 stdio MCP 도구에만 영향을 미치며, 호스티드 MCP 도구는 호스티드 서버 레이블과 도구 메타데이터를 유지합니다.
